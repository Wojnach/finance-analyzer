## P0
- `portfolio/log_rotation.py:573-580`: **[NEW]** P0: `rotate_text()` does `_gzip_file(filepath)` then `open(filepath,"w")` truncate, with NO lock and NO atomic snapshot. `scripts/win/pf-loop.bat:14` holds `loop_out.txt` open the whole run via `>>`, and rotation runs BOTH in-process (`main.py:408 rotate_all()`) AND as a separate hourly task (`install-log-rotate-task.ps1`). Failure modes: (a) separate-process rotation opens `"w"` while cmd.exe holds the file → Windows sharing violation → `rotate_all` swallows the per-file exception (632) → `loop_out.txt`/`agent.log` **never rotate, grow unbounded**; or (b) gzip-read→truncate window is non-atomic → log lines written in between are lost from BOTH archive and live file → the auth-outage detector / postmortems lose data. FIX: prefer (1) make the loop log via Python `RotatingFileHandler` (as logging_config does for portfolio.log) and drop the bat `>>` redirection; or (2) gate `rotate_text` behind a sidecar lock the writer honors; or (3) `r+` + `os.ftruncate(0)` only after an exclusive cross-process lock. Never `open("w")` a file another process appends to.
- `portfolio/file_utils.py:295` (+ `outcome_tracker.py:340`): **[NEW]** P0: `jsonl_sidecar_lock` uses `msvcrt.locking(lfd, LK_LOCK, 1)` as a "blocking" lock, but Windows `_LK_LOCK` only retries ~10s then raises `OSError(EDEADLOCK)`. `log_rotation.rotate_jsonl()` holds this same sidecar lock across read + gzip + write of files up to 50 MB (`signal_log.jsonl`; 379-504, gzip 476-488). On a loaded disk that exceeds 10s → every concurrent `atomic_append_jsonl()` (signal_log, critical_errors, claude_invocations, telegram_messages…) raises `OSError`, the append is **silently lost or crashes the worker thread** — corrupting the append-only journals every subsystem trusts. FIX: wrap the `msvcrt.locking` acquire in a retry loop to a real deadline (true blocking lock), and/or move the gzip/archive write in `rotate_jsonl` OUTSIDE the lock (only the read-classify + final `os.replace` need it; archive gzip can happen on a copy after release).
