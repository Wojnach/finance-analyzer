## P0 / P1
- `portfolio/signals/statistical_jump_regime.py:97-110`: **[NEW]** P0: regime-mislabel bug. In the `neutral` branch, BOTH `j>0` (99-105) and `j<0` (106-110) increment the *same* `current_count` without resetting on direction change. Alternating up/down jumps accumulate the counter → after 3 mixed jumps the regime commits to whichever direction the 3rd jump was → fabricates bull/bear regimes from noise. Feeds `jump_vote` (184-189), 1 of 3 votes. Plausible root cause of the >15pp accuracy collapse. FIX: track separate `up_count`/`down_count`, reset the opposing counter on direction flip before committing.
- `portfolio/signals/statistical_jump_regime.py:201-204 & 211-216`: **[NEW]** P1: vote double-counting. `vol_vote` (low-vol branch) and `trend_vote` both derive direction from the SAME SMA(20) 5-bar slope → 2 of 3 votes perfectly correlated, so any SMA slope sign drives a 2/3 majority regardless of the jump regime, collapsing the "composite" to a single SMA trend follower. FIX: drop SMA slope from `vol_vote` (make it a confidence multiplier) or use a distinct lookback.
- `portfolio/signals/cross_asset_tsmom.py:148-171`: **[REPEAT]** P1: polarity error. `_compute_bond_momentum` (TLT) and `_compute_equity_momentum` (SPY) return the same direction (positive return → BUY) for ALL 5 tickers. FIX: branch on asset class — invert bond-momentum for risk assets, equity-momentum for safe-havens (mirror `_SAFE_HAVENS` in treasury_risk_rotation).
- `portfolio/signals/eth_btc_ratio_roc_zscore.py:51 (+89-93)`: **[NEW]** P1: timeframe mismatch corrupts the ratio. `_fetch_counterparty_close` hard-codes `interval="5m"`, but the engine runs this across 7 timeframes; the target `df` (e.g. 1d bars) is divided element-wise against 5m counterparty bars after `reset_index(drop=True)`, pairing unrelated times.
