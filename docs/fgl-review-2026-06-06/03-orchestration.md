## P0

portfolio/agent_invocation.py:1224: P0: invoke_agent's single-agent Popen spawns the Claude subprocess but NOTHING enforces a wall-clock timeout via communicate()/wait — the only timeout enforcement is check_agent_completion polling _agent_proc.poll() every cycle / 30s watchdog. The Popen has stdout=log_fh, stderr=STDOUT, stdin=DEVNULL (good, no stdin-block), but the subprocess result is never read for auth markers at spawn; relies entirely on the post-hoc agent.log scan. If the completion watchdog thread dies AND run()'s cycle stalls, a hung Claude runs unbounded. The watchdog never-dies guarantee rests on `except Exception` only — a BaseException (MemoryError) escaping the tick kills it silently. → Add a hard secondary guard: have the watchdog re-arm itself, or have loop() assert watchdog liveness each cycle and respawn.

portfolio/multi_agent_layer2.py:218: P0: wait_for_specialists iterates `for proc in procs` and calls proc.wait(timeout=remaining) where `remaining = max(1, deadline - time.time())`. The deadline is shared across all 3 procs, but the loop waits on them SEQUENTIALLY. If specialist #1 hangs the full window, remaining for #2/#3 collapses to the `max(1, …)` floor of 1s, so they are force-killed after 1s even if healthy — and the synthesis then proceeds on partial/empty specialist reports (only quorum==0 is blocked; 1/3 success proceeds). A slow-but-valid specialist gets killed and its empty .md is read by synthesis as if valid. → Wait on all three concurrently (e.g. poll all polls until shared deadline) instead of sequential per-proc waits.

portfolio/claude_gate.py:62 / portfolio/agent_invocation.py:854: P0 (operational, by-design but load-bearing): Claude is doubly disabled (CLAUDE_ENABLED=False AND config.layer2.enabled gate). agent_invocation.invoke_agent BYPASSES the claude_gate kill switch entirely — it Popens claude directly (line 1224) and only respects config.layer2.enabled (line 854). So the master CLAUDE_ENABLED=False in claude_gate does NOT block the Layer 2 trade agent; only config.json does. This is the documented "silent half-on" trap. Confirmed the comment at claude_gate.py:54-58 accurately describes it, but the code path makes it easy to re-enable trading by flipping only one switch. → Route invoke_agent through claude_gate (or have it check CLAUDE_ENABLED) so one kill switch governs all Claude spawns.

portfolio/escalation_router.py:243: P1: held_sell_flip detection matches reason substrings "->sell"/"flipped buy->hold"/"flipped buy->sell" against `r.lower()`, but trigger.py emits flips as `f"{ticker} flipped {triggered_action}->{current_action} (sustained)"` e.g. "flipped buy->sell". ## Risk summary
The highest-impact exposures are the sequential specialist wait (kills healthy specialists and feeds partial context into a real trade decision) and the kill-switch asymmetry where the Layer 2 trade agent bypasses claude_gate's master switch — both can silently change trading behavior without an obvious failure signal. The loop-survival guarantees are otherwise strong (broad try/except, _safe_crash_recovery floor sleep, singleton locks released on exception), but the Claude-subprocess timeout enforcement depends on a single daemon watchdog whose only safety net is `except Exception`, leaving a narrow path to an unbounded hung agent if that thread dies.
