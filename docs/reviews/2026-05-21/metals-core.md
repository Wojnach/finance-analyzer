## Critical (P0 — real-money loss path)

- `portfolio/fin_snipe_manager.py:476`: P0: `compute_exit_plan` is invoked with `financing_level=None` regardless of the warrant's real barrier/financing level (the manager actually reads `barrierLevel`/`financingLevel` at line 606-607 of `_summarize_market`, but never passes them in). With `financing_level=None`, `_apply_risk_overrides` cannot fire the "Knock-out danger < 3%" override at exit_optimizer.py:431-436, the KNOCKOUT_PROB risk flag at exit_optimizer.py:396-400 is silently skipped, and the stop_prob path at exit_optimizer.py:643-646 also degrades to using a stop the optimizer doesn't know about. Net effect: a MINI cert approaching its barrier produces an exit plan that quietly ignores the proximity. Fix: thread `barrier_level` / `financing_level` from `_summarize_market` into the `Position(financing_level=…)` call; default to None only for daily certs (parity-priced products with no knockout).

- `portfolio/fin_snipe_manager.py:536`: P0: `_compute_stop_plan` sets the stop trigger at `position_avg * (1 − HARD_STOP_CERT_PCT)` (5% of warrant price below entry) with no reference to the underlying barrier. For a 5x MINI silver/gold cert, a 5% warrant drop is only ~1% underlying — well within the barrier risk band, so the "stop must never sit within 3% of barrier" rule from `memory/feedback_mini_stoploss.md` is structurally unreachable. Fix: load `barrier_level` from the snapshot's market-guide `keyIndicators`, compute the implied underlying at the trigger, and reject the trigger if it falls within `MIN_BARRIER_DISTANCE_PCT` (5%) of the barrier — same gate as `_tier_skip_for_knockout` in `grid_tiers.py`. Also widen the MIN_STOP_DISTANCE_PCT floor from 1% to 3% to match the documented rule.

- `portfolio/grid_tiers.py:222`: P0: `build_exit_levels` computes the stop-loss purely off the warrant fill price (`fill_price * (1 − GRID_STOP_PCT/100)`, default 3.5%). Knockout safety is enforced only on BUY ladder entries (`_tier_skip_for_knockout`), never on the stop the rotator then places. With 5x leverage a 3.5% warrant drop maps to only ~0.7% underlying move; if the cert was already placed near its barrier (knockout passes at 8%-of-barrier on the buy side), the stop can sit *closer* than the configured safety threshold. Fix: pass `underlying_price`, `barrier`, `leverage` into `build_exit_levels` and reject/widen any stop that would land inside `GRID_KNOCKOUT_SAFETY_PCT` of the barrier — symmetric with `_tier_skip_for_knockout`.

- `portfolio/fin_fish.py:732`: P0: BEAR MINI barrier guard has `pass` instead of a `continue`. For a SHORT MINI the knockout fires when underlying *rises* above barrier; the code detects the condition (`spot >= barrier`) but does not skip the warrant, then continues to `barrier_distance = abs(spot - barrier) / spot * 100` which evaluates to near-zero and slips past `MIN_BARRIER_DISTANCE_PCT`. End result: a knocked-out BEAR cert is still ranked and recommended. Fix: replace `pass` with `continue` so the warrant is dropped, mirroring the LONG branch at line 730-731.

- `portfolio/fish_instrument_finder.py:151-174`: P0: `find_best_instruments` accumulates every search hit with no minimum-barrier-distance filter. Sort key sorts by `barrier_distance_pct` only as a secondary tie-breaker; a 1% or 2%-from-barrier MINI will be returned first if its spread is tight. Auto-discovery for new fishing instruments therefore routinely picks the most knockout-prone cert. - `data/metals_loop.py:2258`: P0: position reactivation sets `pos["stop"] = round(pos["entry"] * 0.95, 2)` — a hard-coded 5% below warrant entry, ignoring the cert's barrier entirely. - `data/metals_loop.py:4902`: P0: hardware stop placement only checks that the trigger is ≥3% below current bid — not that the trigger's implied underlying is ≥3% from the cert's barrier. - `data/metals_loop.py:2469`: P0: `sell_price = round(trigger_price * 0.99, 2)` — stop-loss limit price is fixed at trigger − 1%. - `portfolio/fin_snipe.py:46-54`: P2: `fetch_stop_losses` swallows all exceptions and returns []. If the GET fails repeatedly (auth issue), the manager will see "no managed stops" and create new ones every cycle on top of the un-fetched live stops. - `portfolio/grid_fisher.py:430-433`: P2: `_INLINE_COOKIE_PATTERN` regex matches "cookie:" and "authorization:" but not Avanza's specific custom headers. - `data/metals_loop.py:1099-1115`: P2: `_silver_fetch_xag` uses raw `requests.get` (no `http_retry`, no `_binance_limiter`). Called every 10s during active positions. - `portfolio/fish_monitor_smart.py:148-153`: P2: cross-asset price fetch wraps every iteration in `with suppress(Exception)`. - `portfolio/grid_fisher.py:631-644`: P2: `should_halt_global` uses session-wide P&L only; the docstring at line 632-636 admits "richer drawdown tracking lives in risk_management.py and is wired in via a later batch" — but the per-session loss limit (`500 SEK × N_instruments = 1500 SEK`) is well above any single trade's expected loss, so the halt never fires in practice. ## Patterns

- **Pervasive missing barrier-awareness on stop-losses**: every code path that places a stop computes the trigger as a fixed warrant-price % below entry or current bid (`grid_tiers.build_exit_levels`, `fin_snipe_manager._compute_stop_plan`, `metals_loop:2258`/`2469`/`4902`, `exit_optimizer:397`/`644`). The system has barrier-aware *entry* gates (`fin_fish.MIN_BARRIER_DISTANCE_PCT`, `grid_tiers._tier_skip_for_knockout`, `_apply_risk_overrides.distance_pct<3`) but they do not bind the *stop* level. With 5x leverage a 3% warrant move is 0.6% underlying — a stop that sits 5% below warrant entry can be inside the 3%-of-barrier band. Recommend: a single `barrier_aware_stop(entry_warrant, leverage, barrier, current_underlying)` helper used by all three call sites.

- **Cancel-then-place sequencing instead of place-then-cancel**: `grid_fisher.rotate_on_buy_fill` (line 1484-1515), `metals_loop._update_stop_orders_for` (line 2448-2519), `fin_snipe_manager._stage_replacements` (when emergency=True) all cancel the old stop before confirming the new placement. A flap window of seconds-to-minutes leaves positions naked. Reverse the order universally.

- **Raw `requests.get` against Binance FAPI**: `fin_fish.fetch_live_spot/fetch_daily_ranges`, `orb_predictor.fetch_klines`, `fish_monitor_smart._fetch_price`, `data/metals_loop._silver_fetch_xag/_gold_fetch_xau` all bypass `portfolio.http_retry` and `_binance_limiter`. Under FAPI rate-limit pressure (free tier is 2400 req/min on weight-1 endpoints, shared globally) these create silent cliff failures. Standardize on `portfolio.http_retry.fetch_json`.

- **Mixed thin-spread vs constant-offset assumptions**: stop-loss limit prices are hard-coded at `trigger * 0.99` (`metals_loop:2469`), `trigger * 0.995` (`grid_fisher:1493`), and `bid * 0.99` (`grid_fisher:1843`). All three break when the cert spread exceeds the offset (BEAR GULD VON4 is 2.2%). A spread-aware helper would catch all three.

- **Single-thread fast-tick + shared global state**: `_silver_fast_tick`, `_silver_entry_fast_tick`, `_gold_entry_fast_tick` all mutate module globals (`_silver_session_low/high`, `_silver_alerted_levels`, `_silver_fast_prices`, `_xag_entry_prices`, `_xau_entry_prices`) from `_sleep_for_cycle` in the main thread. No locks needed (single-thread), but the state-reset logic at `_silver_reset_session` is brittle to fresh-position races (line 7208 of `metals_loop.py`). Recommend: encapsulate fast-tick state in a class with explicit reset/init semantics.

## Out of Scope but Spotted

- `portfolio/exit_optimizer.py:619`: minor — `np.array([_compute_pnl_sek(... for p in np.percentile(terminal, [10, 25, 50, 75, 90])])` constructs a 5-element array from a 5-element percentile list, computes mean. Identical to `np.mean(percentile_pnls)` over 5 quantiles, which under-weights the distribution. The hold-to-close EV is therefore biased toward the median. Use the full terminal distribution directly.

- `portfolio/iskbets.py:892-913`: stock-only price fetch uses Alpaca's "iex" feed by default — equity feed for `MSTR`-style tickers only. The ticker space configured for iskbets in `tickers.py` may contain crypto tickers; for those, the iex-feed branch is unreachable but still listed.

- `portfolio/grid_fisher.py:849`: `self._buying_power_cache` documented as `(mono_ts, value)` but the staleness check at line 924-930 mixes `time.monotonic()` with the assignment at line 916. Self-consistent, but the constant `GRID_BUYING_POWER_STALE_GRACE_SECS` is in wall-clock seconds — fine because monotonic seconds and wall-clock seconds tick at the same rate, but a process suspended for hours (Windows hibernate) will have the stale-grace evaluated against monotonic which doesn't advance during hibernate. After resume, a 5-hour-old buying-power reading appears fresh. Low impact but worth a comment.

- `portfolio/fish_instrument_finder.py:42-50`: Avanza search uses `_api/search/filtered-search` with a freeform `query` field — Avanza's response includes both warrants and certificates but no SOR filter on issuer. The fallback to "BULL SILVER AVA" can return Vontobel (VON) certs if Avanza's search ranking is fuzzy. Combined with the missing barrier-distance filter, the auto-discovery can substitute a VON cert (wide spread, sometimes near barrier) for a missing AVA cert.

- `data/metals_loop.py:413,425-426`: comment "default OFF: only place stop orders on explicit request" applies to `STOP_ORDER_ENABLED`, but `HARDWARE_TRAILING_ENABLED = True` defaults the trailing-stop logic ON. The trailing path calls into `_update_stop_orders_for` which calls `place_stop_loss` — so "stop orders default off" is misleading.

- `portfolio/grid_fisher.py:1730-1735`: cooldown after signal-mismatch cancel does not exist (see P1 above), and the existing `inst.cooldown_until` cooldown set only at flip/rapid-cancel paths is too narrow. A signal that oscillates ±LONG/HOLD without crossing into SHORT never triggers the flip path, so the no-mismatch-cooldown bug above is the real entry point.

- `portfolio/orb_predictor.py:357-368`: percentile computation uses integer-index `sorted_list[idx]` with `idx = int(len(sorted_list) * pct / 100)` — not actually interpolated percentiles. For small sample sizes this biases toward the lower-bound. Fine for n≥20 but predictor explicitly accepts `min_sample=5`. Use `statistics.quantiles` or `numpy.percentile` for correctness.
