## Critical (P0)

portfolio/fomc_dates.py:13-22: P0: 2026 FOMC dates look off by one day for the January meeting. Code has `date(2026, 1, 28), date(2026, 1, 29)` but the published Fed 2026 schedule for the first meeting is Tue/Wed Jan 27-28, 2026. Every other 2026 entry matches the official Tue/Wed pattern; only January 2026 is shifted +1. Because today's date (2026-05-21) is past that meeting, the bug has already silently mis-dated the post-event-relief BUY signal and the `econ_calendar` proximity gate for that announcement. Layer 2 saw the gate clear a day late. Fix: change to `date(2026, 1, 27), date(2026, 1, 28)` and update `FOMC_ANNOUNCEMENT_DATES[0]` to `date(2026, 1, 28)`. Cross-check against `https://www.federalreserve.gov/monetarypolicy/fomccalendars.htm` before committing.

portfolio/alpha_vantage.py:32, 162-168: P0: Daily budget tracker is in-memory only and resets to 0 on every process restart. With PF-DataLoop's logon + auto-restart task, a single Windows crash recovery wipes the 25/day count, allowing a fresh 25 calls. Repeat crashes burn through the entire monthly free-tier allotment in hours. The persistent cache file (`fundamentals_cache.json`) holds the data but does NOT hold the budget counter. Fix: persist `_daily_budget_used` and `_budget_reset_date` to disk (e.g. inside `fundamentals_cache.json` under a reserved key like `_budget_state` or in a sibling file) and reload during `load_persistent_cache()`. Apply the same fix to `shared_state._newsapi_daily_count` (line 304).

portfolio/shared_state.py:304-346 + portfolio/sentiment.py:225-235: P0: NewsAPI daily counter only increments on truthy result, but the API returns `200 OK` + empty `articles[]` when rate-limited (some plans) or returns the `code: "rateLimited"` payload that the parser silently discards via `data.get("articles", [])`. After the real 100/day quota is hit, our counter is stuck at whatever pre-rate-limit value it was at, `newsapi_quota_ok()` keeps returning True, and every 20-min metals refresh re-hits NewsAPI for nothing. Fix: (a) inspect response body for `code == "rateLimited"` or HTTP 429 and force the counter to budget+1; (b) increment the counter on EVERY call attempt (success and empty), accept the small over-count rather than the unbounded retries; (c) persist the counter across restarts.

portfolio/data_collector.py:282 + line 290: P0: Cache key `tf_<source_key>_<label>` does NOT include the data source. For Alpaca tickers (e.g. MSTR), when market is closed the dispatcher uses `yfinance_klines` (with `prepost=True`), and when market opens it switches to `alpaca_klines` (with `feed=iex`, no prepost). The cache entry persists across that boundary up to its TTL (15m for "12h" timeframe, 24h for "6mo"). So pre/post-hours yfinance bars leak into Alpaca IEX cache space and vice-versa, and the "live prices first" rule is violated for the cached TTL. Fix: include source-source-id in cache key, e.g. `f"tf_{source_key}_{label}_{'yf' if yf else 'live'}"`, or invalidate the per-ticker cache when `_current_market_state` transitions.

## Important (P1)

portfolio/onchain_data.py:208-243: P1: No daily budget gate on the 6 BGeometrics endpoints. Each `_fetch_all_onchain` call burns 6 calls (one per metric) plus 5×1s sleeps. The 15/day free quota is supposedly defended by the 12-hour `ONCHAIN_TTL` and the persistent cache seed in `get_onchain_data()`, but if `_save_onchain_cache` fails (line 100) or the cache file gets corrupted, the next process restart re-fetches all 6 endpoints with no counter. Fix: add a per-day counter (persisted to disk like Alpha Vantage's `_daily_budget_used`) gating `_fetch_all_onchain` at 12 calls/day.

portfolio/onchain_data.py:238-243: P1: `_fetch_all_onchain` saves the cache and returns whenever `any_success` is True. So a 1-of-6 successful fetch overwrites the persistent cache with 5 missing fields, and the in-memory cache returned by `_cached` has 5 metrics as `None`. The downstream signal (`onchain_btc`) reads those as null → may emit HOLD instead of using the previous (complete) cached values. Fix: when a fetch is partial, merge into the previous-cache values rather than replacing with a hollowed-out dict; or require ≥4-of-6 fields to persist.

portfolio/onchain_data.py:240: P1: BGeometrics integers come back as strings and `_safe_float` returns `None` on parse failure. `interpret_onchain` correctly treats None as "no zone", but `_save_onchain_cache(result)` writes `{"mvrv": None, ...}` to disk — and a subsequent restart with that persistent file makes the seeded in-memory cache look fresh while returning None for half the metrics. Fix: don't persist a metric value that's None; or write a single timestamp per metric so the loader can selectively re-fetch the stale ones.

portfolio/onchain_data.py:31-67: P1: `_coerce_epoch` silently coerces unparseable timestamps to 0.0 (forcing a cache miss). This DOES protect the loop, but combined with the no-counter gap above, a corrupted cache file forces 6 API calls every process startup. Pair this with the "auto-restart on crash" PF-DataLoop policy and a single corrupt write turns into a budget-burn loop. Fix: when `_coerce_epoch` returns 0.0 and the cache file exists, log at WARNING (not DEBUG) and back off the auto-fetch by 1 hour to prevent restart-storm budget burn.

portfolio/signals/credit_spread.py:102-117: P1: FRED `BAMLH0A0HYM2` observation date is never checked. Code takes `values[0]` as "current" but FRED HY OAS typically updates 1 trading day late and during holidays can lag 3+ days. Z-score, momentum, and crisis-gate decisions all key off `values[0]` regardless of staleness — a long weekend after a Friday rally means Monday/Tuesday signals trade on Thursday data labelled as "now". Fix: parse `obs["date"]` for the latest observation, compare to `datetime.now(UTC).date()`, and gate the signal to HOLD if data is >2 calendar days old. Apply the same fix to `metals_cross_asset.py:_fetch_fred_values` (lines 105-166) which has identical pattern for EPU and TIPS.

portfolio/signals/credit_spread.py:69-122: P1: No daily budget tracking for FRED. Free FRED is 120/min per IP but unlimited daily — however the in-memory `_oas_cache` is reset on every process restart, so each restart re-fetches 300 observations of HY OAS. Multiply by 3+ FRED-using signals (`credit_spread`, `metals_cross_asset` for EPU + TIPS) and a single crash-restart cycle is ~9 FRED hits. Not budget-killing but noisy. Fix: persist FRED responses to disk with the same atomic_write_json pattern used by `fundamentals_cache.json`, keyed by series_id.

portfolio/data_collector.py:96: P1: `df["time"] = pd.to_datetime(df["open_time"], unit="ms")` produces a tz-naive Timestamp from a UTC-epoch ms field. The yfinance branch on line 245 sets `df["time"] = df.index` which is tz-aware (Etc/UTC or America/New_York). Mixing tz-aware and tz-naive timestamps from the same `time` column across cache reads/writes causes pandas `TypeError: Cannot compare tz-naive and tz-aware` whenever downstream code does `df['time'] > some_ts`. Fix: set `df["time"] = pd.to_datetime(df["open_time"], unit="ms", utc=True)` for Binance branches, and `df["time"] = df.index.tz_convert("UTC")` (or `tz_localize`) on the yfinance path.

portfolio/data_collector.py:222-229: P1: `yf.download(..., prepost=True, auto_adjust=True)` is used for ALL Alpaca-fallback fetches. Pre/post-market yfinance bars have wildly different volume profiles than IEX regular-session Alpaca bars. Cached "Now" timeframe at market-close transition shows ~6 hours of after-hours bars with thin volume that the indicators read as low RVOL → forces HOLD via the RVOL <0.5 gate. The volume/ADX gate then mis-classifies the regime as ranging. Fix: when caching the result, tag it with `extended_hours: True/False` and invalidate at the regular-session open boundary.

portfolio/data_collector.py:209-247: P1: `yfinance_klines` swallows yfinance's MultiIndex single-ticker quirk but does NOT pass the `_yfinance_lock`. ## Patterns

1. **In-memory rate-limit budgets that reset on restart.** Alpha Vantage (P0), NewsAPI (P0), implicitly BGeometrics (P1), FRED (P1). The auto-restart-on-crash Task Scheduler policy turns every crash into a fresh quota bucket. Persistent budget tracking is needed across all rate-limited free-tier APIs.

2. **FRED observation date never validated.** `credit_spread`, `metals_cross_asset` (EPU + TIPS), `breakeven_inflation_momentum`, `gold_real_yield_paradox` all do `values[0]` assuming "current" — but FRED lags 1-3 days routinely and 5+ days around holidays. Centralize the freshness gate.

3. **yfinance thread-safety inconsistent.** Some call sites hold `_yfinance_lock`, others don't. `fetch_vix`, `yfinance_klines` direct calls, `_fetch_yahoo_headlines`, `_fetch_earnings_yfinance` all lack the lock. The "yfinance is not thread-safe" comment at data_collector.py:274 should be enforced at the function entry, not the call site.

4. **Cache keys don't encode source.** Multiple modules cache by ticker+timeframe but switch underlying source (Alpaca→yfinance, Binance→cached) without versioning the cache key. Stale-across-source-boundary is a recurring class of bug.

5. **Silent exception swallowing at DEBUG level.** Many `except Exception: logger.debug(...)` blocks hide rate-limit/quota failures. Once → fine; sustained → invisible degradation. Distinguish transient (DEBUG) from quota/auth (WARNING) by exception type/response code.

6. **Hardcoded calendar dates with no expiry guardrail.** FOMC, CPI, NFP, GDP all hardcoded through 2027. No telemetry alarms when the calendar runs out. Pending_pickups system exists — use it for an annual reminder.

7. **MultiIndex defensive flatten everywhere.** Each yfinance call site has its own `if isinstance(df.columns, pd.MultiIndex): df.columns = df.columns.get_level_values(0)`. Factor into a shared helper.

8. **`data.get("X")` without checking for None.** Multiple places parse JSON expecting keys to exist but rely on optional chaining only one level deep. Then `float(None)` raises and gets caught by the outer try/except, but the response data is silently lost for the cycle.
