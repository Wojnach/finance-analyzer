## Findings

### Critical (P0)

portfolio/signal_engine.py:4011: P0: Core gate uses pre-persistence-filter `active_voters` and pre-persistence-filter `core_active`, so the early/raw-action branch (line 4012-4025) can emit a raw consensus that is later inconsistent with `weighted_action` at 4239. Although the weighted path re-checks at 4239 with `post_persistence_voters`, `extra_info["_raw_action"]` is published at 4251 with the unfiltered count and is consumed by reporting/triggers. Fix: apply persistence filter before computing active_voters/core_active OR move the raw-consensus branch to operate on consensus_votes.

portfolio/signal_engine.py:2562: P0: The accuracy gate compares `acc < effective_gate` but for signals with zero samples the accumulator returns the blended-default `accuracy=0.5` (see blend_accuracy_data line 928), and `samples >= ACCURACY_GATE_MIN_SAMPLES` is the only guard. A signal with `total=200` but `accuracy=None` is sanitized to drop both fields (line 2305), at which point `stats.get("accuracy", 0.5) = 0.5` and `stats.get("total", 0) = 0`, so the gate skips it — masking a poisoned-cache row as a passing vote. Fix: also gate when samples==0 OR add explicit "samples present and accuracy missing" force-HOLD.

portfolio/signal_engine.py:4116: P0: Per-ticker accuracy override discards directional sample counts when the per-ticker entry lacks them. The override at line 4119 copies only listed fields; signals missing `total_buy/total_sell` retain the GLOBAL ones from `accuracy_data`. Subsequent directional gate at 2596 then compares per-ticker overall acc against global directional sample counts — a category mismatch that can let a signal vote with 80% global SELL accuracy when the per-ticker SELL accuracy is 20% and uncopied. Fix: drop directional fields entirely from `override` when the ticker stats don't carry them so downstream `.get(..., overall_acc)` fallback applies.

portfolio/signal_engine.py:2602: P0: BUG-182 fix uses `_DIR_WEIGHT_MIN_SAMPLES=20` but the directional gate at 2596 uses `_DIRECTIONAL_GATE_MIN_SAMPLES=30`. A signal with 25 BUY samples at 25% accuracy passes the gate (insufficient samples) yet contributes its 25% as the vote weight here. Net: actively-wrong directions weight consensus at their failure rate, not at neutral 0.5. Fix: use the same min-samples threshold for both gate and weight, OR fall back to overall acc when samples < gate threshold.

portfolio/signal_engine.py:3162: P0: MACD crossover comparison uses `ind["macd_hist_prev"] <= 0` and `>= 0`, so a single bar with `macd_hist == 0` produces both BUY (current > 0 and prev == 0 — wait, no, current > 0 satisfies first arm) — but if `hist_prev == 0` and `hist == 0` neither arm fires (correctly). However, the dead-zone helper at 3166 then runs on an *equality* boundary and emits a soft vote from a 3-bar slope. Result: at exact zero-crossings the MACD signal can flip BUY→soft-SELL→BUY within 3 cycles. Fix: handle the `hist == 0 == hist_prev` case explicitly or require non-flat magnitude in both branches.

portfolio/signal_engine.py:3146: P0: RSI thresholds `rsi_lower = max(rsi_lower, 15)` / `rsi_upper = min(rsi_upper, 85)` clamp the adaptive thresholds AFTER `.get(..., 30)` default — so on an indicator dict missing `rsi_p20` (typical cold-start or sparse-data path), thresholds default to 30/70 and STILL get clamped to 15/85 only if the original values exceeded those bounds. A cold-start ticker with rsi=20 votes BUY (correct) — but the comment claims "adaptive thresholds" while the actual path is the fixed default. Cosmetic-style issue, but more importantly: when `ind["rsi"]` is None or NaN, `ind["rsi"] < rsi_lower` raises TypeError. Fix: None/NaN guard on `ind["rsi"]`.

portfolio/signal_engine.py:3147: P0: `ind["rsi"]` is dereferenced with `[]` not `.get()`. If indicators dict has no "rsi" key the function crashes the entire generate_signal for that ticker; main.py's pool-timeout pattern would then silently leave votes empty. Fix: use `.get("rsi")` + None-guard, mirroring the macd_hist/ema9/ema21 pattern below which uses bracket access too — same risk for all four core indicators.

portfolio/signal_engine.py:4239: P0: `post_persistence_voters < min_voters` enforces the MIN_VOTERS gate, but metals MIN_VOTERS_METALS=2 means a 2-voter consensus where both voters are the SAME correlation-group leader (e.g., both pure_trend members ema+heikin_ashi) bypasses correlation deduplication on the entry decision. The weighted path applies the penalty but the gate doesn't, so the count-based check passes even when the *information* is from one source. Fix: count effective independent voters (post-correlation-dedup) for the min_voters gate.

portfolio/signal_engine.py:4384: P0: `conf = min(conf, 0.80)` confidence cap applied AFTER per-ticker consensus gate. For `consensus_gate` force-HOLD entries that earlier set conf=0.0, this is fine, but for the linear factor model boost at 4336 (`conf *= 1.10`), a 0.74 confidence boosted to 0.814 is capped to 0.80 — silently erasing 1.4pp of signal. More importantly, the comment at 4382 says >80% is anti-correlated with accuracy — yet 3h horizon below at 4389 applies CONFIDENCE_CAP_3H which may be tighter; the cascade is right but the second cap re-applies a `min()` without re-clamping after any subsequent intermediate boost. Fix: apply confidence cap as a final post-processing step after every multiplier closes.

portfolio/outcome_tracker.py:472: P0: `change_pct = round(((hist_price - base_price) / base_price) * 100, 2)` uses 2-decimal rounding which conflicts with `_MIN_CHANGE_PCT = 0.05` in accuracy_stats.py — a true 0.045% move rounds to 0.04, falls below _MIN_CHANGE_PCT, gets treated as neutral and skipped; a true 0.055% move rounds to 0.06, passes — the rounding boundary changes whether borderline outcomes get scored. Fix: store more precision (e.g., 4 decimals) so the neutral threshold isn't quantized.

portfolio/outcome_tracker.py:458: P0: `cache_key = (ticker, int(target_ts // 3600))` quantizes prices to 1-hour buckets — but `_fetch_historical_price` for Binance uses `interval=1h, startTime=start_ms, limit=1` which gets the candle that STARTS at start_ms. If two horizons (e.g., 3h and 4h) for the same entry fall in the same hour bucket but different actual times, they SHARE the same price — yet they should be evaluated at different timestamps. Fix: use the actual minute-precision target_ts as part of the cache key, OR document that the 1h granularity is acceptable.

portfolio/outcome_tracker.py:436: P0: Inner-all-filled check `if all_filled and all(...)` is a redundant double-check; the outer `all_filled` was just set from the same iteration. portfolio/accuracy_stats.py:1654: P0: `check_accuracy_changes` requires only `old_data.get("total", 0) < 10` — 10 samples is far below the 30-sample gate used elsewhere. A 10-sample signal at 90% accuracy compared to 7-day-ago 10-sample signal at 40% triggers an alert with extreme noise. Fix: raise to ACCURACY_GATE_MIN_SAMPLES (30) for both old and new samples.

portfolio/accuracy_stats.py:225: P0: `change_pct = outcome.get("change_pct", 0)` — defaulting to 0, then later `if change_pct is None` at 226. ### Patterns

- **Silent fail-through via broad except**: every module has dozens of `except Exception: logger.debug(...)` that swallow errors and continue. accuracy_stats.py (1474, 1482, 1523), signal_engine.py (1488, 1714, 3551, 3873, 4109, etc.), forecast_accuracy.py (141), meta_learner.py (361). The pattern is consistent — return empty dict / fall through to default — but `accuracy_data = {}` means accuracy gate is silently bypassed. Some sites guard with H3-style fail-closed; many don't.

- **Default-0.5 accuracy mask**: `stats.get("accuracy", 0.5)` is used in 20+ places as a neutral default. Combined with `samples = 0` and `samples >= 30` gate, every NEW or MISSING signal gets a free pass to consensus at coin-flip weight, often summing to be the deciding vote.

- **Inconsistent sample thresholds**: `_DIR_WEIGHT_MIN_SAMPLES=20`, `_DIRECTIONAL_GATE_MIN_SAMPLES=30`, `MIN_SAMPLES_FOR_IC=30`, `ACCURACY_GATE_MIN_SAMPLES=30`, `_PER_TICKER_MIN_SAMPLES=30`, `_DYNAMIC_HORIZON_MIN_SAMPLES=50`, `_PTC_MIN_SAMPLES=500`, `MIN_SAMPLES=15` (postmortem), `min_recent_samples=30`, `total < 10` (check_accuracy_changes). The lack of a single source of truth means gating policy is inconsistent — a signal can be "trustworthy enough" for one check but not another.

- **Horizon collapse to 1d**: 3d/5d/10d horizons are explicitly mapped to "1d" cache at signal_engine.py:4035 (TODO comment present). Per-ticker consensus gate, accuracy data load, regime accuracy load all reuse "1d" data for long horizons. Means: a 5d consensus is gated on 1d signal accuracy — fundamentally wrong-horizon evaluation.

- **Read-modify-write atomicity**: backfill_outcomes added a lock (good), but per-horizon accuracy cache writes (write_accuracy_cache, write_regime_accuracy_cache) use only the `_accuracy_write_lock` — concurrent same-horizon writers serialize fine but cross-process writes (PF-OutcomeCheck task + main loop) have no inter-process coordination beyond `_atomic_write_json`. Cross-process race during outcome backfill can clobber the legacy "time" key.

- **Module-load assertions**: Used at signal_engine.py:1041-1064 and 941-953. Python `-O` flag strips assertions. Critical config validity then becomes runtime-only — silent miscoercion under `-O`.
