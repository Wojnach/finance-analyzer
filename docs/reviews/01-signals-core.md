## P0 — money / corruption / silent stall

- `portfolio/outcome_tracker.py:211-231` — **P0: outcome backfill uses a future bar's close (lookahead/forward-shift).** Binance `/klines` with `startTime=target_ts, interval="1h", limit=1` returns the bar whose **open** is at/after `target_ts`; `data[0][4]` is that bar's **close**, i.e. the price ~1h AFTER `target_ts`. Causal chain: every crypto/metals outcome at horizon H is actually scored against the price at H+~1h → `change_pct` is systematically shifted forward → `signal_accuracy`/`consensus_accuracy`/IC all measure a horizon that's longer than labeled. Worst at the 3h horizon (33% relative error in the realized window). This inflates/deflates measured edge versus what the live trade actually captures and feeds straight into the accuracy gate, weighting, and Mode-B probabilities the user trades on. → Fetch the bar that *ends* at `target_ts`: request `startTime=target_ts-3600_000, endTime=target_ts, limit=1` (or use `/ticker/price` only when `now≈target_ts`) and take the close of the bar closing at `target_ts`; assert the returned `open_time + interval <= target_ts + tolerance`.

- `portfolio/signal_engine.py:4581-4583 + 3902-3916` — **P0 (latent, MSTR only): cross-ticker consensus cache makes consensus a function of itself across cycles.** `_cross_ticker_consensus[ticker]` is written with the *final* post-penalty `action` at end of every `generate_signal`, and MSTR injects `votes["btc_proxy"] = btc_cons["action"]` read from that same cache. Because the cache persists across cycles and MSTR is a core-voter-eligible signal path, a stale BTC HOLD/BUY from a prior cycle can drive MSTR consensus; worse, the write happens for *every* ticker/horizon (7 horizons overwrite the same `{ticker: ...}` slot within one cycle), so the value BTC-proxy reads is whichever BTC horizon ran last, not the 1d/decision horizon. → Key the cache by `(ticker, horizon)` and only write the canonical decision horizon (1d); document the one-cycle staleness contract and gate btc_proxy on a freshness timestamp so a stalled BTC thread can't feed MSTR a frozen vote forever.

## P1 — wrong signal/accuracy math or unhandled failure path

- `portfolio/accuracy_degradation.py:638-648 + 67-77` (root cause of seeded "12 signal(s) dropped >15pp") — **P1: degradation diff compares two non-overlapping 14d windows with only a 2σ same-window SE gate, so a real regime shift fires a CRITICAL en masse.** The current side is recent-14d; the baseline side is the *14-day-old* snapshot's own recent-14d block (`_find_baseline_snapshot` targets now-14d). These windows do not overlap, so a genuine regime change (rally→pullback) moves many signals 15-30pp at once with `n≈200-1000` where the 2σ SE gate (~±4pp at n=600) does not suppress them. `_classify_severity` escalates to CRITICAL at ≥3 simultaneous drops → 12 drops = CRITICAL, repeatedly, which is exactly the seeded 4× fire. This is a *false alarm* class: it flags the market regime, not signal degradation. → Either (a) require the drop to exceed `2σ` computed against the *overlapping* trailing window (compare recent-14d to the immediately-prior 14d, not the 14-day-old snapshot), or (b) gate the whole alert behind the existing macro/regime detector and raise `CRITICAL_MIN_SIGNAL_COUNT` / add a "system-wide drop ⇒ regime, not degradation" suppression when >N signals drop together (a broad simultaneous drop is the signature of a regime change, not N independent degradations).

- `portfolio/accuracy_degradation.py:617-648` — **P1: per-ticker scope (`_diff_against_baseline` block #2) reuses the 200-sample / global SE gate but per-ticker-per-signal samples are far smaller, so it is the small-sample instability the prompt warns about.** `_per_ticker_recent` produces `{ticker: {sig: {accuracy, total}}}` where `total` for a single ticker+signal at 14d is routinely 30-150; with `MIN_SAMPLES_HISTORICAL=MIN_SAMPLES_CURRENT=200` most are filtered, but signals on high-activation tickers cross 200 and then swing wildly (binomial SE at n=200 ≈ ±3.5pp → 2σ≈7pp, well under a 15pp threshold that noise hits ~1/100). These per-ticker alerts inflate the alert count that drives the ≥3 CRITICAL classifier. → Use a separate, higher `min_samples` for the per-ticker scope (e.g. 400) and/or require the SE multiple to scale with scope (3σ for per-ticker), so a single noisy per-ticker pair can't push the aggregate into CRITICAL.

- `portfolio/signal_engine.py:4318-4329` (utility boost) — **P1: utility boost multiplies accuracy *after* it already passed the gate, can push a 47% signal's weight up to 0.95 and is applied on top of direction-specific weighting inconsistently.** `boosted_acc = min(raw_acc * boost, 0.95)` overwrites `accuracy_data[sig]["accuracy"]` but leaves `buy_accuracy`/`sell_accuracy` untouched; downstream `_weighted_consensus` uses direction-specific weight when `total_buy/sell>=20` (line 2723-2726), so the boost silently does nothing for mature directional signals and over-weights only the immature ones — the opposite of intent. → Apply the utility multiplier to the *final* weight inside `_weighted_consensus` (where direction-specific weight is already chosen), not to the raw `accuracy` field that the gate and directional path read.

- `portfolio/signal_engine.py:2682-2704` — **P1: directional rescue runs even under circuit-breaker relaxation, double-relaxing the gate.** A signal below the (already relaxed) `effective_gate` can still be rescued at up to 0.95x weight if its directional accuracy ≥55%, so during a regime transition where the breaker has *already* dropped the gate to 0.45, a 44%-overall signal whose SELL side happens to be 55% votes at near-full weight. The "high-sample tier is never relaxed" guarantee (SC-P1-2) is bypassed for the directional path because rescue reads `buy_accuracy/sell_accuracy`, not the gated overall accuracy + high-sample floor. → Skip directional rescue when `relaxation > 0`, or apply the high-sample floor to the rescue check (`if samples >= HIGH_SAMPLE_MIN and acc < 0.50: no rescue`).

- `portfolio/outcome_tracker.py:286-289` — **P1: yfinance stock backfill falls back to the *first* available bar when no bar ≤ target_date exists.** `if candidates.empty: return float(h["Close"].iloc[0])` returns a price from up to 5 days *before* the snapshot, producing a wildly wrong `change_pct` for that outcome rather than skipping. For MSTR this mislabels accuracy on any horizon whose target lands before the first returned bar (early history / holiday gaps). → Return `None` (skip) when `candidates.empty`, matching the "no data ⇒ no outcome" contract used on the Binance paths.

- `portfolio/forecast_accuracy.py:158-161 + 142` — **P1: forecast outcome counts `change_pct==0` and a missing change as "down".** `actual_change = outcome.get("change_pct", 0)`; `actual_up = actual_change > 0`; a 0 or absent move is scored as a SELL-correct / BUY-wrong outcome instead of being treated as neutral/skip like `_vote_correct` does everywhere else. → Reuse `accuracy_stats._vote_correct` (or replicate its `None`/`|x|<_MIN_CHANGE_PCT` neutral skip) so forecast accuracy doesn't get a free directional bias from flat bars.

- `portfolio/ic_computation.py:130 + 150-160` — **P1: rolling-ICIR window slides over a list whose order is entry-iteration order, not time order, so ICIR (the IC stability gate) is computed on a shuffled series.** `signal_data[sig]["votes"]/["returns"]` are appended in `for entry ... for ticker ...` order; entries come from `load_entries()` (SQLite/JSONL) with no guaranteed chronological sort, and tickers interleave. `_rolling_ic` then takes contiguous 50-element windows assuming temporal adjacency, so ICIR is meaningless and the `_IC_STABILITY_MIN=0.10` gate that decides whether IC adjusts weight is gating on noise. → Sort the (vote, return) pairs by entry `ts` before rolling, and keep per-ticker series separate (don't interleave tickers into one rolling window).

- `portfolio/signal_engine.py:1596-1612` (`_get_horizon_weights`) — **P1: dynamic horizon weights fall back to *static* only on cache miss, but on a partially-populated `accuracy_cache.json` (e.g. `3h_recent` present, comparison `1d_recent` absent) it returns `{}` (no multipliers) instead of the static dict.** `_compute_dynamic_horizon_weights` returns `HORIZON_SIGNAL_WEIGHTS.get(horizon, {})` on missing cross-horizon data, but the path at line 1565-1569 builds `cross_data` from whatever is present and can yield an empty `weights` dict that *is* truthy-empty → the `weights if weights else HORIZON_SIGNAL_WEIGHTS` guard catches it, OK — but `_get_horizon_weights` then does `cast(...) if weights else {}`, and `_cached` can cache an empty `{}` for the full TTL, silently dropping horizon weighting for an hour. → Distinguish "computed empty" from "no data": have `_compute_dynamic_horizon_weights` always return the static dict (never `{}`) when it can't compute, and don't cache empty results.

- `portfolio/signal_engine.py:2818-2827` (`_weighted_consensus` tie path) — **P1: consensus uses a hard `>= 0.5` weight-share threshold, so a 50/50 weighted split returns HOLD with `conf = 0.5` but a 50.0001/49.9999 split returns a directional trade at ~0.5 confidence.** Combined with the MIN_VOTERS gate this is mostly fine, but the boundary is knife-edge and asymmetric (`buy_conf > sell_conf and buy_conf >= 0.5`): an exact tie in *weight* (common when two opposing rescued signals get identical penalized weights) yields HOLD, while any float jitter flips it to a low-conviction trade. → Require a minimum margin (`buy_conf - sell_conf >= ε`) before emitting a directional action, so near-ties are HOLD deterministically.

## P2 — latent footguns

- `portfolio/signal_engine.py:35-37, 2225-2226, 46-47, 82-83, 621-622, 650-651` — **P2: many module-global mutable caches/dicts are shared across the 8-worker ThreadPoolExecutor and 7 horizons.** `_adx_cache`, `_ic_data_cache`, `_persistence_state`, `_cross_ticker_consensus`, `_macro_window_cache`, `_phase_log_per_ticker` are lock-guarded (good), but `_persistence_state` keys on `ticker` only — the same ticker is processed at 7 horizons in parallel, so all 7 horizon threads mutate one ticker's persistence counters, corrupting the cycle count (a 3h flip increments the same counter a 1d read then trusts). → Key `_persistence_state` by `(ticker, horizon)`; persistence is horizon-specific by design.

- `portfolio/signal_engine.py:4224` — **P2: fail-closed accuracy fallback writes `total: 999` for every signal, which then satisfies the high-sample tier and other `>=N` sample gates with fabricated data.** On accuracy-load failure all signals get `{accuracy: 0.0, total: 999}` — correct for force-HOLD, but `999 < 7000` so it dodges the high-sample tier; harmless today, but any future gate keyed on `total` (e.g. IC `_IC_ZERO_MIN_SAMPLES=500`, bias `_BIAS_MIN_ACTIVE`) will treat the sentinel as real volume. → Use a sentinel flag (`_gate_forced: True`) rather than a fake sample count, and have downstream gates honor it.

- `portfolio/signal_engine.py:1058 + 1635, 3766, 3883` — **P2: `_TICKER_DISABLED_SIGNALS = _TICKER_DISABLED_BY_HORIZON["_default"]` is a dict of `{ticker: frozenset}` but is indexed with `.get(ticker, ())` returning a frozenset — works, but several call sites assume it's a set of signals.** The name reads like "signals disabled for all tickers" yet it's keyed by ticker; a contributor adding a global entry will silently mis-shape it. → Rename to `_TICKER_DISABLED_BY_TICKER_DEFAULT` and add a module-load assertion that every value is a frozenset (the existing assert at 1024-1033 covers the outer struct, not this alias's usage contract).

- `portfolio/accuracy_stats.py:225, 1358, 695` — **P2: `change_pct = outcome.get("change_pct", 0)` defaults missing data to 0 (neutral) in three hot accuracy functions but `consensus_accuracy`/`signal_accuracy` then pass it to `_vote_correct` which treats 0 as neutral — fine — yet `signal_accuracy_by_regime` (1358) and the EWMA path (344) do the same with no `null_change_pct_skipped` telemetry.** A backfill regression writing `change_pct: null` would silently shrink every regime/EWMA sample set with no log. → Centralize on `_vote_correct(vote, outcome.get("change_pct"))` (pass None through) and add the same skip-count logging the main path has.

- `portfolio/ic_computation.py:127-128` — **P2: `ic_buy`/`ic_sell` are *mean returns*, not ICs, but stored under IC-named keys.** `ic_buy = mean(buy_returns)` is an average return magnitude, not a rank correlation; any consumer reading `ic_buy` as a coefficient (range/scale differs by orders of magnitude) will mis-weight. → Rename to `avg_buy_return`/`avg_sell_return`.

- `portfolio/signal_weights.py:25-103` — **P2: `SignalWeightManager` is dead/abandoned but still ships a non-atomic-safe single-writer assumption.** `outcome_tracker.py:510-513` documents MWU was removed as dead code, yet this class persists and its docstring promises single-threaded use; if anything re-wires it into the (multi-process) backfill it will lose writes. → Either delete the module or add the `threading.Lock` + cross-process file lock its own docstring says is required before reuse.

- `portfolio/signal_engine.py:1764` — **P2: dynamic correlation grouping drops near-constant signals via `df.std() > 0.01` computed on the BUY/HOLD/SELL→{1,0,-1} encoding,** which can erroneously keep a signal that is 99% HOLD with a single flip (std>0.01 on a long vector is trivially exceeded) or drop a genuinely active but balanced signal — the threshold is on the wrong statistic for agreement-based clustering. → Gate on activation count (non-HOLD votes ≥ `_DYNAMIC_CORR_MIN_PAIRS`) rather than vote-encoding std.

- `portfolio/outcome_tracker.py:160-166` — **P2: SQLite dual-write failure in `log_signal_snapshot` is only WARN-logged; JSONL and SQLite can silently diverge** and `load_entries()` *prefers* SQLite (`accuracy_stats.py:147-153`), so a persistently-failing SQLite write means accuracy is computed off a stale DB while JSONL (the supposed source of truth) silently grows ahead. → On repeated SQLite write failure, record a critical_errors entry (the project's silent-divergence guard) rather than only debug/warn.

- `portfolio/forecast_accuracy.py:357-369` — **P2: `_lookup_price_at_time` does a full linear scan of `price_snapshots_hourly.jsonl` per prediction with no early-exit,** O(predictions × snapshots); fine now but is the same unbounded-rescan pattern the rest of the subsystem fought (BUG-178). → Index snapshots by hour bucket once per backfill call.
