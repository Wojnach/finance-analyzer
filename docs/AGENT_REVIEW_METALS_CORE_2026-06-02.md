## P0 — money / safety
- `data/metals_swing_trader.py:2760`: **[REPEAT]** P0: stop-sell limit buffer still 1% — wick-bypass. `sell_price = round(trigger_price * 0.99, 2)` → the hardware stop's SELL limit sits only 1% below trigger. On a fast leveraged wick the MM-quote trigger fires but the limit gaps unfilled → naked 5x position. grid_fisher (`*0.995`) and fin_snipe (`HARD_STOP_SELL_BUFFER_PCT=0.01`) share the thin buffer; the swing trader (primary entry path) was never widened toward the 3% class. FIX: widen to ~3% (`trigger_price * 0.97`) for swing silver/gold, or place a marketable limit anchored to live bid − 3%.
- `data/metals_loop.py:1086 (+1088,1417)`: **[REPEAT]** P0: `_has_active_silver()`/`_get_active_silver()` read ONLY the legacy POSITIONS dict, so every exit-side fast-tick alert silently no-ops for swing-managed silver. POSITIONS has 3 hardcoded keys; on ingest the swing trader flips the matching legacy entry to `active=False` (metals_swing_trader.py:1125, reason `migrated_to_swing`). Result: for any swing-owned silver warrant `_has_active_silver()` is False → `_silver_fast_tick()` returns at 1418 → the entire −3%/−5%/−7%/−10%/−12.5% ladder (485-491) AND the 3-min velocity alert never fire. Entry-side ticks DO run (position-agnostic, 508) → protection dead while opportunism alive. Compounds with 2760: a sub-60s flush blows the thin hardware-stop limit AND gets no 10s fast-tick catch. FIX: source active-silver from the swing-trader state file (positions where `underlying=="XAG-USD"`), unioned with legacy POSITIONS.
- `data/metals_loop.py:6765 (+7703-7744)`: **[REPEAT]** P0: raw `subprocess.Popen` of claude with NO auth-failure scan; exit-0 auth outage silently records "success". The completion handler imports `_parse_claude_json_stdout` only for token/cost; reads the log tail (7722) but never scans it. `claude -p` printed "Not logged in" while exiting 0 during Mar-Apr. The correct guard exists (`claude_gate.detect_auth_failure` + `invoke_claude_text` returning `status=="auth_error"`, used by iskbets.py:332) — metals_loop bypasses all of it. FIX: route metals invocation through `claude_gate.invoke_claude_text`, or at minimum call `detect_auth_failure(tail, "metals_loop_complete")` in the completion branch at 7723 and skip trade-queue processing on auth_error.
- `data/metals_loop.py:1051 / 1056-1064`: **[REPEAT]** P0: cycle overrun yields zero fast-ticks and the velocity deque has no time anchoring. On overrun (`remaining<=0`) the while-loop breaks immediately → no fast-ticks that cycle. `_silver_fast_prices` (873) is a plain `deque(maxlen=18)` with no timestamps, yet the velocity alert (1485-1488) computes `(price-oldest)/oldest` labeled "% in 180s" (1497). When ticks are dropped (LLM-heavy cycles) the 18 samples span far more than 3 min → false "RAPID DROP" on slow moves + real flushes mis-windowed. FIX: store `(monotonic_ts, price)` tuples; compute velocity over actual elapsed span; alert only when the window's real duration ≈ `WINDOW*INTERVAL`.
