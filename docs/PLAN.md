# Stale Data Cleanup Plan

## Problem
Multiple scripts in `data/` use hardcoded position data (entry prices, units, leverage, active status)
that doesn't match actual state in `metals_positions_state.json`. All positions were sold (Mar 2-3, 2026),
but scripts continue reporting P&L and sending Telegram messages as if positions are still open.

## Root Cause
Scripts were written during active trading and never updated to read from canonical state files.
The `data/` directory is gitignored so these scripts are untracked and drift silently.

## Execution Plan (6 Batches)

### Batch 1: Delete Dangerous Scripts [DONE]
Removed 6 scripts that had hardcoded `active: True` positions and could execute real trades:
- `metals_trader.py` — autonomous trader with real Avanza sell execution
- `metals_monitor.py` — hardcoded gold/silver monitoring with wrong P&L
- `metals_monitor_v2.py` — v2 with 3 hardcoded positions
- `set_stoploss.py` — stop-loss placer with wrong prices
- `silver_tg_send.py` — generated by silver_monitor Claude, hardcoded values
- `send_tg_metals.py` — Telegram sender with stale data

### Batch 2: Fix silver_monitor.py [DONE]
- Added `_has_active_silver_position()` guard — exits immediately if no active position
- Replaced hardcoded `REFERENCE_PRICE`, `LEVERAGE`, `POSITION_SEK` with `_load_position_params()`
  that reads from `metals_positions_state.json`
- Falls back to defaults if state file unavailable

### Batch 3: Fix metals_loop.py POSITIONS_DEFAULTS [DONE]
Changed all defaults from `active: True` with stale values to `active: False` with zeroed values.
The `_load_positions()` function correctly reads from state file, so defaults only matter on first run
or if state file is missing — safe defaults prevent phantom trading.

### Batch 4: Fix metals_risk.py LEVERAGE_MAP [DONE]
Replaced hardcoded `LEVERAGE_MAP` dict with `_load_leverage_map()` that reads from
`metals_context.json` warrant_catalog. Falls back to `_LEVERAGE_DEFAULTS` dict.
Added `silver_sg` to defaults (was missing).

### Batch 5: Delete Stale Utility Scripts [DONE]
Removed 8 stale files:
- `_quick_status.py`, `_quick_check.py` — ad-hoc debugging scripts
- `analyze_daily_ranges.py` — one-time analysis
- `metals_accuracy_review.py` — stale accuracy review
- `_buy_silver_v2.py`, `_set_silver_stops.py` — order scripts with wrong prices
- `metals_decision_log.py`, `metals_decision_temp.json` — abandoned decision logging

### Batch 6: Document Claude CLI Dependencies [DONE]

# Telegram Signal Quality & Probabilities (2026-03-05)

## Problem
Telegram "AUTO BUY" messages fire on thin consensus (e.g., 2B/4S/19H) and omit probability/accuracy context, so alerts read as strong buys even when signals are weak or historically unreliable.

## Plan
1. Tighten BUY gating in `portfolio/autonomous.py` ticker prediction: require minimum BUY votes and BUY dominance over SELL before classifying as BUY. Aim: suppress BUY recommendations when raw vote count is weak/inverted despite weighted flips.
2. Expose probability & accuracy in Telegram output (signals mode): append per-ticker conviction (probability) and surface consensus accuracy so recipients see both likelihood and model hit-rate.
3. Tests: add/adjust unit tests for `_ticker_prediction` gating and Telegram line formatting to cover new fields and thresholds.

## Risks
- Over-gating could silence legitimate contrarian BUYs driven by accurate inverted signals; mitigate by choosing moderate thresholds and keeping SELL behavior unchanged.
- Telegram message length could approach the 4096 limit; must keep additions compact.
- Tests may assume old formatting; need targeted updates to avoid brittle snapshots.

## Execution Order
1. Implement BUY gating logic and ensure prediction output remains backward-compatible except for classification changes.
2. Add probability/accuracy fields to Telegram builder with concise formatting.
3. Update/create unit tests for gating and message formatting.
4. Run targeted tests (`tests/test_autonomous.py`, `tests/test_telegram_formatting.py`), then quick lint/check if available.
5. Review sample Telegram payload to confirm readability and length.
