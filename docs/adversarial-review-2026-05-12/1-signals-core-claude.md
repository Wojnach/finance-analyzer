# Claude adversarial review: signals-core (2026-05-12)

## Summary

The signals-core subsystem has matured significantly since the 2026-05-11 review — prior P0s on the SQLite-connection-per-thread bug, the raw `open()/json.load` in `signal_decay_alert.py`, and the missing intra-process lock in `signal_history.py` are now either FIXED or have explicit suppression comments documenting that the residual risk is accepted. However, several P0/P1 correctness defects remain: (1) `signal_db.SignalDB` still serves a single connection from any thread that touches it, and `outcome_tracker.backfill_outcomes` opens that connection on the dual-write hot path while `accuracy_stats.load_entries` opens *another* connection in parallel ticker workers — concurrent writes/reads on the same `sqlite3.Connection` are documented unsafe by the stdlib; (2) `ic_computation._spearman_rank_correlation` still receives temporally-unsorted votes/returns, so the "rolling IC / ICIR" reported to `_compute_ic_mult` is computed on a random ordering of samples and the resulting ICIR is meaningless; (3) `signal_history` has only an intra-process lock — the metals_loop, crypto_loop, and main loop all import this module and race on the shared `signal_history.jsonl`; (4) `accuracy_degradation._diff_against_baseline` reloads the entire signal log inside the throttle gate even though `save_full_accuracy_snapshot` just loaded the same data minutes before — and the IC and degradation caches both have NO mtime check against the source data.

## P0 — Blockers (production breakage / data loss / silent wrong trades)

- **portfolio/signal_db.py:31-37** — Single shared `sqlite3.Connection` (`self._conn`) returned by `_get_conn()` is used from concurrent threads. `accuracy_stats.load_entries()` constructs `SignalDB()` in `get_or_compute_*` callers that run inside the 8-worker `ThreadPoolExecutor` (`signal_engine.generate_signal` → `get_or_compute_accuracy` → `signal_accuracy` → `load_entries` → `SignalDB().load_entries()`), AND `outcome_tracker.backfill_outcomes` holds an open `SignalDB` while the main loop opens its own per call. `sqlite3.Connection` objects are not safe to share across threads unless `check_same_thread=False` is passed AND the application serializes all access — neither is done. Result: `ProgrammingError: SQLite objects created in a thread can only be used in that same thread` raises intermittently; the broad `except Exception` in `accuracy_stats.load_entries` line 157 silently falls through to the JSONL tail reader. Why it bites: the JSONL path reads only the last 50,000 entries, so post-fallback signal accuracy is computed on a truncated window that excludes outcomes older than ~2 months. Worse, because the exception is logged at `logger.debug`, the fallback is invisible in normal operation. Fix: open one connection per call (instantiate `SignalDB()` inside `load_entries`, close in finally), OR pass `check_same_thread=False` + serialize via a module-level lock around every `execute`. The current "open once per `SignalDB` instance" design forces every caller to know the lifetime rules.

- **portfolio/ic_computation.py:73-147** — `compute_signal_ic` collects `(vote_num, change_pct)` pairs by iterating `entries` then per-ticker — but `entries` is whatever order `accuracy_stats.load_entries()` returns. SQLite returns rows `ORDER BY ts` (signal_db.py:188) but the per-ticker inner loop interleaves five tickers per snapshot, so the *flattened* votes list is ordered by (ts, ticker) — NOT strictly chronological per signal. `_rolling_ic(votes, returns, window=50)` then computes Spearman over rolling 50-sample windows that contain blocks of contemporaneous BTC/ETH/XAU/XAG/MSTR samples rather than a time series. The resulting `icir = ic_mean / ic_std` is reported as a temporal stability metric, fed into `signal_engine._compute_ic_mult`, and used as a weight multiplier on every signal vote in production. Why it bites: a signal can have ICIR=2.5 reported and ICIR=0.1 actual just due to ordering — the production weight multiplier `ic_mult` derived from this is a confused metric. Fix: sort `votes`/`returns` by entry `ts` before passing to `_rolling_ic`. Better: maintain (ts, vote, return) tuples and group by ticker for per-ticker rolling IC. The function was flagged in the 2026-05-11 review — comment at line 73-147 was not addressed.

- **portfolio/signal_history.py:64-98** — `update_history` uses `_history_lock = threading.Lock()` (process-local). The metals_loop, crypto_loop, and oil_loop subprocesses also import `signal_history` (it's imported by `accuracy_stats` and `reporting` indirectly) and write to the same `data/signal_history.jsonl`. Per CLAUDE.md, these loops run as separate scheduled tasks (PF-MetalsLoop, PF-CryptoLoop, PF-OilLoop) AND main PF-DataLoop. Each process holds its own `threading.Lock` — they don't see each other. Two concurrent `_save_history` calls last-writer-wins because the read-modify-write straddles `_load_history()` → `_save_history()`, and `atomic_write_jsonl` only guarantees atomicity of the write itself. The 2026-05-11 review fix added the intra-process lock; the cross-process race remains. Why it bites: persistence scoring and noisy-signal detection feed into `_apply_persistence_filter`, which gates whether a vote participates in consensus. Lost writes mean a ticker's votes never accumulate the streak count needed to pass the filter for 1-3 cycles after the race — silent vote suppression. Fix: use `file_utils` sidecar lockfile pattern (`_hold_signal_log_lock` in outcome_tracker.py:281 is the template). Or accept the risk and stop writing from non-main loops — but no caller audit appears to have been done.

## P1 — High (will cause incidents)

- **portfolio/signal_decay_alert.py:34-39** — Still uses raw `with open(accuracy_cache_path, encoding="utf-8") as f: cache = json.load(f)` on a target that is written by `accuracy_stats.write_accuracy_cache` via `atomic_write_json`. Per CLAUDE.md rule 4 ("Atomic I/O only … Never raw `json.loads(open(...).read())`"), this is a defect. The previous review flagged it. The risk is small (atomic write means the file is either old-valid or new-valid, never torn), but the rule is "no exceptions" because the audit cost of distinguishing safe from unsafe raw reads is too high. Fix: `from portfolio.file_utils import load_json; cache = load_json(accuracy_cache_path)`.

- **portfolio/accuracy_degradation.py:436-451** — `_diff_against_baseline` calls `load_entries()` *AGAIN* after `save_full_accuracy_snapshot` just did the same minutes earlier (line 145). `save_full_accuracy_snapshot` is invoked once per day by `maybe_save_daily_snapshot`; `_diff_against_baseline` is invoked hourly (HOURLY_THROTTLE_S). The hourly call rescans 50,000+ entries every check. Comment claims the share is "mandatory" but the actual share happens only within one diff invocation — not across the snapshot+diff sequence. Why it bites: with the SQLite-load-races-with-writes P0 above, every hourly call risks reproducing the SQLite ProgrammingError and falling back to JSONL tail. Fix: cache `load_entries()` at module level for the duration of one main-loop cycle (300s TTL would suffice).

- **portfolio/ic_computation.py:255-262** — `load_cached_ic` returns `None` when `time.time() - cache.get("time", 0) > IC_CACHE_TTL`. But cache invalidation has NO mtime check against `signal_log.db` or `signal_log.jsonl`. After `outcome_tracker.backfill_outcomes` writes new outcomes (with up to several hundred new `change_pct` entries), the IC cache continues serving stale IC values for up to 3600s. `outcome_tracker` only invalidates `signal_utility_cache` (line 569) and NOT `ic_cache` — which is consumed by `signal_engine._compute_ic_mult` to multiply every signal weight. Why it bites: after a daily backfill, IC-based weights are stale for an hour. The IC cache rebuild costs seconds, so the invalidate call is cheap. Fix: add `from portfolio.ic_computation import invalidate_ic_cache; invalidate_ic_cache()` in `outcome_tracker.backfill_outcomes` after the loop completes. The invalidate function does not exist — add it.

- **portfolio/outcome_tracker.py:472** — `change_pct = round(((hist_price - base_price) / base_price) * 100, 2)` assumes `base_price > 0` (guarded line 449) but does NOT guard against `hist_price` being None-coerced or non-finite. `_fetch_historical_price` can return `float("nan")` in pathological yfinance responses. Then `hist_price > 0` may be True but `change_pct` becomes NaN, persists to `outcomes[ticker][h_key]["change_pct"]`, and `accuracy_stats._vote_correct` does `abs(change_pct) < threshold` which evaluates `False` for NaN, then `change_pct > 0` evaluates `False`, then `bool(change_pct < 0)` evaluates `False` — silently labeled "not correct" rather than "skip". Fix: `if hist_price is None or not math.isfinite(hist_price): continue`.

- **portfolio/signal_engine.py:3886-3889** — Fail-closed branch substitutes `{"accuracy": 0.0, "total": 999}` for every signal when accuracy stats load fails. The 47% accuracy gate then triggers — but the directional rescue path at line 2486-2504 still fires and rescues SELL votes when `stats.get("sell_accuracy", 0.0) >= 0.55`. Since the fail-closed dict has no `sell_accuracy` key, `stats.get("sell_accuracy", 0.0)` returns 0.0, and `total_sell` returns 0 — so rescue is denied. But the next branch at line 2513 `dir_acc = stats.get("buy_accuracy", acc)` falls back to `acc=0.0` → `dir_n=0`, which skips the directional gate (line 2519 requires `dir_n >= _DIRECTIONAL_GATE_MIN_SAMPLES`). Then line 2526 falls through to `weight = 0.5` (samples<20). Result: when `_accuracy_failed=True`, every voting signal contributes weight=0.5, NOT zero — the system still produces a directional consensus during a total accuracy-load failure. Why it bites: this contradicts the stated "fail-closed" intent. Fix: when `_accuracy_failed=True`, set `accuracy_data = {sig: {"accuracy": 0.0, "total": 999, "buy_accuracy": 0.0, "sell_accuracy": 0.0, "total_buy": 999, "total_sell": 999} for sig in SIGNAL_NAMES}` so the directional gate also fires.

- **portfolio/linear_factor.py:107-110** — Training R² uses `y_pred = X_arr @ beta + intercept` against the *training* set, calling it `r_squared` and persisting it in the model file. `signal_engine`'s linear factor confirmation boost (signal_engine.py:4142) gates on `lf_conf > 0.3` which is derived from `score_to_action` magnitude — not r_squared — so the misleading metric isn't directly toxic to live trades, but it's surfaced to dashboards and Layer 2 context as the model's predictive power. With ridge regularization and z-scored features, in-sample R² is essentially the upper bound; the walk-forward `oos_corr` is what should be reported. Why it bites: operators see "Linear factor R²=0.42" and over-trust it. Fix: rename `r_squared` to `train_r_squared` and compute an actual OOS R² (or refuse to report a single R² without specifying split).

## P2 — Medium (correctness / robustness)

- **portfolio/feature_normalizer.py:32, 43-51** — Module-level `_buffers: dict` is a singleton with no locking. `update()` is called from per-ticker threads (multiple signal modules), and `deque.append` IS thread-safe in CPython, but `_ensure_buffer` does `_buffers[key] = deque(...)` non-atomically — two threads racing on a never-before-seen `(ticker, indicator)` pair both create a deque, one wins, the other's first sample is lost. Trivial impact (1 sample, 100-sample window) but the pattern is broken in cross-process scope: every process gets its own buffers, so the z-score from the metals_loop differs from main loop's z-score for the same `(ticker, indicator)`. Fix: persist buffers to disk per-ticker, or document that this is process-local intentionally.

- **portfolio/accuracy_stats.py:182-186** — `_vote_correct` returns `bool(vote == "SELL" and change_pct < 0)`. The `bool(...)` wrapper is dead code (the expression is already bool); but more importantly, when `vote == "BUY"` and `change_pct < 0` the function returns `False` (not `None`). That's fine, but when `vote` is anything other than `"BUY"`/`"SELL"` (e.g., a typo'd `"buy"` or `"Hold"`), the function returns `False` — silently counted as incorrect. Fix: explicit `if vote not in ("BUY","SELL"): return None`.

- **portfolio/accuracy_stats.py:225** — In `signal_accuracy`, `change_pct = outcome.get("change_pct", 0)` defaults to 0 if missing, but the subsequent `_vote_correct(vote, change_pct)` would treat that 0 as a neutral skip via `abs(0) < threshold`. But further down line 226 explicitly handles `change_pct is None` only — meaning a missing key (default 0) is silently dropped while a present-but-None is logged. Inconsistent handling: missing key SHOULD be just as suspect as None. Fix: `change_pct = outcome.get("change_pct"); if change_pct is None: null_change_pct_skipped += 1; ...`.

- **portfolio/accuracy_stats.py:1601-1607** — `_find_snapshot_near` indent at line 1605 is wrong: lines 1605 & 1606 are inside the `if delta <= ... and (...)` block, but the indentation has 12 spaces vs 8 for `delta = ...` — this works because Python is flexible about indent depth as long as it's consistent within the block, but it's misleading. (Verified by reading the file directly — see lines 1604-1606.) Cosmetic, but a future edit could insert a line at the wrong level.

- **portfolio/signal_weight_optimizer.py:121-135** — Weight stability formula `1 - (std / mean_abs)` can return negative values when `std > mean_abs` (a sign-flipping signal). Capped at `max(0.0, ...)`. But the recommended_weight uses `mean(values)` which is unbiased for a sign-flipping signal — i.e., a signal whose weight is +0.1 in half the windows and -0.1 in the other half gets recommended_weight ≈ 0. Then ranking by `|mean_weight| * stability` correctly de-ranks it, but the produced `recommended_weights[sig] = 0` could be loaded by `LinearFactorModel.load()` and used as the production weight — zero weight is correct but the *zero* came from sign-flipping noise, not from a stable zero edge. Fix: log warning when `std > mean_abs` and exclude from recommended_weights, or recommend `0` only when both `|mean| ≈ 0` AND `std ≈ 0`.

- **portfolio/outcome_tracker.py:458** — `cache_key = (ticker, int(target_ts // 3600))` rounds to 1-hour buckets. Two snapshots at 14:01 UTC and 14:59 UTC both query the same hour bucket — sharing one historical price across them is intentional, but `_fetch_historical_price` for the 14:01 snapshot is anchored at `target_ts = entry_ts + 86400`. If both snapshots share the same hour, *one* of them will be slightly mis-aligned. With Binance 1h candles this is acceptable but it's an unstated assumption — the hour boundary is at :00, not at the snapshot time. Fix: document, or use 15-minute buckets for stocks (Alpaca minute bars).

- **portfolio/accuracy_degradation.py:558-573** — `_binomial_diff_se_pp` uses naive `p*(1-p)/n` — fine for small p_hat away from 0/1 but at p_hat near 0 or 1 (which happens for "always BUY" signals like calendar, fear_greed) the SE underestimates. Wilson score interval would be more correct. Practical impact minimal because absolute floor gates at 50%. Acceptable.

- **portfolio/signal_postmortem.py:121-180** — `compute_vote_correlation` walks every entry and every signal pair O(n_signals²) per entry. With ~52 registered signals (~26 active per asset), that's 1326 pairs per ticker per entry — for 50K entries it's a hot loop. Currently called from `generate_postmortem` which is invoked by the after-hours research path, so not on the cycle hot path, but the function has no time budget. A simple time limit or per-ticker subsample would protect future use.

- **portfolio/signal_engine.py:1582** — Loads pandas DataFrame from `rows` (potentially 50K+ rows × 30+ signals) inside `_compute_dynamic_correlation_groups`, which is cached for 7200s. First-run cost is high (~2-5s). It's wrapped in `_cached` so subsequent calls are O(1) — acceptable, but the first call after a process restart blocks ticker threads. The 2026-04-16 BUG-178 thundering-herd lock fix doesn't protect this path because `_cached` doesn't use `_accuracy_compute_lock`. Add the lock or accept the 5-second hit.

- **portfolio/train_signal_weights.py:73** — `change_pct = outcome if isinstance(outcome, (int, float)) else outcome.get("change_pct")` — the `isinstance` short-circuit accepts a bare float `outcome` as the change_pct. The signal_log format uses `{ts, price_usd, change_pct}` dicts though; the float branch is for an older format that should no longer appear. Defensive code, but if some test fixture or old JSONL row leaks through, this silently mis-interprets a price_usd as change_pct.

- **portfolio/forecast_accuracy.py:355-366** — `_lookup_price_at_time` linear-scans the entire `price_snapshots_hourly.jsonl` for every prediction backfill — O(predictions × snapshots) = potentially millions of ops. Acceptable for the daily backfill job but the function is also called from `backfill_forecast_outcomes` which processes up to 500 entries per call. For a snapshot file with 10K rows, that's 5M comparisons. Fix: bisect by timestamp if the file is time-sorted.

## P3 — Low (style / dead code / minor)

- **portfolio/signal_weights.py:25-121** — `SignalWeightManager` class is dead code. The `outcome_tracker.backfill_outcomes` previously called `batch_update` but that path was removed (comment at outcome_tracker.py:497-500). The class still loads/saves `data/signal_weights.json` and nothing reads it. Remove or document why retention is intentional.

- **portfolio/signal_engine.py:294-298** — Trailing `return "HOLD", 0.0` after `if abs(current_hist) >= MACD_DEAD_ZONE_MAGNITUDE_THRESHOLD: return "HOLD", 0.0` — both branches return same. The conditional is essentially documentation. Acceptable but flag for future cleanup.

- **portfolio/accuracy_stats.py:177** — `threshold = min_change_pct if min_change_pct is not None else _MIN_CHANGE_PCT` is fine but the parameter `min_change_pct` is never overridden by any caller (verified via Grep). Dead parameter.

- **portfolio/ticker_accuracy.py:225** — `forecast_p_up = 0.5 + min(max(pct_move * 0.1, -0.3), 0.3)` — comment says "cap at 0.2-0.8" but the clamp produces 0.2-0.8. The 0.1 scale factor is magic. Document the calibration source (or this is overfit to a single observation).

- **portfolio/feature_normalizer.py:69-71** — `if std < 1e-12: return 0.0` — returning 0 when std is degenerate is correct, but the function signature doesn't make clear that "0.0 = no signal" vs "0.0 = exactly at mean". A returned-NaN sentinel would be cleaner.

- **portfolio/linear_factor.py:160** — `confidence = min(abs(score) / (threshold * 5), 0.8)` — magic factor `5` and cap `0.8`. The comment says "Confidence proportional to score magnitude, capped at 0.8" but doesn't justify the `5`.

- **portfolio/short_horizon.py:22-33** — `_TOD_SCALE` is a hardcoded dict of measured time-of-day scaling factors. No version/timestamp on the measurement. Add a comment `# Measured YYYY-MM-DD on N samples — refresh quarterly`.

- **portfolio/signal_db.py:33** — `timeout=10` second timeout on SQLite open. With write contention and an active backfill, 10s can be insufficient. Bump to 30s, or document the bound.

## Status of prior P0s (2026-05-11)

- **signal_db.py:31-37 (shared sqlite3 connection across 8-worker pool)** — STILL PRESENT (with partial mitigation). The code stores `self._conn` and re-uses it across calls. Test `test_signal_db.py` may verify within-thread use; the in-production callsite (`accuracy_stats.load_entries`, `outcome_tracker.backfill_outcomes`) instantiates a new `SignalDB` per call and closes it in finally — but the prior P0 was about *concurrent* calls. With each per-call instance opening its own connection, the original race no longer literally exists IF no caller passes a `SignalDB` instance between threads. I could not find a caller doing so via Grep (`SignalDB` is always instantiated locally). On that basis, downgrade to "FIXED IN PRACTICE BUT STRUCTURAL RISK REMAINS" — the class still exposes the foot-gun. Recommendation in this review's P0: refactor to per-call connection or document `check_same_thread=False` requirement.

- **ic_computation.py:73-147 (IC samples not sorted chronologically; mislabel of avg returns as IC)** — STILL PRESENT. Re-inspection confirms `signal_data[sig_name]["votes"].append(vote_num)` and `["returns"].append(change_pct)` push in (entry, ticker) iteration order, not by (signal, ticker, ts). The `ic_buy` / `ic_sell` fields at lines 127-128 are now correctly labeled as *average* returns (not IC), but the rolling-IC / ICIR computation at lines 130-136 still treats unsorted samples as a time series. **Not fixed.**

- **signal_decay_alert.py:34-39 (raw open()/json.load on atomic-write target)** — STILL PRESENT. Code unchanged.

- **signal_history.py:64-98 (intra-process lock only, cross-process race)** — PARTIAL.
