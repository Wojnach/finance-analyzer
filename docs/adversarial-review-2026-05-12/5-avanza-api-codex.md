# Codex adversarial review: avanza-api
## Summary
I did not find a repeat of the March 3 stop-loss endpoint misuse in the legacy session path; `portfolio/avanza_session.py` is still calling `/_api/trading/stoploss/new`. The blockers are elsewhere: sync Playwright still leaks into public helpers that async callers can hit, and the newer `portfolio.avanza` package has dropped account-scoping and execution invariants that the legacy path already enforces.

## P0 — Blockers
- Sync Playwright is still reachable from public helpers on the caller thread, so the 2026-05-11 “Sync API inside asyncio loop” failure can recur. `portfolio/avanza_session.py:145-148,180-195,238-379` starts and uses `sync_playwright()` inline; `portfolio/avanza_client.py:65-79,148-178` auto-selects that path when a BankID session exists; `portfolio/avanza/scanner.py:53-56,73-90` silently falls back to the same session API when `AvanzaClient` was not preinitialized. `portfolio/avanza_account_check.py:158-177` already contains the correct mitigation (single-worker thread offload), which makes the missing protection everywhere else a real regression.
- The unified `portfolio.avanza` trading path can route live mutations to any configured or caller-supplied account. `portfolio/avanza/client.py:59-70` accepts `avanza.account_id` verbatim, and `portfolio/avanza/trading.py:80-92,128-138,159-161,239-278,351-355` uses `account_id or client.account_id` with no whitelist check. That bypasses the guard present in `portfolio/avanza_session.py:590-592,749-751`, so a config mistake or explicit override can place, modify, cancel, or stop-loss against KF/pension/other accounts.

## P1 — High
- Read-side account scoping is inconsistent and can reintroduce the pension account depending on auth path. `portfolio/avanza_client.py:172-179` returns `portfolio.avanza_session.get_positions()` directly when BankID auth is live, but `portfolio/avanza_session.py:682-717` does not filter `ALLOWED_ACCOUNT_IDS` at all. The typed read API also defaults to “all accounts” in `portfolio/avanza/account.py:27-61,97-140`. The same process can therefore hide the pension account under TOTP and leak it back in under session auth or typed calls.
- Order and stop-loss placement do not enforce legal ticks or integer volumes anywhere in the live path. `portfolio/avanza_session.py:608-615,777-794` and `portfolio/avanza/trading.py:84-92,132-138,256-278` forward raw `price`, `trigger_price`, `sell_price`, and `volume` straight through. `portfolio/avanza/tick_rules.py:53-99` exists but is never used here, and `volume` is only checked with `< 1`, so floats pass. That violates the project rule to round to legal tick before order entry.
- `portfolio/avanza_session.py:648-664` fails open on open-order reads: both the primary endpoint and the fallback path collapse transient `RuntimeError`s to `[]`. Unlike stop-losses, there is no strict variant. Any reconciliation or tracker logic using this helper cannot distinguish “no open orders” from “Avanza did not answer”, which is exactly how duplicate orders and phantom clears happen after partial/truncated reads.
- `portfolio/avanza_account_check.py:52,109-116,285-317` has drifted into a no-op category guard. `DISALLOWED_CATEGORY_FRAGMENTS` is empty, so `_category_disallowed()` can never fire and `verify_default_account()` effectively only proves that the account ID exists. Given the stated allowed-category policy (`INVESTERINGSSPARKONTO`, `KAPITALFORSAKRING`, `PENSION`), the startup safety check no longer enforces anything meaningful.

## P2 — Medium
- `portfolio/avanza/types.py` is out of sync with the response shapes the rest of this subsystem already knows about. `SearchHit.from_api()` expects `orderbookId`/`name` (`types.py:364-371`) while scanner consumes `orderBookId`/`title` (`portfolio/avanza/scanner.py:188,197-208`); `InstrumentInfo.from_api()` expects top-level `leverage`/`barrier`/`underlyingPrice` (`types.py:494-506`) while live parsing uses `keyIndicators` and `underlying.quote.last` (`scanner.py:237-250`); `AccountCash.from_api()` ignores the alias cash fields that `portfolio/avanza_session.py:480-490` already had to add; `StopLoss.from_api()` drops nested `account.id` ownership (`types.py:333-345` vs `portfolio/avanza_session.py:989-995,1042-1043`). The typed layer can silently zero out leverage/cash, lose IDs, and lose the owning account for stop-losses.

## P3 — Low
- The BankID scanner fallback does not actually apply the requested instrument type and never paginates the result universe. `portfolio/avanza/scanner.py:77-79,173-201` posts only `{"query","limit"}` to `/_api/search/filtered-search`; when `instrument_type=None` it sends the same search twice and deduplicates client-side, and when a type is requested the server-side filter is still absent. That is both extra request pressure and a candidate-selection bug.
- Sensitive identifiers are logged more broadly than they need to be. `portfolio/avanza_orders.py:135-142` logs the live `confirm_token` at INFO, `portfolio/avanza/auth.py:110-113` logs `customer_id`, and full account IDs are logged/alerted in `portfolio/avanza_client.py:272,395` and `portfolio/avanza_account_check.py:275-299`. The token is a short-lived execution secret, and the IDs are avoidable PII.

## Tests missing
- Async-safety tests that call `avanza_client.get_price()`, `avanza_client.get_positions()`, and `scanner.scan_instruments()` from inside a running asyncio loop and assert they never invoke sync Playwright on the event-loop thread.
- Account-routing tests proving every mutator in `portfolio.avanza.trading` rejects non-whitelisted accounts and that read helpers consistently suppress the pension account.
- Order-invariant tests that feed illegal ticks and non-integer volumes through order, modify-order, stop-loss, and trailing-stop entry points and assert preflight rejection/rounding before any API call.
- Reconciliation tests that force both open-order endpoints to fail or truncate and assert callers get a fail-closed signal instead of `[]`.
- Response-shape fixtures for search hits, market-guide payloads, categorized accounts, and stop-loss lists matching the fields already handled elsewhere (`orderBookId`, `title`, `keyIndicators`, nested `account.id`, alias cash fields).
- Log-redaction tests ensuring confirmation tokens, customer IDs, and full account IDs do not appear in normal INFO/ERROR logs.
