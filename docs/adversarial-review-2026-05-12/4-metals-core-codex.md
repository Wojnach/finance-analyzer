# Codex adversarial review: metals-core
## Summary
Seven actionable findings in the reviewed scope: 1 blocker, 4 high, 1 medium, 1 low. The most serious defect is `grid_fisher`’s EOD-flat path: it submits liquidation sells that are never tracked back into state, so repeated 60s cycles can re-send full-size closes after full or partial fills.

I did not find a concrete ORB look-ahead bug in the reviewed ORB files; the walk-forward code trains on prior days only (`portfolio/orb_backtest.py:4-5, 340-351`, `portfolio/orb_predictor.py:305-328`). `portfolio/fin_fish_manager.py` is not present in this checkout, so it could not be reviewed.

## P0 — Blockers
- **`grid_fisher` can re-send EOD liquidation sells against stale inventory, including after partial fills.** `tick()` enters the EOD-flat path every cycle once the close window opens (`portfolio/grid_fisher.py:1247-1250`). `eod_market_flat()` then places a full-size sell for `inst.inventory_units` but never stores the returned order id and never decrements local inventory (`portfolio/grid_fisher.py:1393-1438`). Reconciliation only marks sell fills for tracked ladder orders (`portfolio/grid_fisher.py:666-669`), and unmatched live-vs-local inventory is only logged, not corrected (`portfolio/grid_fisher.py:686-691`). Result: a full fill leaves stale positive `inventory_units`, and a partial fill can make the next cycle submit another sell for the original full size.

## P1 — High
- **Commodity-warrant session hours are wrong in both `elongir` and `golddigger`.** `elongir` hard-codes `08:30-21:30` in config and enforces exactly that window (`portfolio/elongir/config.py:52-55, 94-97`, `portfolio/elongir/risk.py:132-135`). `golddigger` does the same (`portfolio/golddigger/config.py:47-50`, `portfolio/golddigger/bot.py:94-103`). Per your project rule, Avanza commodity warrants trade `08:15-21:55` Stockholm time, so both bots start 15 minutes late and flatten 25 minutes early every day.
- **There is no in-scope shared broker/order lock between the metals loop stack and the separate `golddigger` process.** `data/metals_loop.py` only protects one metals-loop instance with its own singleton lock (`data/metals_loop.py:892-930`) and runs both swing execution and `grid_fisher.tick()` against XAG/XAU in the same process (`data/metals_loop.py:7481, 7490-7555`). `golddigger` uses a different singleton lock file (`portfolio/golddigger/runner.py:40, 98-112, 228-230`) and independently calls `place_order()` (`portfolio/golddigger/runner.py:177`). Inside `grid_fisher`, the only lock I found is a thread-local state-file lock, not a broker submission lock (`portfolio/grid_fisher.py:330, 366-367`). That leaves concurrent order submission across processes to luck rather than an enforced critical section or symbol/account partition.
- **`fin_snipe_manager` places new stops without checking MINI barrier proximity.** `_compute_stop_plan()` derives trigger/sell prices solely from `position_average_price` and `current_bid` (`portfolio/fin_snipe_manager.py:529-560`). The snapshot already carries `barrier_level`, `financing_level`, `leverage`, and `parity` (`portfolio/fin_snipe_manager.py:606-609`), but `_compute_stop_plan()` never reads `barrier_level`. The only guard is distance from the current warrant bid (`portfolio/fin_snipe_manager.py:541-549`), which does not protect against arming stops on products already close to knockout.
- **Expired/failed Avanza sessions are swallowed by `grid_fisher` and not surfaced by the parent metals loop.** `_safe_session_call()` converts timeouts/exceptions into `default` after logging (`portfolio/grid_fisher.py:799-842`). If `get_open_orders()` or `get_positions()` fails, `tick()` logs `tick_fetch_degraded`, persists state, and returns `{"error": "reconcile_fetch_failed"}` (`portfolio/grid_fisher.py:1185-1196`). The parent loop calls `grid_fisher.tick(...)` and ignores its return value; it only logs when `tick()` raises (`data/metals_loop.py:7549-7555`). A BankID/session-expiry failure therefore degrades reconciliation and EOD handling without an operator-visible escalation at the orchestration layer.

## P2 — Medium
- **`grid_fisher` violates the “no re-entry on flip without bounce confirmation” rule.** The config and code implement only a time cooldown after a direction flip (`portfolio/grid_fisher_config.py:79-81`, `portfolio/grid_fisher.py:456-476`). Re-arming after a flip is gated only by `inst.in_cooldown(...)` (`portfolio/grid_fisher.py:1302-1315`). I did not find any bounce/price-confirmation check before the same warrant can be armed again once the cooldown expires.

## P3 — Low
- **`metals_loop` still carries a legacy `17:25` close fallback in one end-of-day context path.** The module-level fallback constant is `EOD_HOUR_CET = 17.0` (`data/metals_loop.py:401`). When `EXECUTION_ENGINE_AVAILABLE` is false, `hours_remaining` is computed as `17:25 - cet_hour()` (`data/metals_loop.py:5955-5956`), even though the same context advertises `"market_close_cet": "21:55"` (`data/metals_loop.py:5965`). This does not appear to be the primary trade gate, but it can poison downstream end-of-day reasoning when that fallback path is active.

## Tests missing
- A regression test that calls `grid_fisher.tick()` through the EOD window with a full fill and a partial fill, and proves `inventory_units` reaches zero exactly once without duplicate close orders.
- Session-boundary tests for `elongir` and `golddigger` covering `08:15` open, `21:55` close, and Europe/Stockholm DST transitions.
- An integration test where `grid_fisher.get_open_orders()`/`get_positions()` fails from session expiry and the parent metals loop must raise an operator-visible alert or halt new placement.
- A stop-placement test for MINI products near barrier that proves `fin_snipe_manager` refuses to arm a new stop when knockout distance is below threshold.
- A multi-process test proving `metals_loop`/`grid_fisher` and `golddigger` cannot concurrently place orders on the same account/symbol set without a shared broker lock or partition.
- A flip/re-entry test proving the same warrant cannot be re-armed on a signal flip unless a bounce-confirmation condition is satisfied.
