# Signals-Core Adversarial Review — 2026-05-17

**Scope reviewed:** `portfolio/signal_engine.py`, `signal_registry.py`, `signal_weights.py`, `signal_utils.py`, `signal_db.py`, `signal_postmortem.py`, `signal_history.py`, `signal_state_since.py`, `signal_decay_alert.py`, `accuracy_stats.py`, `accuracy_degradation.py`, `outcome_tracker.py`, `ticker_accuracy.py`, `forecast_accuracy.py`, `signal_weight_optimizer.py`, `train_signal_weights.py`, `correlation_priors.py`, `perception_gate.py`.

## P1 Findings

- `portfolio/ticker_accuracy.py:130-131` SEVERITY: hidden accuracy gate at 50% — `if accuracy < 0.50: continue` silently drops every signal below 50% from the Mode B probability computation. CLAUDE.md and `.claude/rules/signals.md` mandate gating at 47% (with tiered 50% for 7K+ samples), not a blanket 50%. This means signals at 47–49.9% accuracy that *would* still vote in `_weighted_consensus` are dropped entirely from Telegram P(up). Worse, the function then renormalizes weights without these signals, biasing toward higher-accuracy survivors. Replace with the same gate constant used in signal_engine (`ACCURACY_GATE_THRESHOLD`, with tiered high-sample handling).

- `portfolio/signal_engine.py:3823, 3951` SEVERITY: per-ticker accuracy override uses wrong horizon for 3d/5d/10d consensus. `acc_horizon = horizon if horizon in ("3h", "4h", "12h") else "1d"` silently maps `3d`, `5d`, `10d` horizons to `1d` per-ticker stats. The override at 4032–4049 then writes 1d-accuracy values into `accuracy_data` used for the 3d/5d/10d weighted consensus. Per-ticker accuracy can diverge by 30pp between 1d and 5d (the 5d horizon-disable list at lines 875–889 explicitly cites ema 42.3%, heikin_ashi 44.1% at 5d) — meaning a 5d consensus path is gating signals at 1d accuracy. Use a per-horizon cache key.

- `portfolio/outcome_tracker.py:430-431` SEVERITY: `outcomes[ticker] = {h: None for h in HORIZONS}` mutates `entry["outcomes"][ticker]` to a flat `None`-valued dict when no outcome exists yet for that ticker. Subsequent code at 433–438 then iterates `outcomes[ticker].get(h_key)`, which returns None — fine — but downstream readers (`signal_accuracy`, `consensus_accuracy`) call `outcomes.get(ticker, {}).get(horizon)` and treat falsy (None) as "no outcome." A backfill that runs but fails to fetch a price will *persist* this `None` value, so subsequent backfills correctly retry. However, the same dict pattern is later JSON-serialised — see line 495 `entry["outcomes"] = outcomes` — and `outcomes[ticker][h_key]` is *never deleted* when remains None. This is OK, but combined with `signal_accuracy` line 225 (`outcome.get("change_pct", 0)`), an outcome dict that is `{"3h": None}` is filtered out (`if not outcome: continue`), but an outcome dict that is `{"3h": {"change_pct": None}}` falls through and gets treated as neutral via `_vote_correct` — counted in `total_outcomes_seen` but never in `null_change_pct_skipped` unless `change_pct` is explicitly None (the `change_pct=0` missing-key case is silently swallowed). Verify by `grep "change_pct" data/signal_log.jsonl | grep null` and confirm the diagnostic `signal_accuracy[%s]: skipped...` warning actually fires.

## P2 Findings

- `portfolio/signal_engine.py:3540-3545` SEVERITY: bare `except Exception: pass` swallows `is_promoted()` failures with no logging. If `shadow_registry.json` becomes corrupt or `portfolio.shadow_registry` import fails, every disabled signal silently stays disabled and operators have no signal that the promotion path is broken. Add `logger.debug(..., exc_info=True)` like the surrounding catches.

- `portfolio/signal_engine.py:3408-3410` SEVERITY: `raw_action = orig["action"]` will raise `KeyError` if the cached/enqueued ministral response lacks an `action` key. The outer try only catches `ImportError`. A malformed model response (or stale cache from a previous schema) crashes the ticker thread mid-signal-generation. Use `orig.get("action", "HOLD")`.

- `portfolio/forecast_accuracy.py:357` SEVERITY: `_lookup_price_at_time` re-loads `price_snapshots_hourly.jsonl` from disk for every call. Called O(entries × 2 horizons) times inside `backfill_forecast_outcomes` — at 500 entries that's 1000 full-file scans of a JSONL that grows monotonically. The whole-file scan inside the inner loop is also why `best_price` and `best_delta` overshoot on a noisy file (they update unconditionally even when the snapshot ticker dict lacks the ticker, then later check `if ticker in prices`). Load snapshots once, build a `{ts: {ticker: price}}` index, and reuse.

- `portfolio/signal_engine.py:1597-1601` SEVERITY: `_compute_dynamic_correlation_groups` mixes votes across ALL tickers into one matrix (`for _tk, tdata in entry.get("tickers", {}).items(): ... rows.append(row)`). Two signals that agree because they both follow BTC + ETH (same direction, different reasons) get a spuriously high cross-ticker agreement rate that has nothing to do with within-asset signal correlation. The output then drives correlation-group leader gating in production at line 2313–2356. Either correlate per-ticker and aggregate, or document this is intentional cross-asset structure.

- `portfolio/signal_history.py:77-98` SEVERITY: `update_history` reads + rewrites the entire history JSONL inside the lock for every per-ticker call. With 5 tickers × per-cycle calls, that's 5 full read+parse+write rounds every 60s on a 250-entry file. The lock makes this safe but the write amplification is large. Either batch writes (one write per cycle) or switch to JSON-Lines append-only with a separate trimmer.

- `portfolio/outcome_tracker.py:215-231` SEVERITY: `_fetch_historical_price` for Binance asks for kline `startTime=target_ts, interval=1h, limit=1` and reads `data[0][4]` (close). The close is the price at `target_ts + 1h`, not at `target_ts`. This adds a uniform 1h delay to every outcome regardless of horizon. For the `3h` horizon that's a 33% temporal misalignment between the prediction window and the measurement window. Use the open price (`data[0][1]`) or request `endTime=target_ts` to get the prior kline's close.

- `portfolio/accuracy_stats.py:225-228` SEVERITY: `change_pct = outcome.get("change_pct", 0)` silently coerces a missing key to 0 (treated as neutral), while only the explicit `change_pct is None` path is counted in `null_change_pct_skipped`. A data-quality regression where backfill stops writing the field entirely (vs writing `null`) would be invisible. Distinguish missing-key from None, and warn on both.

- `portfolio/accuracy_stats.py:765-777` SEVERITY: `best_worst_signals` qualifies with `v["total"] >= 5` — five samples is too few to claim "best" or "worst." Even a single mis-classified outcome flips a 5-sample signal by 20pp. Used by the dashboard/reporting which presents these as authoritative.

- `portfolio/signal_engine.py:4032-4049` SEVERITY: per-ticker accuracy override copies `t_stats["accuracy"]` straight into `accuracy_data[sig_name]` with `_PER_TICKER_MIN_SAMPLES = 30`, lower than the global `ACCURACY_GATE_MIN_SAMPLES = 30` constant but also lower than the recent-window blend's `_RECENCY_MIN_SAMPLES = 30`. After this override, the strict `ACCURACY_GATE_HIGH_SAMPLE_MIN = 7000` tier gate at line 2503 checks `samples >= 7000` against the per-ticker sample count (which will *almost never* reach 7000 per-ticker per-signal). Net effect: the tier gate that's supposed to enforce 50% on established signals is effectively disabled whenever per-ticker data exists. Verify against actual sample counts in `data/ticker_signal_accuracy_cache.json` — if no per-ticker / per-signal cell has 7000+, this gate never fires.

- `portfolio/signal_engine.py:1675-1677, 1453-1455` SEVERITY: `_get_correlation_groups` and `_get_horizon_weights` cast `_cached(...)` to dict but recent fix wraps with `if groups else {}` / `if weights else {}`. The cast still happens before the `if` — `cast(dict[str, frozenset[str]], groups) if groups else {}` evaluates `cast()` always (a no-op at runtime but signals false confidence). More importantly, when `_cached` returns None and the fallback `{}` is returned, the *next* caller of `_get_correlation_groups` at line 2313 has `_active_corr_groups = _get_correlation_groups() or _STATIC_CORRELATION_GROUPS` — so empty dict falls back to static. But the cached value `{}` is also written, meaning subsequent calls within the TTL keep returning `{}` and forcing static fallback. Verify `_cached` does NOT cache empty/None values; if it does, dynamic correlation groups silently never recover until TTL expires.

- `portfolio/signal_engine.py:3711-3725` SEVERITY: BTC→MSTR cross-ticker proxy reads `_cross_ticker_consensus["BTC-USD"]` whichever the BTC ticker thread last wrote. If MSTR is processed before BTC in the same cycle (ThreadPoolExecutor ordering is not deterministic), MSTR gets the *previous cycle's* BTC consensus — explicitly acknowledged at line 559 ("Stale reads ... acceptable — the 60s loop ensures data is at most one cycle old"). But the cache is updated only at line 4310–4313, AFTER all the gating — meaning if a future change throws or returns early before line 4310, BTC's stale value silently persists for the rest of the loop's lifetime. Add an "freshness_ts" field and gate on it.

- `portfolio/signal_engine.py:3494` SEVERITY: `fed = get_fed_calendar()` is not wrapped in `_cached`, unlike `get_dxy` and `get_treasury` on the preceding lines. If `get_fed_calendar` does I/O, it pays the cost on every ticker × horizon. Wrap consistently.

- `portfolio/accuracy_stats.py:1729` SEVERITY: `best_horizon` cache file write uses `_atomic_write_json` but the cache read at line 1726 is `load_json(BEST_HORIZON_CACHE_FILE)` with no TTL check inside; instead the TTL check happens at line 1729 inside the `if cached.get("time", 0) ...` branch. If `cached` is a non-dict (e.g. legacy list format), the inner `try` catches `KeyError`/`TypeError` and falls through to recompute. OK, but the `try` also masks unintended exceptions. Use explicit type check before .get.

- `portfolio/accuracy_stats.py:280-289` SEVERITY: `signal_accuracy_recent` always computes the cutoff from `datetime.now(UTC)`. Cached via `get_or_compute_recent_accuracy` (1h TTL). A loaded cache may have a 7-day window ending up to 1h before the call returns — the entries inside the cache are still relative to the moment the cache was populated, not the moment of the read. Documented elsewhere as acceptable, but worth checking whether the regime-degradation tracker that compares "today's 7d" vs "baseline 7d" is using consistent windowing. Currently it calls `signal_accuracy(..., entries=recent_entries)` directly, which is correct.

## P3 Findings

- `portfolio/signal_engine.py:971-977` SEVERITY: `MIN_VOTERS_METALS = 2` violates the rule documented in `.claude/rules/signals.md` ("MIN_VOTERS = 3 for all asset classes"). The in-code comment (2026-05-11) is a unilateral override of the rule. Either update the rule doc to reflect the per-asset-class divergence, or restore the original 3.

- `portfolio/signal_engine.py:1263-1287` SEVERITY: `_get_regime_gated` uses REPLACE semantics for horizon-specific overrides (3h replaces _default) while `_get_horizon_disabled_signals` (line 913) uses UNION semantics. The asymmetry is documented at length but counter-intuitive — adversarial reviews 04-24, 04-29, 05-01 all re-flagged it. Consider renaming `REGIME_GATED_SIGNALS["3h"]` to `REGIME_GATED_SIGNALS_3H_OVERRIDE` or `REGIME_GATED_SIGNALS_REPLACE_AT_3H` to make the semantics self-documenting.

- `portfolio/signal_engine.py:1742` SEVERITY: `CORRELATION_GROUPS = _STATIC_CORRELATION_GROUPS` is a public alias documented as "for backward compatibility (used by tests and reporting)" — but the *production* path uses dynamic groups via `_get_correlation_groups()`. Tests/reporting reading `CORRELATION_GROUPS` see the static fallback only and may produce misleading reports. Document explicitly that dashboard/reporting do not see dynamic groups.

- `portfolio/signal_weights.py:14-103` SEVERITY: Entire `SignalWeightManager` class is dead code per the comment at `outcome_tracker.py:497-500` ("C6: MWU weight update removed — SignalWeightManager.batch_update() wrote to data/signal_weights.json but signal_engine.py never read it. The entire MWU adaptation path was dead code..."). The class file is still present and reachable via import — remove or mark deprecated.

- `portfolio/signal_weights.py:120` SEVERITY: `_load` method ends abruptly mid-block — the comment "Honour stored eta only if caller did not override it" is followed by no actual statement. Dead/incomplete code.

- `portfolio/signal_engine.py:3287` SEVERITY: `votes["ml"] = "HOLD"` is unconditional; the entire ML signal computation is dead. The comment says "still tracked for accuracy monitoring but never votes" — but since the vote is hardcoded HOLD, no accuracy is tracked. Either compute ML's would-be vote into `_raw_votes` like shadow signals, or remove the placeholder line.

- `portfolio/signal_engine.py:1764-1774` SEVERITY: `_META_CLUSTER_GROUPS` is hardcoded with only one entry (`trend_mega`). The `_STATIC_CORRELATION_GROUPS` was split 2026-04-30 specifically to enable independent leader selection. If a future split changes sub-cluster names without updating `_META_CLUSTER_GROUPS`, the meta-cluster penalty silently stops firing. Add an assert that all entries in `_META_CLUSTER_GROUPS` values exist as keys in `_STATIC_CORRELATION_GROUPS`.

- `portfolio/signal_engine.py:4108` SEVERITY: `accuracy_gate = sig_cfg.get("accuracy_gate_threshold", ACCURACY_GATE_THRESHOLD)` allows config to lower the gate below the documented 47% floor. If `config.json` ever ships with `accuracy_gate_threshold: 0.30`, the entire accuracy-gate protection is silently disabled. Add a `max(config_value, 0.45)` floor.

- `portfolio/signal_engine.py:3160-3192` SEVERITY: `get_sustained_fear_days` reads streak BEFORE updating, then `update_fear_streak(fg["value"])` updates inside `if ticker in ("BTC-USD", None)`. The `None` check is for `if ticker is None` — but the iteration order means non-BTC tickers in the same cycle see the *pre-update* streak. If BTC happens to be processed last (ThreadPoolExecutor ordering), no ticker in the cycle uses the updated streak. Acceptable since streak only changes day-by-day, but worth documenting that the streak update is order-sensitive.

- `portfolio/accuracy_stats.py:1248` SEVERITY: `_count_entries_with_outcomes` exits the inner loop on the first ticker that has the horizon — counting entries, not (entry × ticker) pairs. The variable name and use in `print_accuracy_report` line 1257 implies entries-with-outcomes, so OK, but the leading `break` inside the for-loop is easy to misread as "iterate all and break on first match."

- `portfolio/ticker_accuracy.py:227` SEVERITY: `forecast_p_up = 0.5 + min(max(pct_move * 0.1, -0.3), 0.3)` — Chronos `pct_move` is in percent already (per the signal docs); multiplying by 0.1 then clamping to [-0.3, +0.3] means a Chronos prediction of +3% maps to forecast_p_up = 0.5 + 0.3 = 0.8 (clamped at +30pp from neutral). A prediction of +30% (clearly an extreme) hits the same 0.8. The clamp is correct but the *scaling factor 0.1 is arbitrary*. Document the rationale or expose as a tunable constant.

- `portfolio/forecast_accuracy.py:152-156` SEVERITY: `sub_horizon = sub_name.split("_", 1)[1]` assumes every sub-signal name has the form `{model}_{horizon}` (e.g. `chronos_24h`). A sub-signal named without an underscore (e.g. `kronos`) would be silently skipped. Either assert the format or document the contract.

- `portfolio/signal_engine.py:4300` SEVERITY: `conf = min(conf, 0.80)` is a hardcoded global cap that overrides the entire calibration cascade. The CLAUDE.md rule says "never recommend trades below 60% calibrated confidence." If this cap silently shrinks every "high conviction" signal to 80%, the entire confidence distribution collapses into a narrow band — making downstream Kelly sizing useless. Worth verifying empirically that signal confidence distribution post-cap has any usable spread.

- `portfolio/signal_decay_alert.py:73` SEVERITY: `drop_pp = (alltime_acc - recent_acc) * 100` — drop is positive when accuracy fell. But it doesn't apply the SE-significance gate that `accuracy_degradation._maybe_alert` uses. The two systems can disagree: `signal_decay_alert` will fire 1-in-33 false positives at N=100 that `accuracy_degradation` would correctly suppress. Either unify with `_binomial_diff_se_pp` or document that decay-alert is the "noisy" early-warning system.

- `portfolio/signal_engine.py:4136-4146` SEVERITY: `soft_confidences=extra_info` — passes the *entire* `extra_info` dict (which contains hundreds of keys including dataclass-like values) for the soft-conf lookup. Inside `_weighted_consensus`, only `f"_soft_conf_{signal_name}"` keys are read. Passing extra_info wholesale couples consensus to the entire extra_info schema and means any innocent rename of an extra_info key cascades into consensus behaviour. Build a small `{name: soft_conf}` dict at the call site.

## Notes / Cross-cutting concerns

- The accuracy pipeline has *three different sample-count floors* in active use: `ACCURACY_GATE_MIN_SAMPLES = 30`, `_RECENCY_MIN_SAMPLES = 30`, `_PER_TICKER_MIN_SAMPLES = 30`, `_DIRECTIONAL_GATE_MIN_SAMPLES = 30`, `_DIRECTIONAL_RESCUE_MIN_SAMPLES = 30`, `_DIR_WEIGHT_MIN_SAMPLES = 20`, `_BIAS_MIN_ACTIVE = 30`, `_ACCURACY_GATE_HIGH_SAMPLE_MIN = 7000`, `_DYNAMIC_HORIZON_MIN_SAMPLES = 50`, `_IC_MIN_SAMPLES = 100`. The 30 values are likely meant to be the same constant — a single `MIN_SAMPLES_DEFAULT = 30` referenced everywhere would prevent silent drift.

- `signal_engine.py` is 4315 lines with one function (`generate_signal`) of ~1280 lines. The phase-log instrumentation is extensive precisely because the function is too large to reason about. Consider splitting into `compute_core_votes()`, `dispatch_enhanced_signals()`, `apply_gates()`, `compute_consensus()`, `apply_penalties()`.

- `_STATIC_CORRELATION_GROUPS` has comments documenting that several historical clusters were dissolved when their members were disabled (`pattern_based`, single-member groups). The dynamic correlation engine at `_compute_dynamic_correlation_groups` will *recreate* such clusters if signals start agreeing again — but the dynamic engine names them `dynamic_0, dynamic_1, ...` so the rich semantic naming is lost in the live system. Operators reading penalty logs see opaque names. Either propagate `_STATIC_CORRELATION_GROUPS` member-name → static-group-name when a dynamic cluster overlaps significantly with a known static one, or label dynamic groups by their highest-accuracy member.

- The dual-write pattern in `outcome_tracker.log_signal_snapshot` (JSONL + SQLite) means JSONL and SQLite can diverge if either path fails — JSONL is preserved but SQLite write is silently swallowed at line 165. Backfill reads SQLite-preferred (`accuracy_stats.load_entries`) so any divergence means accuracy is computed off the SQL view. Verify divergence with `python -c "from portfolio.signal_db import SignalDB; print(SignalDB().snapshot_count())"` vs `wc -l data/signal_log.jsonl`.

- `correlation_priors.py` defines only 2 correlation pairs (BTC/ETH, XAG/XAU). MSTR has documented 0.58 correlation with BTC but is absent. If a future code path consumes `get_correlated_pairs()` it won't see MSTR↔BTC, possibly producing incorrect position-risk sizing for the MSTR/BTC pair.

- The `_KNOWN_SHADOW_LLMS` frozenset at `signal_engine.py:658-664` is a static fallback used when the dynamic `shadow_registry` check fails. The comment says "Keep in sync with the 'shadow'-status entries in data/shadow_registry.json" — a manual sync is fragile. Add a startup assertion that verifies these names exist in the registry (or auto-sync via the registry on first call).
