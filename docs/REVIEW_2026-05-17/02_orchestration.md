# Orchestration Subsystem Audit — 2026-05-17

Files reviewed: portfolio/main.py (1532 lines), portfolio/agent_invocation.py (1634), portfolio/trigger.py (645), portfolio/trigger_buffer.py (202), portfolio/autonomous.py (846), portfolio/multi_agent_layer2.py (246), portfolio/claude_gate.py (813), portfolio/gpu_gate.py (266), portfolio/market_timing.py (342), portfolio/session_calendar.py (211), portfolio/escalation_gate.py (229), portfolio/escalation_router.py (269), portfolio/reporting.py (1319).

## Critical Findings

portfolio/trigger.py:172-184, 218-221, 506: 🔴 critical: empty-signals cycle wipes the entire `triggered_consensus` baseline, causing a Layer 2 invocation storm on the next successful cycle. `check_triggers` unconditionally sets `state["_current_tickers"] = set(signals.keys())` at L221. When the upstream ticker pool times out (BUG-178 at main.py:630-661 — the `_TICKER_POOL_TIMEOUT = 360` path) or every ticker thread raises, `signals` is `{}` and `_current_tickers` is an empty set. `_save_state` at L177-182 then prunes `triggered_consensus` against that empty set — every previously-tracked consensus is removed. On the next cycle when signals are populated again, every ticker with `action in (BUY, SELL)` matches `last_tc == "HOLD"` at L288 and re-fires a `{ticker} consensus {action} ({pct}%)` reason. With 5 Tier-1 tickers this can produce 5 simultaneous consensus crossings → tier=3 first-of-day OR tier=2 invocation → wastes ~600s of T2 budget and consumes Claude quota for HOLD decisions that aren't actually new. Fix: skip the prune branch when `len(_current_tickers) == 0` (treat empty as "unknown, don't mutate") OR early-return `(False, [])` at L221 when `not signals`.

portfolio/main.py:244, 247-260; portfolio/trigger.py:452-454: 🔴 critical: sentiment-triggered Layer 2 invocations get NO ticker context. `_TICKER_PAT = re.compile(r'^([A-Z][A-Z0-9]*(?:-[A-Z]+)?)\s+(?:consensus|moved|flipped)')` only matches `consensus|moved|flipped` keywords. `check_triggers` produces sentiment reason strings shaped `f"{ticker} sentiment {last_stable}->{sent} (sustained)"` at trigger.py:452, which DO NOT match. Consequence chain: main.py:850 `triggered_tickers = _extract_triggered_tickers(reasons_list)` returns empty set → reporting.py:1212 `triggered_tickers = triggered_tickers or set()` → reporting.py:1217 `full_detail_tickers = held_tickers | triggered_tickers` falls back to held-only → the sentiment-triggered ticker (if not held) gets only the medium-detail or price-only entry in agent_context_t2.json. Layer 2's prompt explicitly says "Analyze triggered tickers and held positions" but the triggered ticker's signal voting detail is missing. Fix: extend the regex to include `sentiment` (and arguably also accept the bare `\bTICKER\b` form scanned across the full string, matching escalation_router.py:149-178's approach).

portfolio/agent_invocation.py:580-670, 623-626: 🔴 critical: `taskkill` has no timeout while holding `_completion_lock`. `_kill_overrun_agent` calls `subprocess.run(["taskkill", "/F", "/T", "/PID", str(pid)], capture_output=True)` at L623-626 — NO `timeout=` parameter. Both call sites that reach this helper (`invoke_agent` at L745-769 and `_check_agent_completion_locked` at L1357 / watchdog at L98-108) hold `_completion_lock`. If taskkill itself hangs (rare under disk/memory pressure or stuck driver), the entire completion path deadlocks: the watchdog stops, the next `invoke_agent` blocks on lock acquisition, and the per-tier wall-clock timeout becomes unenforceable. Inconsistent with claude_gate.py:386-388 which uses `timeout=5` on its taskkill. Fix: add `timeout=5` parameter and handle `subprocess.TimeoutExpired` by logging an error and proceeding with `kill_ok=False` (caller already refuses to spawn a replacement when kill_ok is False).

portfolio/multi_agent_layer2.py:127-186: 🔴 critical: specialist subprocesses bypass the centralized claude_gate kill switch, daily rate limiter, in-process lock, and auth-failure pre-detection. `launch_specialists` directly invokes `subprocess.Popen([claude_cmd, "-p", ...], ...)` at L172, in explicit violation of claude_gate.py:8-10: "Direct subprocess.Popen([claude_cmd, "-p", ...]) calls are FORBIDDEN. Doing so bypasses the kill switch, rate limiter, and invocation tracking." Three parallel specialists per multi-agent T2/T3 invocation — `CLAUDE_ENABLED = False` cannot block them, `_DAILY_WARN_THRESHOLD` does not see them, `_invoke_lock` does not serialize them, and `claude_invocations.jsonl` does not log them. Auth-failure detection only runs post-hoc on the specialist log file (multi_agent_layer2.py:223-231) — if auth is broken, three specialists will burn time printing "Not logged in" before the post-hoc scan catches it. Same caveat applies to agent_invocation.py:1074 (which has its own watchdog/timeout machinery to compensate), but specialists have only the wait_for_specialists deadline, no kill switch, no rate limit. Fix: route specialist spawns through invoke_claude (or a new `invoke_claude_async` primitive) so all gates apply.

## Risk Findings

portfolio/main.py:807-837, 484-497 of trigger.py: 🟡 risk: trigger_buffer defers a trigger AFTER check_triggers has already advanced the on-disk baselines, producing orphaned reasons. When `_batch_window > 0` and `triggered=True`, `check_triggers` has already written `state["last"] = {prices, signals, sentiments, ...}` (trigger.py:484-497), `state["last_trigger_time"]`, and `state["last_trigger_date"]` — flipping every per-ticker tracking baseline to "current state". main.py:822-830 then buffers the reasons and sets `triggered=False`, so Layer 2 never sees them. Next cycle, `prev["signals"]` reflects the now-advanced state, so the same flip cannot re-fire. When the buffer flushes 5 minutes later, it emits reasons referencing state that may have already reverted (e.g. a flip back-flipped during the buffer window). Fix: have trigger_buffer.add return the buffered set BEFORE check_triggers commits the new baseline, OR rollback `state["last"]` when buffering defers, OR move the buffer ahead of state mutation in check_triggers.

portfolio/agent_invocation.py:644-670: 🟡 risk: `_kill_overrun_agent` does not clear `_agent_start`, `_agent_start_wall`, `_agent_tier`, `_agent_reasons`, `_journal_count_before`, `_telegram_count_before` — only `_agent_proc` and `_agent_log`. Compare `_check_agent_completion_locked` at L1548-1557 which clears ALL of them. Not currently exploitable (because `_agent_proc is None` short-circuits `_check_agent_completion_locked` at L1342), but a future change that reads `_agent_tier` without checking `_agent_proc` first would see stale data from the previous invocation. Defensive symmetry: clear all module-level state at the end of `_kill_overrun_agent`.

portfolio/escalation_gate.py:160: 🟡 risk: `_dt.datetime.utcnow().isoformat() + "Z"` uses deprecated naive UTC. Deprecated in Python 3.12+, will be removed in a future version. Every other orchestration file (main.py:1331, agent_invocation.py:11/172/1376, trigger.py:165) uses `datetime.now(UTC).isoformat()`. Cosmetic but inconsistent; fix: `_dt.datetime.now(_dt.timezone.utc).isoformat()` or import `from datetime import UTC` and use `datetime.now(UTC)`.

portfolio/escalation_gate.py:202-215: 🟡 risk: `cancel_futures=True` on a ThreadPoolExecutor cannot cancel a running Python thread. When Ministral runner times out (L207), the orphan thread keeps blocking inside `query_llama_server`. On a llama_server hang, every triggered cycle adds another zombie thread to that singleton-use executor — but since `_ex = ThreadPoolExecutor(max_workers=1)` is created locally per call (L203), each invocation gets a fresh executor and the previous zombie thread holds its own resources until llama_server times out or recovers. Acknowledged by the comment ("Don't wait for the hung thread on shutdown") but still leaks one daemon thread per timeout. Fix: front the call with a wall-clock timeout inside the runner itself (e.g. pass `request_timeout=10` to `query_llama_server`) instead of relying on cancellation.

portfolio/agent_invocation.py:1014-1019: 🟡 risk: pf-agent.bat fallback runs at Tier 3 regardless of the requested tier, but the timeout passed to it is still `_agent_timeout = tier_cfg["timeout"]` for the requested tier (L1071). A request for T1 with `claude_cmd=None` falls back to pf-agent.bat (which doesn't accept `--max-turns` or model args at all) but runs under the 150s T1 timeout — likely insufficient for a full T3 review. The `effective_tier` correction at L1114 is only applied to `health_state.last_invocation_tier` (for loop_contract's grace window), NOT to `_agent_timeout`. Fix: when `not claude_cmd`, override `_agent_timeout = TIER_CONFIG[3]["timeout"]`.

portfolio/agent_invocation.py:1078-1081: 🟡 risk: `stderr=subprocess.STDOUT` merges stderr into stdout but `claude_gate.detect_auth_failure` (claude_gate.py:298-326) was designed to scan stdout and stderr SEPARATELY (BUG-ECHO 2026-04-16 fix at claude_gate.py:594-604) to avoid concat-without-newline merging the "Not logged in" marker into the last stdout line. In `invoke_agent`, both streams are funneled into `agent.log`; the post-hoc scan at L1409 reads the merged file. If the agent emits stdout text that ends without a newline immediately before the CLI prints "Not logged in" to stderr, the marker collapses into the prior line and `_is_real_auth_marker_line` rejects it. This is the same class of bug claude_gate.py explicitly fixed. Fix: capture stderr to a separate temp file or pipe and scan it independently.

portfolio/main.py:1357-1377: 🟡 risk: on a crashed cycle inside `run()`, `_safe_crash_recovery(tb_text)` is called but `last_cycle_started` is updated to `cycle_started` at L1385 only AFTER the crash recovery completes. Combined with `_sleep_for_next_cycle` (L1259-1265) which anchors cadence to `last_cycle_started`, a crash recovery that sleeps ≥ `sleep_interval` causes the next `_sleep_for_next_cycle` to log "Loop overran target cadence by X seconds" and immediately re-run (remaining<=0 → no sleep). This is by design (don't lose cycles to crash backoff), but means a persistent crash loop can hit "no sleep between attempts" if the floor sleep already consumed the cycle interval. The `_safe_crash_recovery` floor sleep IS supposed to cover this but its cap is 30s (`_CRASH_FLOOR_SLEEP_CAP`) which is less than the 600s loop interval — so the loop will tight-poll between successive crashes after backoff is exhausted. Not catastrophic (backoff scales up to 300s normally), but the "guaranteed minimum" sleep is misleadingly small.

portfolio/agent_invocation.py:1409-1412, 661: 🟡 risk: auth-error log scan uses `_agent_log_start_offset` which was captured at `invoke_agent` entry (L1028), but on the timeout-kill path the offset reflects the START of THIS invocation's output — good. However if a previous invocation's `_agent_log` was opened but never had its offset reset (e.g., spawn failed at Popen, L1144-1148 closes log_fh but doesn't update offset), the NEXT invocation will scan from a stale offset. Currently mitigated because the offset is re-captured at L1028 of EVERY successful spawn before Popen runs, but the order is fragile: any future edit that moves the offset capture out of the success path could re-introduce a multi-invocation scan that triggers a false "Not logged in" on echoed text from a prior session.

portfolio/escalation_router.py:140-145, 229-249: 🟡 risk: `_ticker_held` does 2 disk reads (portfolio_state.json + portfolio_state_bold.json) per invocation, called inside a per-reason loop. With 5+ reasons in a flushed buffer flush and 2 strategies, that's 10+ disk reads per `should_escalate_to_claude` call. Cheap individually but cumulative under load. Compare `reporting._get_held_tickers` (reporting.py:850-870) which caches per `_run_cycle_id`. Fix: hoist the load_json calls outside the for loop, or share `_get_held_tickers` with reporting.

portfolio/trigger.py:226-228: 🟡 risk: startup grace skip uses PID equality to detect "same process". On Linux, PID reuse after fork or under aggressive process recycling could cause a fresh process to share a PID with the previously-killed loop — `_startup_grace_active and saved_pid != current_pid` becomes False, grace is skipped, every signal re-triggers as if no baseline existed. Rare but real (kernel cycles 4M PIDs by default on Linux, ~32K on Windows). On Windows, PID reuse after explorer.exe restart or schtasks recycle is plausible. Better: combine PID with start_ts or use a fresh per-process UUID.

portfolio/main.py:404, 416 / shared_state usage: 🟡 risk: `_ss._last_log_rotation_ts` is assigned via `getattr` then mutated directly without any lock, and `shared_state` is imported as `_ss`. Currently called from the main thread only (inside `_run_post_cycle`), so single-threaded. But the comment at L401-402 says "so it survives cross-module access within one process" — implying other modules MIGHT eventually read it. If a worker thread ever reads `_ss._last_log_rotation_ts`, the unsynchronized 64-bit float write is technically not atomic on 32-bit Python (irrelevant on this 64-bit Windows install but a latent issue). Note: the GIL protects against torn reads in CPython, but free-threaded Python 3.13+ will not. Cosmetic for now.

portfolio/multi_agent_layer2.py:189-233: 🟡 risk: `wait_for_specialists` calls `proc.wait(timeout=remaining)` (L204) but `proc.kill(); proc.wait(timeout=5)` on TimeoutExpired (L211-212) uses Popen.kill() which on Windows only kills the direct child, NOT the Node.js descendants. Compare claude_gate.py:373-417 `_kill_process_tree` which uses `taskkill /T /F` for full tree teardown. Each timed-out specialist potentially leaves orphaned MCP server / claude-helper Node.js processes. Same class of issue claude_gate fixed in A-IN-2. Fix: import and call `_kill_process_tree(proc, label=f"specialist_{name}")` from claude_gate.

## Nit Findings

portfolio/agent_invocation.py:1074: 🔵 nit: comment at L1064-1068 explains the ordering of `_agent_start` assignments relative to `Popen` (avoid watchdog race), but `_agent_log = log_fh` is assigned AFTER Popen (L1082). If Popen raises, the except handler (L1144-1148) closes `log_fh` (the local), but `_agent_log` is still `None` from a prior cleanup — OK. The exception ordering is consistent but the comment doesn't mention this branch. Minor.

portfolio/main.py:247, agent_invocation.py:285: 🔵 nit: duplicate `_extract_triggered_tickers` functions with different semantics. main.py's regex requires the ticker to be at the START of the string followed by `consensus|moved|flipped`. agent_invocation.py's scans the full string for `[A-Z]{2,5}-USD` and stock-ticker-like patterns. They're called from different sites (main.py for tier filtering, agent_invocation for drawdown context), but the semantic split is implicit and the names collide. Either consolidate or rename one to disambiguate.

portfolio/main.py:1267: 🔵 nit: `_sleep_for_next_cycle` logs "Loop overran target cadence by X seconds" with `abs(remaining)` but only when `remaining <= 0`. The phrasing is correct but the conditional branch already implies overrun — the abs() is decorative. Minor.

portfolio/claude_gate.py:344-351: 🔵 nit: `_count_today_invocations` scans the entire `claude_invocations.jsonl` linearly per `invoke_claude` call. Bounded by the 5000-line prune cap, but on a busy day this is ~5000 JSON.loads per Claude invocation. Cache the count per minute or precompute it once per process boot.

portfolio/agent_invocation.py:178-191: 🔵 nit: comment at L179 says "T1 timeout 120 → 150s after 3d audit" then L184 says "150 → 180s. Even after collapsing 3 Reads into 1 Bash cat...". The TIER_CONFIG dict at L187-191 shows 180 for T1, consistent with the second comment, but the first comment claims 150. The first comment is stale.

portfolio/market_timing.py:13-14: 🔵 nit: `MARKET_OPEN_HOUR = 7` is "kept at 7 (summer value)" for backward compat but the comment says "Callers that need DST-aware EU open should use _eu_market_open_hour_utc()". Grep shows no current callers of bare `MARKET_OPEN_HOUR` outside the re-export at main.py:166 — safe to remove the constant entirely and remove the backward-compat comment.

portfolio/agent_invocation.py:65-66, _consecutive_stack_overflows handling: 🔵 nit: `_consecutive_stack_overflows` is loaded at module import time (L176) but mutated globally without a lock. Only mutated from `_check_agent_completion_locked` (which holds `_completion_lock`) and read from `invoke_agent` (which also enters the lock at L745). portfolio/escalation_gate.py:117: 🔵 nit: `_JSON_RE = re.compile(r"\{[^{}]*\}", re.DOTALL)` does not handle nested JSON. If Ministral ever emits `{"escalate": true, "details": {"why": "X"}}`, the regex captures only `{"why": "X"}`. Currently the prompt asks for a flat object so the failure mode is hypothetical, but the parser at L120-144 only checks `"escalate" not in obj` and would raise on the inner object missing the key. Fix: use the brace-counting walker pattern from claude_gate.py:99-127.

portfolio/trigger.py:127, 228, 255, 259: 🔵 nit: module-level `_startup_grace_active = True` and the reset to `False` at L255 + L259 happen in BOTH branches of the if/else at L228. The unconditional `_startup_grace_active = False` at L259 means the L255 set inside the grace branch is redundant. Cosmetic.

## Summary

- 4 critical: empty-signals wipes trigger baseline (storm risk); sentiment triggers lose Layer 2 context; taskkill has no timeout while holding completion lock; multi-agent specialists bypass claude_gate
- 12 risks: trigger_buffer/state race; incomplete kill cleanup; deprecated datetime.utcnow; futures.TimeoutError leaks daemon threads; pf-agent.bat fallback uses wrong timeout; stderr/stdout merge defeats auth-failure split-scan defense; crash floor cap < cycle interval; auth-scan offset ordering fragility; per-reason disk reads in escalation_router; PID-based startup grace; unprotected shared_state field; specialist proc.kill leaves Node descendants
- 8 nits: log_fh assignment ordering comment; duplicate _extract_triggered_tickers; overran-cadence log; linear count_today_invocations scan; stale T1 timeout comment; dead MARKET_OPEN_HOUR; undocumented serialization of stack-overflow counter; brace-counting regex; redundant grace flag reset

The completion-watchdog architecture (agent_invocation.py:87-155) is sound, as is the lazy file-lock sweeper in gpu_gate.py. The auth-failure detection at claude_gate.py:274-327 correctly handles the BUG-ECHO feedback loop. Crash backoff with jitter (main.py:1177-1186) and singleton lock (main.py:57-92) protect the loop from runaway storms within a single tier of failure modes — but the trigger-baseline-wipe bug is exactly the kind of cross-tier failure that bypasses those protections (the loop runs fine, but every cycle generates spurious Layer 2 work).
