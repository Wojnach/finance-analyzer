# Portfolio-Risk Adversarial Review — 2026-05-17

**Scope reviewed:** `portfolio/portfolio_mgr.py`, `portfolio_validator.py`, `trade_guards.py`, `trade_risk_classifier.py`, `trade_validation.py`, `risk_management.py`, `circuit_breaker.py`, `equity_curve.py`, `monte_carlo.py`, `monte_carlo_risk.py`, `cost_model.py`, `alert_budget.py`, `exposure_coach.py`, `warrant_portfolio.py`, `cumulative_tracker.py`.

## P1 Findings

- `portfolio/monte_carlo.py:47-65` SEVERITY: `volatility_from_atr` annualizes with `sqrt(252/14)` (factor ~4.24), but the `atr_pct` it consumes from `agent_summary["signals"][T]["atr_pct"]` is computed from the "Now" 15-minute kline timeframe (`indicators.py:148`, `reporting.py:185`, `main.py:489`). Correct annualization for 15-min candles on a 24/7 series is `sqrt(525600/15/14) ≈ sqrt(2500) = 50x`. Result: real ATR values pulled live (BTC 0.17, ETH 0.22, XAG 0.08, XAU 0.04) produce annualized vols of 0.7%–0.9%, which then get floored to `MIN_VOLATILITY=0.05` for every ticker. Verified by `python -c "from portfolio.monte_carlo import volatility_from_atr; print(volatility_from_atr(0.17))"` → `0.05`. Consequence: every `simulate_ticker` / `compute_portfolio_var` call uses the same 5% vol floor regardless of which asset class is being analyzed, so all MC-derived numbers (price bands, stop-hit probabilities, VaR, CVaR) are flat noise rather than tracking actual market conditions. Either annualize by the candle timeframe the ATR was derived from, or pull a longer-timeframe ATR (e.g., 12h or 1d) for risk modelling.

- `portfolio/monte_carlo_risk.py:407` SEVERITY: `fx_rate = agent_summary.get("fx_rate", FX_RATE_FALLBACK)` re-introduces the exact bug that `risk_management._resolve_fx_rate` was added (05-02, P1-15) to fix. When `agent_summary` contains the stale `fx_rate=1.0` value or a parse failure leaves an out-of-band number, `compute_portfolio_var` accepts it without sanity-banding. Verified by running `compute_portfolio_var` against an `{"fx_rate":1.0, ...}` summary: SEK exposure reads 50,000 instead of 525,000; `var_95_sek` reads −268 SEK instead of −2,820 SEK. The 10x understatement of VaR/CVaR in SEK silently bypasses any Layer 2 sizing that consumes the SEK-denominated risk fields. Fix: replace the raw `.get` with the validated chain in `risk_management._resolve_fx_rate` (which is even already in the same `data/` namespace).

- `portfolio/portfolio_mgr.py:28-41,108-114,136-159` SEVERITY: `_state_locks` and `update_state` use `threading.Lock`, which only protects against concurrent threads inside the *same* process. The Layer 2 Claude subprocess (`agent_invocation.py:232-263` instructs the agent to `cat data/portfolio_state.json … portfolio_state_bold.json` and `docs/TRADING_PLAYBOOK.md:127` explicitly tells it to "Edit `data/portfolio_state.json`") executes outside the main loop's interpreter and bypasses the lock entirely. If Layer 2 is mid-Edit when Layer 1 calls `save_state` (or vice-versa), one of the two writes silently disappears via `os.replace` clobber. Required fix: use the project's existing cross-process file lock (`portfolio/file_utils.py:203` `jsonl_sidecar_lock` or `avanza_order_lock.py`) around the read-modify-write window, not `threading.Lock`. Same class of bug as the warrant-portfolio finding below; both have been flagged in multiple prior reviews and remain unfixed.

- `portfolio/warrant_portfolio.py:25-48,182-266` SEVERITY: `load_warrant_state` / `save_warrant_state` / `record_warrant_transaction` perform load-mutate-save with no lock — not even the process-local `threading.Lock` used in `portfolio_mgr`. Grid fisher (`portfolio/grid_fisher.py`) and the metals loop can both fire concurrent BUY/SELL recordings within the same process, and `atomic_write_json` only guarantees the *final write* is atomic, not that the read-modify-write cycle is. Two concurrent BUYs of the same `config_key` will both read `old_units=N`, compute `new_units = N + bought_a` vs `new_units = N + bought_b`, and the second write wins — losing one entire BUY's worth of units while still appending the transaction record. The portfolio_validator's holdings-shares check will then start screaming "Holdings mismatch". Fix: introduce `update_warrant_state(mutate_fn)` equivalent to `portfolio_mgr.update_state`, ideally backed by a cross-process file lock.

- `portfolio/equity_curve.py:494-528` SEVERITY: Win/loss bucketing uses gross `pnl_pct > 0` (line 494) while `profit_factor`, `total_pnl_sek`, and the Calmar-feeding equity curve use net `pnl_sek` (lines 468-469, 532, 538). A trip that nets to −10 SEK after fees but has a positive 0.1% pre-fee `pnl_pct` is counted as a *win* by `win_rate`, `max_consecutive_wins`, `avg_win_pct`, and `expectancy_pct`, while contributing to `gross_loss` in `profit_factor`. With WARRANT/CRYPTO round-trip costs of ~50-100 bps, the price-vs-cost-basis disagreement is common. Either compute `pnl_pct_net = pnl_sek / buy_notional` and bucket on that, or convert all downstream metrics to consume `pnl_sek`. Comment at lines 397-402 acknowledges the gross-vs-net split but doesn't propagate the choice consistently.

## P2 Findings

- `portfolio/exposure_coach.py:30,44,89,92,102` SEVERITY: `_REGIME_MULTIPLIERS` and the new-entries / bias / rationale checks all key on `"range-bound"`, but `indicators.detect_regime` (`indicators.py:187,194,196,203`) returns `"ranging"` for the same regime. `reporting.py:294` feeds `dominant_regime` from `extra._regime` (set in `signal_engine.py:4179` to `regime` which comes from `detect_regime`) directly into `compute_exposure_recommendation`. Verified by `python -c "from portfolio.exposure_coach import compute_exposure_recommendation; print(compute_exposure_recommendation({'zone':'healthy','score':70}, regime='ranging')['exposure_ceiling'])"` → `1.0` (no penalty), vs `regime='range-bound'` → `0.9`. The 0.9× ranging penalty is silently disabled in production, and the rationale string falsely advertises "ranging regime" as a factor even though no multiplier was applied (`if regime != "range-bound"` is True for "ranging"). Either standardize on one regime vocabulary (the project's `regime_alerts.py:21-24,61-64` documents the alias) or normalize at the boundary.

- `portfolio/risk_management.py:43-110` SEVERITY: `_streaming_max` caches the peak per `(path, value_key)` keyed by file size — but the cache is *process-wide* and the file is read by both the main loop and any dashboard/CLI worker importing the same module. On `--report` or `--accuracy` CLI invocations the cache starts empty and the full file is rescanned every time, which is fine. The lock-protected read-then-write at lines 70-109 has the right scope. However the OSError fallback at line 102-106 returns `float(cached_after["peak"])` *without* updating the offset, so the next call retries from the same broken offset — repeated transient OS errors lead to repeated full file scans (`offset` never advances). Bound recovery by clearing the cache entry on OSError so the next call starts fresh from 0.

- `portfolio/cost_model.py:72-79` SEVERITY: `STOCK_COSTS = CostModel(courtage_bps=6.9, min_fee_sek=1.0, label="avanza_stock")`. The 0.069% rate corresponds to Avanza's "Medium" courtage class (minimum 69 SEK), and the 1 SEK floor corresponds to Avanza's "Mini" class (rate 0.25% = 25 bps). These are mixed from different tariff tiers — `courtage_bps=6.9` with `min_fee_sek=1.0` matches no Avanza class. Cost estimates for any trade above ~145 SEK will use the wrong courtage rate. The user's actual stock-side tier is documented as "Mini" elsewhere (per `avanza_session.py:764` comments referencing the 1000 SEK threshold). Set `courtage_bps=25.0` for "Mini" or `min_fee_sek=69.0` for "Medium" — but not the current mix.

- `portfolio/cost_model.py:64-70` SEVERITY: `WARRANT_COSTS.min_fee_sek=0.0`. Avanza warrants in Mini courtage have a 1 SEK floor per leg (per `avanza/trading.py:71,245-251` which references the 1000 SEK leg threshold that *only* matters because the percentage courtage kicks in below it). For micro-trades the exit optimizer's cost-vs-edge math will allow trades that the broker will actually charge a minimum fee on, biasing toward smaller (sub-1000 SEK) trades that are uneconomic.

- `portfolio/monte_carlo_risk.py:356-376` SEVERITY: `drawdown_probability` measures portfolio loss as a percentage of *exposure* (sum of `shares × price`), not of total portfolio value (which would include cash). `compute_portfolio_var` then surfaces `drawdown_1pct_prob` and `drawdown_5pct_prob` in the result dict alongside SEK VaR. A 5% drawdown of a 10% exposure is only a 0.5% portfolio-level drawdown, but Layer 2 has no way to know which denominator was used. Either rename the keys to `exposure_loss_1pct_prob` or denominate by `cash + total_exposure_sek`.

- `portfolio/portfolio_validator.py:65-67` SEVERITY: `avg_cost_usd is not None and avg_cost < 0` — but `avg_cost = pos.get("avg_cost_usd")` (line 65) overwrites the loop's outer `avg_cost = pos.get("avg_cost_usd")`. If `avg_cost_usd` is 0 (legitimate for a fresh entry that hasn't been priced), the check `< 0` passes silently. Acceptable defensive behavior but `avg_cost_usd == 0` for `shares > 0` is itself a bug condition that goes unflagged. The Check 8 path (lines 215-243) catches missing `avg_cost_usd` (None) but not zero.

- `portfolio/correlation_priors.py:8-11` SEVERITY: `CORRELATION_PRIORS` ships only two pairs: `(BTC, ETH)` and `(XAG, XAU)`. MSTR–BTC (documented as a "leveraged BTC proxy" in CLAUDE.md), XAG–copper (Metals Cross-Asset signal), and XAU–real-yields are all known correlations that are missing. `monte_carlo_risk.build_correlation_matrix` falls back to `_get_prior_correlation` = 0.0 for all missing pairs when historical data is unavailable, so VaR significantly understates joint tail moves whenever MSTR + BTC or XAG + XAU + MSTR coexist. Add the known priors, or document the deliberate conservatism.

- `portfolio/equity_curve.py:495` SEVERITY: `losses = [t for t in trips if t["pnl_pct"] <= 0]` — zero-P&L round-trips are bucketed as losses. They then contribute to `loss_count`, `loss_rate`, and `max_consecutive_losses`, but with `pnl_pct=0` they don't affect `avg_loss_pct`. Cosmetic skew on idle / flat tickers.

- `portfolio/alert_budget.py:41-43` SEVERITY: Emergency-priority alerts bypass the budget check but still `_sent_timestamps.append(time.time())` — so emergency alerts consume budget tokens against future *normal* sends. A burst of 5 emergency stop-loss alerts (a real scenario when the metals book turns against a leveraged position) consumes the entire hourly budget, then suppresses every subsequent normal alert for an hour. Use a separate counter for emergency alerts, or simply don't increment the budget on emergency sends.

- `portfolio/trade_validation.py:75-81` SEVERITY: `max_cash_pct` enforces "fraction of *cash*", not "fraction of total portfolio value". A 50% cash limit after a portfolio is heavily invested can still place a 50% cash trade that's only 10% of total portfolio — fine. But a fully-cash portfolio (e.g., right after a big SELL) allows a 50%-of-cash position that's also 50% of total portfolio with no awareness that this concentration is high. The portfolio-proportional check lives in `risk_management.check_concentration_risk` (40% threshold) but is decoupled — `validate_trade` doesn't know about the broader concentration limit. Add a `max_portfolio_pct` parameter and call from upstream with the live total-value.

- `portfolio/monte_carlo.py:266-339` SEVERITY: `simulate_ticker` accepts `seed: int | None = None`. When unset (the production path in `simulate_all` from `reporting.py`), `MonteCarloEngine` constructs `np.random.default_rng(None)`, which seeds from OS entropy — non-reproducible. The 04-22 follow-up at line 396-397 guards `seed + i` from a None crash but doesn't address that the production cycle gets fresh entropy every cycle, so VaR/CVaR/price-band numbers in successive `agent_summary.json` snapshots are not directly comparable. Either persist a seed in the agent context (`agent_context_t2.json`) or document that the numbers are sample noise around a mean.

## P3 Findings

- `portfolio/portfolio_mgr.py:19` SEVERITY: `INITIAL_CASH_SEK = 500_000` hardcoded. Same value duplicated in `risk_management.py:114`, `equity_curve.py:17`, `portfolio_validator.py:43`, `journal.py`, and elsewhere. Add a single constants module for portfolio defaults.

- `portfolio/trade_risk_classifier.py:11,20-26` SEVERITY: Magic thresholds (20/10/5 for position size, 60/70 for consensus, 40/25 for concentration, 3/6 for level cutoffs) hardcoded with no config override. Tuning requires code edit and redeploy. Hoist to module-level constants at minimum, or to config.

- `portfolio/exposure_coach.py:88-89` SEVERITY: `new_entries = not (zone == "danger" and regime in ("trending-down", "high-vol"))` — note the regime tuple is "trending-down" (correct), but the surrounding `_REGIME_MULTIPLIERS` and `_BIAS_MAP` use `"range-bound"` (the broken naming above). The new-entries gate would also benefit from including `"ranging"` once the vocabulary is fixed, since a danger-zone ranging market arguably shouldn't allow new entries either.

- `portfolio/trade_guards.py:29` SEVERITY: `LOSS_ESCALATION = {0:1, 1:1, 2:2, 3:4, 4:8}` hardcoded; default ticker cooldown and position-window magic numbers (lines 24-28) likewise. Surfacing to config would let an operator throttle during drawdown without code change. Already partially configurable via `config["trade_guards"]` per lines 121, 131-132, 191-199 — but the LOSS_ESCALATION schedule itself isn't overridable.

- `portfolio/cumulative_tracker.py:23,26` SEVERITY: Hard-coded `_MIN_SNAPSHOT_INTERVAL_SEC = 55*60` and `_CUMULATIVE_CACHE_TTL = 300`. Acceptable but should be config-driven.
