## Important (80-89)

- **[P2->P1 borderline, reported as Important] autonomous.py:421 (`_detect_regime`)** — Reads the wrong key: `sig.get("extra", {}).get("regime")`, but `signal_engine` writes the regime under `extra["_regime"]` (verified at `signal_engine.py:4429`; no code anywhere writes `extra["regime"]` without the underscore — `hurst_regime.py`/`kalman_trend_momentum.py` write `indicators["regime"]`, a different dict). Chain: every signal's `extra["regime"]` is absent → `regimes` list stays empty → `_detect_regime` always returns the `"range-bound"` fallback. That stale regime then flows into the autonomous journal entry (`autonomous.py:163`), the decision log (`:179`), and both Mode A/Mode B Telegram bodies (`:669`, `:796`). When `config.layer2.enabled=false` (autonomous-fallback mode) every journal/notification reports "range-bound" regardless of true regime, corrupting the same `regime` field that downstream consumers (e.g. `agent_invocation._write_fishing_context` reads `journal_entry.get('regime')` at `agent_invocation.py:1346` to pick a position-size multiplier) trust. **Fix:** read `extra.get("_regime")` (and accept the price-based regime values `ranging/trending-up/trending-down/high-vol`), matching `trigger.py:303` and `reporting.py:302`.

## P2 — correctness edge / race / dead detector

- **[P2] loop_contract.py:724 + 1897-1898 (signal_count_stable invariant is a dead detector)** — Both the producer and the consumer key on `extra["active_voters"]`, which is never set on the per-cycle signals dict. `signal_engine` writes `_voters` / `_voters_post_filter` / `_total_applicable` (`signal_engine.py:4415-4417`); the only `active_voters` key in the codebase is a field inside reporting's t2 `signal_density` output (`reporting.py:1344`), not on `sig["extra"]`. Chain: `ViolationTracker._update_signal_counts` (`:1897`) finds no `active_voters` → `counts` is empty → `_previous_signal_counts` never populates → `verify_contract`'s `if previous_signal_counts:` (`:717`) is falsy every cycle → check #8 never runs; even if it did, `extra.get("active_voters", prev_count)` (`:724`) defaults current to prev so `drop_ratio` is always 0. The "signal voter count dropped >30%" safety net has never been able to fire. **Fix:** read `extra.get("_voters_post_filter", extra.get("_voters"))` in both `_update_signal_counts` and the check #8 `current_count` lookup.

- **[P2] loop_contract.py:1262-1265 (`_check_atomic_write_residue` scan budget neuters the detector)** — The orphaned-`.tmp` scan counts `scanned` for *every* `DATA_DIR.rglob("*")` path (directories included) and breaks at `scanned > _TMP_RESIDUE_MAX_FILES` (100). `data/` currently holds 713 files across 13 dirs (verified). `rglob` order is arbitrary, so a stuck `.tmp` left by a crashed `atomic_write_json` (the exact failure this invariant exists to catch — a write that silently never landed) is only detected if it happens to fall in the first ~100 paths. In practice the detector inspects <15% of the tree and the atomic-write-residue safety net is effectively dark. **Fix:** count only `.tmp` candidates against the budget (move the `scanned += 1` after the `is_file()` + name filter), or raise the cap to comfortably exceed the live file count and cap on *matched candidates* rather than total paths.

- **[P2] market_timing.py:244-262 (`_is_agent_window`) + main.py:986-992 (Layer 2 gated to NYSE-ish hours for 24/7 assets)** — `_is_agent_window` returns False on weekends, US/Swedish holidays, and outside `eu_open..close_hour` (07:00-20:00 UTC summer). `main.run` only calls `invoke_agent` when `_is_agent_window()` (`main.py:986`); otherwise it logs `skipped_offhours` and never makes a Layer 2 decision. But the system's primary 24/7 instruments are crypto (BTC/ETH) and metals (XAU/XAG via Binance FAPI) — per MEMORY.md crypto is the secondary focus and metals the primary. Chain: a genuine crypto/metals consensus flip or >2% move overnight or on a weekend fires a trigger, writes the tiered summary, then is dropped with `skipped_offhours` — no Patient/Bold decision, no Telegram analysis from Layer 2. This is established behavior (the autonomous fallback and the separate metals_loop cover off-hours), so it is not a regression, but it is a real coverage gap worth surfacing: the simulated Patient/Bold portfolios cannot react to weekend crypto moves. **Fix (if intended to react 24/7):** widen the window for crypto/metals triggers, e.g. always-allow when the trigger reasons reference only `CRYPTO_SYMBOLS | METALS_SYMBOLS`, gating only stock (MSTR) triggers on NYSE hours.

- **[P2] claude_gate.py:8-9 doc vs agent_invocation.py:1224 (primary Layer 2 invocation bypasses the gate)** — `claude_gate` declares itself "the ONLY approved way to invoke Claude Code" and that direct `subprocess.Popen([claude_cmd, "-p", ...])` is FORBIDDEN because it bypasses the master kill switch (`CLAUDE_ENABLED`), the daily rate-limit warn, the in-process `_invoke_lock`, and the unified invocation log. `agent_invocation.invoke_agent` does exactly that direct Popen (`:1224`) and checks only `config.layer2.enabled` — not `claude_gate.CLAUDE_ENABLED`. Chain: flipping the documented master kill switch `CLAUDE_ENABLED=False` stops silver_monitor/iskbets/bigbet/self-heal but does NOT stop the primary Layer 2 trading agent, and Layer 2 spawns are absent from `claude_invocations.jsonl` (the file `_count_today_invocations` / the weekly cost rollup read), so the busiest caller is invisible to the rate-limit and cost accounting. Impact is bounded (Layer 2 deliberately manages its own watchdog/lifecycle and `CLAUDE_ENABLED` is hardcoded True), so P2. **Fix:** at minimum add a `if not claude_gate.CLAUDE_ENABLED: return False` short-circuit in `invoke_agent`, and mirror the spawn into `claude_invocations.jsonl` so cost/rate accounting sees it.

## P3 — maintainability

- **[P3] agent_invocation.py:1104 and 1294 (bare `return` vs `return False` inconsistency)** — `invoke_agent` is documented to return bool, and almost every gate path returns `False`. Two paths return bare `None` instead: the all-specialists-failed quorum branch (`return` at `:1104`) and an implicit fall-through is avoided, but `:1104`'s bare `return` yields `None`. The caller `main.run` does `result = invoke_agent(...)` then `"invoked" if result else "skipped_busy"` (`:989`) / `f"invoked_{why}" if result ...` (`:964`); `None` is falsy so behavior is correct today, but the type contract is violated and a future caller doing `is False` would misclassify. **Fix:** `return False` at `:1104` for consistency.

- **[P3] autonomous.py:819 (`for r in reasons` with no None-guard)** — `_should_send` earlier normalizes `reasons or []` (`:814`) but then iterates the raw `reasons` directly at `:819`. In production `autonomous_decision` is always called with `reasons_list` (a list, never None — defaulted to `["startup"]` in `main.run:853-854`), so this cannot fire today, but the inconsistency between the guarded `:814` and unguarded `:819` is a latent AttributeError if the function is ever called directly with `reasons=None`. **Fix:** iterate `for r in (reasons or [])`.

## Summary

8 files reviewed in full. No P0/P1 critical loop-killer or money-affecting bug owned by this changeset. One Important wrong-context-key bug: `autonomous._detect_regime` reads `extra["regime"]` instead of `extra["_regime"]`, so autonomous-mode regime is permanently "range-bound" and pollutes journal/notification/fishing-context (`autonomous.py:421`). Two dead-detector P2s in `loop_contract.py` that silently neuter safety nets: the `signal_count_stable` invariant (keys on a never-set `active_voters`) and the `atomic_write_residue` scan (100-path budget vs 713 live files). Two design-gap P2s: Layer 2 is gated to NYSE-ish hours for 24/7 crypto/metals assets, and `invoke_agent` bypasses the documented `claude_gate` master kill switch / rate accounting. Two P3 maintainability nits. The adversarial-focus failure classes (CLAUDECODE, `--bare`/exit-0 auth, tier-timeout/orphan, trigger baseline race, crash backoff, compaction JSON) are all handled correctly.
