## Findings

[P0] agent_invocation.py:825-830 — When `claude_cmd` is missing from PATH, fallback `cmd = ["cmd", "/c", str(agent_bat)]` is built without using the carefully prepared `prompt`. The .bat is invoked with **no prompt input** at all, runs whatever default it has internally, and the silent-auth detection still scans agent.log expecting Claude output. A missing claude binary thus produces a "successful" subprocess that never sees the trigger. | FIX: refuse the fallback unless `agent_bat` accepts and forwards the prompt as argv (e.g., `[..., agent_bat, prompt]`), and validate agent_bat handles it; otherwise return False.

[P0] agent_invocation.py:856-866 — `subprocess.Popen` is created WITHOUT `stdin=subprocess.DEVNULL`. PF_HEADLESS_AGENT is set in env, but if the Claude CLI ever falls through the headless guard and writes a prompt to stdin, the process inherits the parent's stdin (the loop's stdin, which is not interactive when run from Task Scheduler — undefined behavior). Worse, on Windows this can wedge the child waiting on a non-existent stdin. | FIX: pass `stdin=subprocess.DEVNULL` explicitly to Popen.

[P0] multi_agent_layer2.py:168-174 — Specialist Popen also omits `stdin`. Same hang risk: CLAUDE.md startup check + a confused or updated CLI version that asks "How would you like to proceed?" on stdin will hang for the full timeout (90-120s × 3 specialists). PF_HEADLESS_AGENT is honored only if the prompt obeys it. | FIX: pass `stdin=subprocess.DEVNULL` to every claude Popen call site.

[P0] agent_invocation.py:574-598 — The reentrancy check inside `_completion_lock` only verifies an old agent ran past `_agent_timeout`. If an agent is running and `elapsed <= _agent_timeout` the function returns False. But `try_invoke_agent` was renamed to `invoke_agent`; the bool returned in this branch (`return False`) is logged by main as `skipped_busy` (main.py:849). Per loop_contract.py:355-369 `skipped_busy` is excluded from `_LEGITIMATE_SKIP_STATUSES`, so layer2_journal_activity will fire violation/critical_errors entries for what is actually a healthy in-flight invocation that just hasn't finished yet. | FIX: separate "busy with running agent" from "spawn failed" — the former should produce a status that suppresses the L2 contract during the in-flight window (the in-flight check at loop_contract.py:333-341 already exists but only suppresses on `status=="invoked"`, not on `skipped_busy`). Either log status="busy_inflight" or extend the in-flight gate.

[P0] trigger.py:1-357 — `check_triggers` writes `triggered_consensus` baseline in section 1 BEFORE any invocation; the baseline is stored regardless of whether Layer 2 actually got invoked. If `_log_trigger(reasons_list, "skipped_busy", ...)` (main.py:849) or perception_gate skip fires (agent_invocation.py:617-619), the consensus event is consumed and the next cycle no longer sees it as new. The user has documented this exact failure mode for the reverse scenario in trigger.py:243 ("Still update baseline so we don't re-trigger next cycle") for ranging-dampening, but for skipped_busy / skipped_gate / blocked_drawdown / blocked_trade_guards the trigger is **lost permanently** even though Layer 2 never analyzed it. | FIX: persist a "pending_consensus" set; only commit `triggered_consensus[ticker] = action` once Layer 2 acknowledged (or autonomous fired).

[P0] main.py:1115-1126 — `_sleep_for_next_cycle(previous_cycle_started, interval_s)` computes `remaining = interval_s - elapsed`. If a single cycle exceeds `interval_s` (always true in 600s mode if anything goes wrong with LLM batch), `remaining <= 0` and the function logs warning and **returns immediately**. This means after any slow cycle the next cycle starts with NO sleep, and crucially `last_cycle_started` is reset to `time.monotonic()` *after* run() returns (main.py:1246), so the next iteration's anchor is 0. Combined with crash-recovery returning early, this can cause runaway tight-loop on a misbehaving cycle (no minimum sleep). | FIX: enforce a minimum sleep floor (e.g., `time.sleep(max(remaining, 5))`) so a runaway cycle still yields the CPU and gives APIs time to recover.

[P0] agent_invocation.py:1196-1235 — `journal_written` and `telegram_sent` rely on `_journal_ts_before` baseline captured at spawn time. If the JSONL files happen to be **renamed/rotated** between spawn and completion (log_rotation runs hourly — main.py:407), the post-completion `_last_jsonl_ts(JOURNAL_FILE)` reads a fresh empty/new file and `journal_ts_after != _journal_ts_before` evaluates True even for a completely silent agent. False-positive "success" classification — exact the silent-failure mode the auth detector was added to prevent. | FIX: compare against a content fingerprint (last entry's `ts` AND a secondary field like decision-source), or coordinate log_rotation to skip files modified within the in-flight window.

[P1] agent_invocation.py:1190-1194 — When `_journal_ts_before is None` (e.g., journal file didn't exist when invoke_agent was called), the code conservatively forces `journal_written=False` AND `telegram_sent=False`. The status falls through to "incomplete" even if the agent successfully created the journal file from scratch. First-ever invocation always reports `incomplete`. | FIX: when baseline is None, treat existence of any post-completion entry as success.

[P1] agent_invocation.py:539-561 — `_load_config()` is called twice in invoke_agent (once at line 553, again at line 750 inside the multi_agent block). [P1] market_timing.py:43,48 — DST boundary uses `datetime(year, 3, last_sun_mar, 1, 0, tzinfo=UTC)` for **EU** DST start. Real EU DST switches at 01:00 **UTC** (which is 02:00 CET → 03:00 CEST), so the constant is correct. But `_is_eu_dst` is called for "now" datetimes; on the exact day of the switch, between 00:00 and 01:00 UTC, code that prepares CET-displayed warrant session windows will compute the wrong offset because session_calendar.py:50-67 reuses a separate `_eu_dst()` implementation. The two DST functions can diverge for an hour twice a year. | FIX: have session_calendar import `_is_eu_dst` from market_timing instead of duplicating.

[P1] market_timing.py:67-89 — `_is_us_dst` boundary uses `datetime(year, 3, second_sun_mar, 7, 0, tzinfo=UTC)` (07:00 UTC). Actual US DST switch is 02:00 LOCAL (which is 06:00 UTC during EST → 07:00 UTC after switch — but you can't pick "after" as the threshold because before the switch local is EST=UTC-5 → 02:00 EST = 07:00 UTC; AFTER switch local is EDT=UTC-4 → 02:00 EDT = 06:00 UTC). The correct threshold is 07:00 UTC (the EST 02:00 instant), so this is right. Equivalent fall-back at 06:00 UTC for the November transition is also right (02:00 EDT). HOWEVER: `_market_close_hour_utc` returns `20` (EDT) or `21` (EST) — fine for hour-level, but `is_us_stock_market_open` adds minute math that assumes NYSE closes at exactly :00. NYSE actually closes at 16:00 ET = 20:00 UTC EDT or 21:00 UTC EST — fine, but on early-close days (e.g., 2026-07-03, 2026-12-24) it closes at 13:00 ET. `us_market_holidays` does NOT model early-close. | FIX: track early-close days; route through a is_us_stock_market_open(now) that accepts a half-day shortcut.

## Critical observations beyond per-line bugs

1. The "claude -p exit 0 with 'Not logged in'" defense is layered (auth detection in detect_auth_failure + critical_errors journal + timeout + journal/telegram baseline check). However the **overall** safety net has a hole at log_rotation (P0 above) — auth detection only scans `_agent_log_start_offset` to current EOF in agent.log, but if log_rotation rolled agent.log mid-flight the offset is now past EOF of a fresh empty file and no scan occurs.

2. Trigger detection idempotency is **NOT atomic**. trigger_state.json is written via atomic_write_json (good), but the `triggered_consensus` baseline update happens BEFORE invocation success is known (P0 above). Same trigger CAN be lost permanently if the invocation is skipped.

3. Loop crash/restart semantics: `_safe_crash_recovery` guarantees a sleep floor (good). `_consecutive_crashes` persists across process boundaries (good). BUT `_consecutive_stack_overflows` in agent_invocation.py is module-level only — if Layer 2 hits 5 stack overflows in process A, then the loop crashes and respawns as process B, the counter file is read on import (line 171) so it DOES persist. That's correct. However the auto-disable check (line 544) applies module-level state; cross-checking against the `last_invocation_status=auth_error` is not symmetric.

4. DST handling has TWO independent implementations (`market_timing._is_eu_dst` + `session_calendar._eu_dst`) for the same calendar rule. Drift inevitable.

5. Health heartbeat catches 5-minute outage well (300s threshold) but does NOT catch:
   - Loop running but ticker pool returning 0 active signals every cycle (no invariant on signals_ok > 0)
   - Loop running but agent.log is being rotated faster than scans complete
   - Loop running with all 3 specialists timing out every multi_agent invocation (only WARNING)
   - Loop running but `_completion_lock` deadlocked between watchdog and run() — heartbeat would still tick from heartbeat_keepalive thread.

6. Signal report generation: reporting.py is robust against missing modules (each in try/except), but downstream consumers (`_write_compact_summary`, `_write_tier1_summary`, `_write_tier2_summary`) often access `summary["signals"][ticker]["extra"]["_buy_count"]` etc with `.get(..., 0)` defaults — a missing `extra` dict entirely (caused by signal-pipeline failure) produces correct zeros. However `summary["signals"][ticker]["price_usd"]` is sometimes missing when the ticker fully failed and is read without `.get()` in places like reporting.py:1135 (`s.get("price_usd", 0)` — fine). `_atomic_write_json(AGENT_SUMMARY_FILE, summary)` always succeeds last, so the most recent on-disk file is always self-consistent.
