# Claude adversarial review: metals-core

## Summary

Reviewed the metals-core subsystem (~24 kLOC) including the grid market-maker, fishing tools, fin_snipe ladder manager, ORB predictor, mstr_loop strategies, GoldDigger, and Elongir. The grid_fisher is well-instrumented, with thread-isolated session calls and explicit probe mode — but it carries one critical safety hole around the stop-loss endpoint, plus several DST/EOD timing inconsistencies. Cross-process race risk between metals_loop and grid_fisher (both trading XAG/XAU certs on the same Avanza session) is structurally present: there is no shared order/position lock; both rely on the per-account REST endpoints and hope the reconcile loop catches drift. The MSTR loop's "live phase" guard is just a `PHASE` env var with no code-side approval gate. ORB simulator and several precompute paths look reasonable. fin_fish has one knockout-skip bug (BEAR MINI continues instead of breaking) and a hardcoded session-end of 21:55 that ignores `todayClosingTime`.

## P0 — Blockers

- `portfolio/grid_fisher.py:1089-1094` — Stop-loss cancel routed through generic `cancel_order` fallback. The rotate path does `cancel_fn = getattr(self.session, "cancel_stop_loss", None)` and only cancels if the dedicated method exists. The comment on 1085-1088 says regular `cancel_order` on a stop ID returns "crossing prices" errors (March 3 incident). If the injected session lacks `cancel_stop_loss` (custom test session, partial import, attribute typo), the new stop replaces the old without cancelling, leaving TWO stops live on one position. Volume sum > position size will cascade-fill at bid the moment either trigger fires (same Mar 3 incident pattern that motivated the project rule "STOP-LOSS API: use `/_api/trading/stoploss/new`, NOT regular order API"). Fix: hard-require `cancel_stop_loss` on session at construction; refuse to operate if missing.

- `portfolio/grid_fisher.py:1097-1098` + `portfolio/grid_tiers.py:208-223` — Stop placement has NO barrier-proximity check. `build_exit_levels` returns `stop_loss = fill_price * (1 - stop_pct/100)` purely on warrant SEK price. For a MINI cert the warrant price approaches zero as underlying approaches barrier; a 3.5% stop on a cert priced 0.50 SEK ⇒ stop at 0.48 SEK. If the cert is already within 5% of barrier (fishing_levels skipped, but the FILLED inventory may have been bought before drift), the stop sits inside the knockout band and either never triggers (silent stop-out via knockout for full loss) or triggers and cascades against a 0.01 quote. This is the exact Mar 3 incident pattern. The `_tier_skip_for_knockout` guard only applies to OPEN buys, not to stops on existing inventory. Fix: re-run knockout safety against `inst.stop_loss_price` before calling `place_stop_loss`, and raise the stop to barrier+8% if it would land inside.

- `portfolio/mstr_loop/config.py:19` + `execution.py:165-169` — MSTR PHASE switch is just an env var with no approval gate. `PHASE = (os.environ.get("MSTR_LOOP_PHASE") or "shadow").strip()`. The project rule "MSTR loop: shadow phase only (NOT live). Approval gate in docs/MSTR_LOOP_NOTES.md" is enforced ONLY by convention; any process inheriting `MSTR_LOOP_PHASE=live` (e.g. a Scheduled Task with a stray env, or a developer copy-pasting an export) jumps straight into `_live_place_buy` / `_live_place_sell` against Avanza account 1625505. There is no on-disk approval file check, no Telegram confirmation, no signed-config requirement. Fix: require `data/mstr_loop_live_approved.json` with a recent timestamp + checksum before `PHASE == "live"` paths are taken; refuse with a Telegram alert otherwise.

## P1 — High

- `portfolio/grid_fisher.py:256-258, 280-282` — EOD cutoff is hardcoded `21:55 Europe/Stockholm` despite the project rule "Check API for `todayClosingTime` — do NOT hardcode 21:55. Varies with DST." (`.claude/rules/metals-avanza.md`). When DST transitions (Mar 31 / Oct 27) shift Avanza's `todayClosingTime` to 22:55 or 20:55 for the OIL cert (Brent tracks LSE/ICE close, not Stockholm equity hours), the grid will cancel buys 60 min late, leaving inventory unsold past Avanza's actual close. Fix: pull `todayClosingTime` per orderbook from `_api/market-guide/...` and use the earliest across active instruments.

- `portfolio/fin_fish.py:730-735` — BEAR MINI knockout check `pass`-es instead of `continue`. For SHORT direction, `if spot >= barrier: pass` (line 735) lets a knocked-out BEAR warrant through. The LONG branch correctly does `continue` (line 731). Result: when underlying rallies above a BEAR MINI's barrier, the warrant gets selected for fishing and ranked into the plan — but it's been knocked out and the snipe manager will place orders on a dead ISIN. Fix: change `pass` to `continue`.

- `data/metals_loop.py:7549` ↔ `portfolio/grid_fisher.py:tick` — Order races between metals_loop swing trader and grid_fisher on the SAME XAG/XAU ob_ids (`1650161`, `2286417`, `738811`, `1047859` per `GRID_ACTIVE_INSTRUMENTS`). Both share `ACCOUNT_ID = 1625505` and call into `avanza_session.place_buy_order` independently. Grid_fisher's `_state_lock` (line 330) is intra-process only; it doesn't coordinate with `metals_loop.POSITIONS` (line 2057). If the swing trader sells inventory on the same cert the grid is rotating, the grid's `record_fill` accounting drifts (inventory drops but `inst.inventory_units` keeps old value until next reconcile). Reconcile catches it but for 60 s the stop is sized to phantom volume. Fix: pass a shared `account_lock` or read `metals_swing_trader.POSITIONS` to gate grid placement on the same ob_id.

- `portfolio/grid_fisher.py:577-580` — Global-halt threshold scales by `len(state.by_instrument)`, not by ACTIVE instruments. `threshold = -abs(GRID_PER_SESSION_LOSS_LIMIT_SEK) * max(len(state.by_instrument), 1)`. With 3 instruments × 500 SEK = 1 500 SEK global halt floor, even if 2 instruments hold zero inventory and 1 instrument blows past its per-instrument cap. A 600 SEK loss on a single XAG bear (already breaching its 500 SEK per-instrument limit) will not halt globally until the entire 1 500 SEK threshold is hit on aggregate. Fix: track per-instrument breach count and halt globally on 2/3 breaches.

- `portfolio/grid_fisher.py:1418-1424` — `eod_market_flat` falls back to `inst.avg_entry_price` when no live quote is available, then places `aggressive = max(bid * 0.99, 0.01)`. If the live quote returned 0 and avg_entry_price is also 0 (state file corruption, or shadow→live phase transition with stale state), `aggressive = max(0.0 * 0.99, 0.01) = 0.01` — i.e. a market-equivalent sell at 0.01 SEK per unit, dumping the position at near-zero across `inst.inventory_units`. Fix: refuse to place when both bid and avg_entry_price are <=0; log `eod_no_quote_skipped` instead.

- `portfolio/grid_tiers.py:71-106` — Barrier safety uses constant-leverage approximation that ignores parity. For MINI certs `fish_warrant_price = (underlying - barrier) / parity * fx`. Leverage is implicit in barrier proximity, NOT a fixed multiplier. The `warrant_drop_pct / leverage` math is wrong by a factor depending on `(spot - barrier)`, so a tier flagged "safe" at 8% may actually be 2% from barrier. Fix: compute `implied_underlying` from `(tier_price * parity / fx + barrier)` directly when parity is known.

- `portfolio/fin_fish.py:307-313` — `fetch_fx_rate()` silently falls back to `10.0` on any exception. SEK/USD has been in the 10.4-11.0 band since early 2026; rounding to 10.0 underestimates SEK proceeds by ~5-10%, causing position sizing to overshoot. `_select_warrants` ranks warrants by spread (line 681-691) then sizes via `evaluate_warrants` (line 799) which uses this rate. Fix: refuse to plan when FX fetch fails; log and skip cycle instead of silently using a stale-by-9% fallback.

- `portfolio/orb_backtest.py:182-202` — `_simulate_trades` assumes buy-then-sell within the same day but uses only `actual_high` and `actual_low` without time ordering. If the day's high precedes the low (gap-down pattern), the simulator records a "winning trade" even though the buy limit at low fills AFTER the sell target at high already passed — the sell can't fire because there's no inventory. Inflates backbacktest hit rate on negative-skew days. Fix: include `high_hour_utc`/`low_hour_utc` in `BacktestDay` and only count "sell_target_hit" if `high_hour_utc > low_hour_utc` (or filter to morning-direction-up days).

- `portfolio/oil_grid_signal.py:151-176` — Cached signal age check uses `cached["ts"]` parsed as UTC, but `compute_signal()` writes `_utcnow_iso()` formatted `"%Y-%m-%dT%H:%M:%SZ"`. If a future write uses any other format (e.g. `datetime.isoformat()` from a different module that touches the file), `_dt.datetime.strptime(... "%Y-%m-%dT%H:%M:%SZ")` raises ValueError, `cached_dt = None`, then refresh fires every cycle, bypassing the 5-min TTL — hammering `fetch_klines(BZ=F, ...)` every 60 s during a metals_loop tick burst. Fix: use `datetime.fromisoformat()` with the Z-replacement pattern used elsewhere in the repo.

## P2 — Medium

- `data/crypto_loop.py:74-85` — `_pid_alive` lazy-imports `subprocess` inside the `if os.name == "nt"` branch (line 76), then the `except (OSError, subprocess.SubprocessError)` clause at line 84 references `subprocess` even on POSIX where it was never imported. Same module imports `subprocess` at top of file (line 32), so the NameError won't fire here — but `data/oil_loop.py:82` does the import inside the function and DOES have the bug in any case where `os.name != "nt"` AND the POSIX `os.kill(pid, 0)` raises a non-OSError. Codex caught a related path on 2026-05-01 per the comment. Risk: small but persistent on WSL/CI.

- `portfolio/grid_fisher.py:1396-1402` — `eod_market_flat` cancels ARMED sell tiers via the GENERIC `session.cancel_order`, even though some sell-side orders may actually be the stop-loss legs from `rotate_on_buy_fill` (line 1099). Those need `cancel_stop_loss`. The code does branch on `inst.stop_loss_id` (line 1404) to use the right cancel endpoint for the registered stop, BUT any orphaned stop whose ID wasn't recorded back into state (e.g. crashed between place and persist) gets cancelled via the wrong endpoint and hits the "crossing prices" path. Fix: list all open stops and cancel via the stop-specific endpoint before falling back.

- `portfolio/grid_fisher.py:738` — `now_fn` defaults to `_utcnow_iso` (returns str) but `in_cooldown` compares string ISO timestamps lexically. Works because they're both Zulu UTC, but if a caller injects `now_fn=lambda: datetime.now().isoformat()` (local time, no Z), the lexical compare silently breaks and cooldown is incorrectly active or expired. Fix: parse and compare as datetime.

- `portfolio/fin_fish.py:1158` — `hours = 13.67` hardcoded fallback. The fallback runs when zoneinfo + dateutil both fail; on those hosts the planner returns "13.67h remaining" regardless of actual time-of-day, so plans run at 02:00 CET ("13.67h until close") and at 21:30 CET ("13.67h until close") look identical. Drift creeps into fill_probability via the `hours` arg. Fix: refuse to compute fishing levels when tz lookup fails.

- `portfolio/golddigger/config.py:47-50` — Session 08:30-21:30 stops 25 min before commodity warrants close (21:55). Gold cert positions opened near close get flatten-orphaned: `_should_flatten()` fires at 21:30 closing position, but if Avanza session goes flaky between 21:30 and 21:55 the position sits naked until next session. Fix: extend session_end to 21:55 to match commodity hours, OR document that GoldDigger intentionally exits early.

- `portfolio/golddigger/bot.py:33-41` + `portfolio/elongir/risk.py:117-128` — Fallback `_CET` class uses fixed `timedelta(hours=1)`, never returning UTC+2 during CEST. ImportError on `zoneinfo` is unlikely on Python ≥ 3.9 but the fallback creates a 1-hour off-by-one all summer if it ever fires (e.g. truncated Python in a stripped Docker image). Stockholm time in summer = UTC+2; this code returns "CET" with UTC+1. Fix: use `datetime.timezone(timedelta(hours=2))` during DST window, or refuse to run when zoneinfo is unavailable.

- `portfolio/grid_fisher.py:842-853` — `__del__` cleanup uses `cancel_futures=True` but `wait=False` — if the worker is mid-Avanza-call, the call is abandoned without waiting for response. Order may have been ACCEPTED by Avanza but the response (with orderId) is lost; state lacks the order_id, reconcile will incorrectly flag it CANCELLED on next tick (`tier.order_id` is None) and place a duplicate. Fix: `wait=True` with a 5-second timeout.

- `portfolio/fin_snipe_manager.py:1389-1396` — `place_stop_loss_no_page` is called with `valid_days` defaulting to `HARD_STOP_VALID_DAYS = 8` (line 63). Avanza stop-loss orders with `valid_days >= 7` get auto-cancelled on the weekend boundary on some account types. Should match the metals_loop default of 8 days (verified) but the `int(action.get("valid_days") or HARD_STOP_VALID_DAYS)` falsy check means `valid_days=0` collapses to 8 silently. Fix: explicit `is None` check.

- `portfolio/mstr_loop/config.py:71-81` — MSTR session window `15:30-22:00 CET` is hardcoded; comment acknowledges DST gap (Mar 8-29 + Oct 25-Nov 1) shifts real close 1h earlier (21:00 CET during those weeks). EOD flatten at 21:45 means during those DST-mismatch weeks the loop flattens 45 min AFTER NASDAQ already closed — flatten attempts hit an empty book and may park orders that fill at next-day open. Fix: query Alpaca clock API for actual session times.

- `portfolio/fin_fish.py:80-84` — `BINANCE_FAPI_TICKER = "https://fapi.binance.com/fapi/v1/ticker/24hr"` hits the FAPI endpoint with 5 s timeout. No retry, no backoff; on Binance's frequent 1418/1419 rate-limit responses the planner returns empty `result` for that ticker and downstream sizing uses 0. Codex review style would flag silently-empty live data. Fix: route through `portfolio.http_retry`.

- `portfolio/grid_fisher.py:644-661` — Reconcile uses inventory delta (`live_vol - inst.inventory_units`) to distinguish fills from cancels. But `live_vol` is a snapshot at fetch time; if the swing trader (metals_loop) sells inventory between `get_positions` and `get_open_orders` calls within the same tick, the cached inventory is higher than live, delta becomes negative, every armed buy tier gets marked CANCELLED — and reconcile won't realize they actually filled. Fix: fetch positions and orders in one atomic call, or treat large negative drift as inconclusive.

- `portfolio/iskbets.py:147-149` — `cutoff_str = iskbets_cfg.get("entry_cutoff_et", "14:30")` parsed lexically with `int(parts[0])` etc. No DST handling for ET → CET conversion; iskbets is gated by ET wall-clock which is correct, but the surrounding system runs CET and any operator setting `entry_cutoff_et` reading the CLOCK on their wall (CET 20:30) gets a 6h-off cutoff. Operator-doc problem more than code, but worth surfacing.

## P3 — Low

- `portfolio/grid_fisher.py:797` — Rate limit window slides every call, including failed calls (line 798 appends BEFORE the place call's success check). Failed Avanza calls still consume rate-limit budget. Cosmetic but can starve a recovery cycle.

- `portfolio/orb_predictor.py:36-46` — `_morning_window_utc` imports `_is_eu_dst` from `portfolio.market_timing` per call instead of caching at module load. Minor overhead per `calculate_morning_range` call.

- `data/crypto_loop.py:284` / `data/oil_loop.py:297` — `run_loop` function signature `-> int` but no `return 0` at the bottom of the `try` block. Falls through to `finally` and returns None. `sys.exit(None)` exits with 0 on POSIX but the contract is sloppy. Fix: explicit `return 0`.

- `portfolio/fin_fish.py:354` — Annualisation uses `math.sqrt(252.0)` for daily sigma, but the source data is calendar-daily, not trading-daily. For continuous-market metals (24/7 spot) the correct factor is sqrt(365). Underestimates vol by 24%.

- `portfolio/grid_fisher_config.py:120-124` — `GRID_ACTIVE_INSTRUMENTS` ob_ids are hardcoded string literals duplicated in `data/fin_fish_config.py`. Drift risk noted in comment, but no schema or test enforces lockstep. Recommend a CI assertion.

- `portfolio/orb_predictor.py:262-274` — `low_in_range` docstring says "aggressive low < conservative low" but the code at line 73 does `predicted_low_aggressive <= actual_low <= predicted_low_conservative` — correct, but the property `low_in_range` is unused in `_simulate_trades`. Dead code or unfinished feature.

- `portfolio/fish_monitor_smart.py:223-244` — `metals_log.read_text().strip().split("\n")` loads the entire jsonl into memory every cycle. File grows ~1 MB/day; after 30 days a single read is 30 MB per 30 s cycle. Fix: `tail` last N lines or use `file_utils.read_last_jsonl`.

- `portfolio/fin_fish.py:171` — `_DEFAULT_BUDGET_SEK = 20_000` hardcoded ABOVE the global `GRID_GLOBAL_MAX_SEK = 6 500` (grid_fisher_config). If user runs fin_fish standalone with default budget the grid fisher's cap is bypassed because they're separate systems sizing independently. Document or enforce a unified budget.

## Cross-cut observations

- Three independent subsystems (metals_loop swing-trader, grid_fisher market-maker, fin_snipe_manager ladder) write to the SAME Avanza account (1625505) on the SAME orderbook IDs with NO cross-process lock. Each runs its own reconcile/state machine. The "live prices first" + "atomic I/O" rules don't prevent two processes from racing on the same `place_buy_order` endpoint within seconds. Recommendation: introduce a per-(account_id, ob_id) advisory lock file checked before every order placement.
- DST handling is per-module ad hoc: grid_fisher uses Europe/Stockholm zoneinfo correctly; mstr_loop hardcodes CET; orb_predictor branches on `_is_eu_dst` directly; golddigger/elongir use `zoneinfo` but with a broken UTC+1 fallback. There is no centralised market-hours/DST utility used by all metals-core modules. The `memory/market_hours.md` reference suggests one exists but the relevant grid/swing modules don't use it consistently.
- Stop-loss placement scattered across at least 5 call sites (metals_loop `place_stop_loss_orders`, grid_fisher rotate, golddigger runner, fin_snipe_manager apply_actions, mstr_loop). Each carries its own barrier/spread sanity checks (or doesn't). Recommend a single `safe_place_stop_loss` wrapper that enforces barrier proximity, position sizing, and the `_api/trading/stoploss/new` endpoint contract.
- `DRY_RUN` for crypto_loop and oil_loop is correctly gated at both the config level AND the `executor=None` injection — defense in depth. MSTR loop lacks this belt-and-braces: PHASE alone gates live execution. Strong recommendation to mirror the crypto/oil pattern by always passing `executor=None` unless an approval file is present.
- `fin_fish_manager.py` is referenced in the in-scope file list but does not exist in the tree (verified via `wc -l`; codex grep on the same path returned "system cannot find the file specified"). Either the path is stale or the manager was renamed to `fin_snipe_manager.py` (which exists and serves a similar purpose). The in-scope list and architecture comments should be reconciled.
