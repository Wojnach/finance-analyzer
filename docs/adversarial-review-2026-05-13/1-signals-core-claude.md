# Claude adversarial review: signals-core

## Summary
The signals-core subsystem is *mature but creaking under accumulated weight* — 4,206 lines in `signal_engine.py` alone, every fix layered on top of the prior layer with extensive code-comment apologies. Most of the obvious bugs (raw I/O on canonical state, blind try/except) have already been hunted down in prior reviews; this pass surfaces a smaller but still meaningful set of real defects centered around (a) the new soft-vote pipeline (2026-05-11 Stage 2), (b) the SQLite single-connection-shared-across-threads anti-pattern in `signal_db.SignalDB`, (c) NaN/None propagation in `change_pct` defaulting to `0` instead of `None`, and (d) a name collision between `accuracy_by_ticker_signal` (two distinct signatures, two different modules). Several rule-drift items: the persistence filter for metals/crypto effectively disables the persistence contract; the directional-rescue path uses inconsistent default fallbacks; `_BIAS_MIN_ACTIVE` reads `samples` which is total-votes including HOLD (silently wrong).

## P0 — Blockers (production breakage / data loss / silent wrong trades)

- `portfolio/signal_db.py:24-90` — **`SignalDB` shares a single `sqlite3.Connection` across threads without `check_same_thread=False` or a lock.** `__init__` lazily creates `self._conn = sqlite3.connect(...)` and every public method (`insert_snapshot`, `update_outcome`, `load_entries`, `signal_accuracy`, `consensus_accuracy`, `per_ticker_accuracy`, `ticker_signal_accuracy`) reuses `self._get_conn()`. Python's `sqlite3` raises `ProgrammingError: SQLite objects created in a thread can only be used in that same thread` by default. The mitigating fact today is that callers create a fresh `SignalDB()` per call (`outcome_tracker.py:162`, `accuracy_stats.py:148`), so the connection is single-thread by accident. But `outcome_tracker.backfill_outcomes` creates `_db = SignalDB()` and then calls `_db.update_outcome` inside a for-loop that *also* may yield to other ticker threads (no — backfill is single-threaded; OK). The real bite is `accuracy_stats.load_entries`, which is called from inside `_accuracy_compute_lock`-protected `get_or_compute_*` paths from 5 ticker threads. Each call creates a new connection (one per thread) which is fine FOR NOW, but the doc-comment "SignalDB() can be reused across calls" and the `_conn` cache pattern is a footgun primed to detonate the moment somebody refactors `load_entries` to share a module-level `SignalDB`. Why it bites: silent `ProgrammingError` swallowed by `except Exception` at `accuracy_stats.py:157` → empty entries → fail-closed all-signal HOLD via `_accuracy_failed` path at `signal_engine.py:3886`. Fix: pass `check_same_thread=False` to `sqlite3.connect`, OR add a `threading.Lock` around `_get_conn()` and every method. WAL mode does not provide thread safety for shared connections.

- `portfolio/signal_engine.py:566-625` — **Persistence filter effectively disabled for metals + crypto in steady state.** `_PERSISTENCE_CYCLES_BY_ASSET = {"METALS": 1, "CRYPTO": 1, "STOCK": 2}` combined with the `if 1 >= min_cycles: filtered[sig] = vote` branch at line 613 means *every fresh non-HOLD vote* (cold start, direction flip, or first cycle in a long quiet period) is passed through immediately. The documented contract at the top of the module ("require 2+ consecutive same-direction votes ... eliminates documented single-check noise pattern") is *false* for 4 of 5 Tier-1 tickers. The user-memory rule `Single-check MACD/RSI/volume improvements are noise — need 3+ sustained` is the explicit reason persistence was added. With `min_cycles=1`, persistence is a no-op for the tickers it was meant to protect most. Why it bites: feeds whiplash flips into `_weighted_consensus`, which the soft-vote dampening cannot offset. Fix: either restore `min_cycles >= 2` for metals + crypto, or document the trade clearly in CLAUDE.md and remove the misleading "persistence filter" name.

- `portfolio/signal_engine.py:2593-2599` — **Directional bias penalty reads `samples = act_data.get("samples", 0)` but `signal_activation_rates` populates `samples` with the TOTAL vote count including HOLDs** (`accuracy_stats.py:850` writes `"samples": total` where `total = s["total"]` and `s["total"]` is incremented unconditionally at line 807). The threshold `_BIAS_MIN_ACTIVE = 30` was meant to require "enough active (non-HOLD) votes to judge bias" — quoted at line 506. With HOLD-dominated signals (90%+ HOLD), `samples=300` actually corresponds to ~30 active votes, but for a signal with 50% activation, `samples=30` corresponds to only 15 active votes — well below the intended floor. So *low-activation high-bias signals get penalized at half the active-sample floor the designer intended.* Why it bites: prematurely halves the weight of rare but informative directional signals (e.g., `news_event`, `hash_ribbons`). Fix: use `act_data["activation_rate"] * samples` or compute active samples directly.

## P1 — High (will cause incidents)

- `portfolio/accuracy_stats.py:556-575` and `portfolio/ticker_accuracy.py:16-82` — **Two distinct functions both named `accuracy_by_ticker_signal` with incompatible signatures.** `accuracy_stats.accuracy_by_ticker_signal(horizon, min_samples)` returns `{ticker: {signal: ...}}`. `ticker_accuracy.accuracy_by_ticker_signal(ticker, horizon, days)` returns `{signal: ...}` for ONE ticker. `signal_engine.py:3712` imports `accuracy_by_ticker_signal_cached` from `accuracy_stats` — fine. But a Layer 2 / dashboard / new caller doing `from portfolio.accuracy_stats import accuracy_by_ticker_signal` vs `from portfolio.ticker_accuracy import accuracy_by_ticker_signal` gets entirely different semantics. Why it bites: the next refactor that lands an import statement on the wrong line silently corrupts probability or accuracy plumbing. Fix: rename one (e.g., `ticker_accuracy.signal_accuracy_for_ticker`).

- `portfolio/signal_engine.py:3593-3608` — **Stage 2 Batch 2 dead-zone soft votes for candlestick + forecast skip the `_TICKER_DISABLED_BY_HORIZON[H][ticker]` blacklist.** Lines 3592-3594 check `DISABLED_SIGNALS` and `_TICKER_DISABLED_SIGNALS.get(ticker, ())` (the `_default` slice only), but never check the horizon-specific blacklist `_TICKER_DISABLED_BY_HORIZON[horizon][ticker]`. So a signal blacklisted at 1d for a ticker (e.g., candlestick disabled at 1d for XAU per line 823) can still emit a soft vote here, which then survives until the horizon-specific gate runs inside `_weighted_consensus` (line 2248-2250). Why it bites: silent vote pollution at 1d for tickers with known bad accuracy on these signals. Fix: also gate via `_get_horizon_disabled_signals(ticker, horizon)`.

- `portfolio/signal_engine.py:1862-1883` — **`_dynamic_min_voters_for_regime` raises the floor for `ranging`/`unknown`/`None` regimes to 5, but the assertion at lines 981-985 only requires `_POST_EXCLUSION_MIN <= _MIN_ACTIVE_VOTERS_SOFT (5)`**. There is no assertion that the regime quorum (5) is reachable given `MIN_VOTERS_METALS = 2`. Metals tickers in ranging/unknown regime hit `dynamic_min=5` in Stage 4 (`apply_confidence_penalties` line 2911) but the upstream MIN_VOTERS gate (`signal_engine.py:3811-3815`) accepts the action with `min_voters=2`. The downstream Stage-4 gate then force-HOLDs it. Net effect: in steady-state ranging metals, **no trade ever fires regardless of signal alignment**, but the system spends compute as if it would. Why it bites: silent zero-trade quiet period; also masks signal quality regressions because nothing reaches outcomes. Fix: either harmonise `_dynamic_min_voters_for_regime` to honour the asset-class floor, or document that metals will be HOLD-only in ranging regime.

- `portfolio/accuracy_stats.py:344, 1344` (and elsewhere) — **`change_pct = outcome.get("change_pct", 0)` defaults missing change_pct to 0.0 instead of None.** `_vote_correct` properly handles `None` (line 182), but only when it receives `None`. Defaulting to 0 makes the path `if change_pct is None: return None` skip, then `abs(0) < threshold` → returns None. So functionally equivalent today, but the line `change_pct = outcome.get("change_pct", 0)` propagates a *typed* 0 into other arithmetic (e.g., `linear_factor.py` and `train_signal_weights.py:73-78` later in the pipeline). `train_signal_weights.py:73` is the explicit one: `change_pct = outcome if isinstance(outcome, (int, float)) else outcome.get("change_pct")` — then checks `if change_pct is None: continue`. But if the source pipeline replaces `None` with 0 anywhere upstream, those entries silently become 0% return training samples. Fix: change to `outcome.get("change_pct")` (no default) and let downstream None-guards work.

- `portfolio/signal_engine.py:2467-2469, 2485` — **`_safe_accuracy` is used in `_count_active_voters_at_gate` but NOT in `_weighted_consensus` main loop.** Line 2467: `stats = accuracy_data.get(signal_name, {})`; line 2468: `acc = stats.get("accuracy", 0.5)`. With the same dataset, `_count_active_voters_at_gate` (line 1821) correctly coerces None/NaN to 0.5 via `_safe_accuracy`, but the main weighting loop reads the raw `stats.get("accuracy", 0.5)`. The sanitization loop at line 2196-2233 strips poisoned `accuracy` keys, so the raw `.get(..., 0.5)` lands the safe default — *but only because of the sanitisation step*. If sanitisation skips one key (e.g., `accuracy=0.0` literal which is "clean" by `_safe_accuracy` but means total disagreement), `acc=0.0` flows into `weight = acc` at line 2531 → zero weight, no vote. Marginal but inconsistent. Fix: route both paths through the same coercion.

- `portfolio/signal_engine.py:3741, 2327` — **Per-ticker exemption (BUG-158) uses `_ticker_acc_data` loaded for `acc_horizon` which is `"1d"` for any horizon outside `("3h","4h","12h")`** (line 3713). That means 3d/5d/10d horizon consensus pulls 1d per-ticker accuracy when deciding regime-gate exemption. A signal that is 60% accurate on `XAG-USD` at 1d but 35% at 5d gets exempted from the 5d regime gate. Why it bites: false BUY/SELL passes through 5d consensus in regimes where it's gated for good reason. Fix: derive `acc_horizon` symmetrically (`horizon if horizon in ALL_HORIZONS else "1d"`) and pass it to both upstream and downstream calls.

- `portfolio/signal_engine.py:566-595` — **Cold-start branch seeds state with `min_cycles if vote != "HOLD" else 0`** then unconditionally `return votes` (line 595). The cold-start path bypasses the filter regardless of `min_cycles`. With persistence enabled for stocks (`min_cycles=2`), the very first cycle after a process restart passes ALL non-HOLD votes through unfiltered (the comment says "first cycle — trust all signals"). This is fine in isolation, but it means a fresh restart immediately after a regime flip lets stale-direction votes through without confirmation, defeating the persistence guarantee at exactly the moment it's most needed. Why it bites: restart-driven whiplash. Fix: on cold start, seed all signals to `cycles=0` and require the natural 2-cycle ramp.

## P2 — Medium (correctness / robustness)

- `portfolio/signal_engine.py:2270` — `_topn_accuracy_key` uses `accuracy_data.get(s, {}).get("accuracy", 0.5)` with the same raw-`.get` pattern noted in P1 above. A missing-data signal lands at 0.5 — which is below any signal with measured 0.5+ accuracy, so it ranks LAST. That's actually safe (it gets excluded by top-N) but means newer/unmeasured signals are systematically de-prioritised, suppressing the very accuracy data they need to accumulate.

- `portfolio/signal_engine.py:1644-1654` — `_get_correlation_groups` returns `{}` when `_cached` yields None/empty, then the caller at line 2290 falls back to `_STATIC_CORRELATION_GROUPS`. But on a transient `_compute_dynamic_correlation_groups` exception (line 1638), `_cached` will *cache the empty dict for `_DYNAMIC_CORR_TTL = 7200`s* — 2 hours of fallback to static groups even after the transient is fixed. Fix: don't cache the empty/failure result, or have `_cached` distinguish "not yet computed" from "computed and empty".

- `portfolio/accuracy_stats.py:280-289, 291-385` — **`signal_accuracy_ewma` ignores entries with `change_pct=None`** (the underlying `_vote_correct` returns None and it's `continue`'d at line 353) but **does NOT count them in `null_change_pct_skipped`** like `signal_accuracy` at line 249 does. Diagnostic blind spot: the EWMA path can silently degrade on outcome backfill regressions while the standard path's log line correctly surfaces the issue.

- `portfolio/signal_engine.py:2541-2548` — **IC per-ticker fallback `ic_per_ticker.get(ticker, {}).get(signal_name)` falls through to global IC when `_ic_info.get("samples", 0) < _IC_MIN_SAMPLES` (100)**. But `_compute_signal_ic_per_ticker` writes `"samples": n` ONLY for signals with `n >= MIN_SAMPLES_FOR_IC = 30` (line 209-218). Signals with 30-99 ticker-specific samples get a per-ticker IC value AND a samples count, and line 2545 demotes them to global IC. That's OK — but the *short-circuit comment* "fall back to global" implies recovery; in reality the global IC is computed over ALL tickers and may be opposite-signed for this ticker. Better to require `_IC_MIN_SAMPLES` for the global fallback too, or document the contract more clearly.

- `portfolio/accuracy_stats.py:919, 923, 925, 927` — `blend_accuracy_data` early-return at `blended = 0.5` (immature signal) **loses the directional fields** (the `_directionals_trustworthy` guard at line 947 sets them only when one of the sample conditions holds). Combined with the IC short-circuit above, an immature signal gets `accuracy=0.5` but no `buy_accuracy`/`sell_accuracy` keys. The directional gate at `signal_engine.py:2513-2521` uses `stats.get("buy_accuracy", acc)` — falls back to 0.5 — so the gate doesn't fire. That's intentional and safe, but on the SAME row the directional-rescue path at line 2488-2495 uses `stats.get("buy_accuracy", 0.0)` (note: 0.0 default, NOT `acc`). An immature signal's rescue check evaluates rescue_acc=0.0 < threshold(0.55) → no rescue. So rescue NEVER fires for immature signals. Probably intended, but inconsistent defaults across two adjacent code paths is a maintenance hazard.

- `portfolio/signal_engine.py:3083-3091` — **MACD dead-zone soft-vote helper `_macd_dead_zone_vote` is called when `ind["macd_hist"] > 0 and ind["macd_hist_prev"] <= 0` is False AND `ind["macd_hist"] < 0 and ind["macd_hist_prev"] >= 0` is False.** That excludes the actual zero-crossing boundary cases (`hist_prev == 0` with `hist == 0` lands in the else branch, fine). But the helper also recomputes the histogram from scratch (line 308-315) when `ind` already carries the current value — wasted compute. Performance, not correctness.

- `portfolio/signal_engine.py:2326, 2389` — `leader_samples = leader_stats.get("total", 0)` and `accuracy_data.get(s, {}).get("total", 0)` use raw `.get` without `_safe_sample_count`. After the sanitisation pass at line 2196 drops poisoned counts, missing-key defaults to 0 — correct — but a partially-written cache row with `total = -1` (impossible today, but the `_safe_sample_count` function exists exactly for this defence) bypasses the check.

- `portfolio/signal_engine.py:3878-3880` — `blend_accuracy_data` is called with `min_recent_samples=_RECENCY_MIN_SAMPLES (30)`, matching the `_BLEND_DEFAULT_MIN_RECENT_SAMPLES = 30`. Good. BUT the docstring at line 477-486 documents `_RECENCY_MIN_SAMPLES = 30  # match ACCURACY_GATE_MIN_SAMPLES (was 50 default)` — the explicit alignment is fine, but a careful reader will notice that `_RECENCY_DIVERGENCE_THRESHOLD = 0.15` and `_RECENCY_WEIGHT_FAST = 0.90` produce a 90% recent weight for a 30-sample window when divergence > 15pp. 30 samples × 0.90 weight against a 10K sample baseline at 0.10 weight gives essentially raw 30-sample noise. The "regime adaptation" is fast enough to be reactive to randomness. Document or raise `_RECENCY_MIN_SAMPLES` to 50.

- `portfolio/signal_history.py:64-98` — **`update_history` holds `_history_lock` across the FULL read-modify-write including `_save_history` (atomic_write_jsonl)** which itself takes a sidecar lockfile. With `MAX_ENTRIES_PER_TICKER = 50` × 5 tickers = 250 entries, the JSONL is small (~25KB), so the contention is mild — but it scales linearly with ticker count.

- `portfolio/signal_engine.py:1396-1408` — `_compute_dynamic_horizon_weights` early returns `HORIZON_SIGNAL_WEIGHTS.get(horizon, {})` (the STATIC fallback) when *insufficient samples* or *no cache*. So far so good. But the success path (line 1392-1408) returns `weights if weights else HORIZON_SIGNAL_WEIGHTS.get(horizon, {})` — if the dynamic compute produces NO weights outside the deadband, it silently falls back to static. Then `_get_horizon_weights` caches the *static fallback* under `dynamic_horizon_weights_{horizon}` for `_DYNAMIC_HORIZON_WEIGHT_TTL = 3600`s. - `portfolio/signal_engine.py:4198-4204` — `_cross_ticker_consensus[ticker] = {"action": action, "confidence": conf}` is set unconditionally per ticker per cycle. For `ticker=None` (tests), it would crash on `_cross_ticker_consensus[None] = ...` — but the `if ticker:` guard at line 4202 prevents that. OK. But the cache has no TTL: a ticker removed from the live universe leaves a stale entry forever. Memory leak is bounded by ticker churn (~once per quarter), not severe.

- `portfolio/outcome_tracker.py:282-339` — **Cross-platform lock implementation uses `msvcrt.locking` (1-byte range) and `fcntl.flock`**, but `_signal_log_lock_path` returns a *sidecar* lockfile at `data/.signal_log.jsonl.lock`. `file_utils.atomic_append_jsonl` should use the same lockfile path (per the comment at line 282-289). - `portfolio/forecast_accuracy.py:130-140` — `entry.get("outcome", {}).get(horizon)` then `outcome.get("change_pct", 0)`. Same `change_pct=0` default pattern as accuracy_stats. - `portfolio/accuracy_stats.py:1727-1734` — `BEST_HORIZON_CACHE_FILE` cache check uses `cached.get("time", 0)` — fine — but the **success path at line 1798-1802 writes via `_atomic_write_json` only when result is non-empty**. If two ticker threads race to insert with the same `ts` (shouldn't happen — `ts` comes from `datetime.now(UTC).isoformat()` at microsecond resolution, but on Windows the resolution is sometimes 15.6ms), one insert wins, the other returns success without committing its `ticker_signals` rows. For a 168-row test window × 30 signals, that's 5040 dict lookups + the model walk. Vectorised matrix multiply would be ~100× faster. - `portfolio/signal_engine.py:2358` — **`leader_directions = {sc: votes.get(ldr, "HOLD") for sc, ldr in meta_leaders.items()}` reads `votes` AFTER `votes` was mutated by the macro-window force-HOLD pre-pass at line 2258-2262.** So a leader that was force-HOLD'd by macro now reads as HOLD, which then `active_dirs = set(...) - {"HOLD"}` skips. - `portfolio/signal_engine.py:611-616` — The conditional `if 1 >= min_cycles` at line 613 evaluates as `if min_cycles <= 1`. Reading the code, this is the persistence-bypass branch (matched at P0 above). The `1 >= min_cycles` form is non-idiomatic and obscures intent — `if min_cycles <= 1` would read as "treat min_cycles=1 like cold-start" immediately.

- `portfolio/signal_postmortem.py:185-249` — `generate_postmortem` reads `acc = load_cached_accuracy("1d")` then falls through to `signal_accuracy("1d")` if cache missing. ## Cross-cutting observations

- **Atomic I/O compliance is GOOD across the in-scope files.** The few `open()` calls left in `outcome_tracker.py:377-401` are intentional (binary scan to count lines without JSON parsing) and the rewrite path uses `tempfile.mkstemp + os.replace` with a sidecar lockfile. `signal_db.py` uses raw `sqlite3` (necessarily, no helper), but writes commit/rollback correctly. The only real concern is the *single shared connection* anti-pattern (P0 above).

- **`change_pct = outcome.get("change_pct", 0)` is repeated in many call sites** (accuracy_stats.py:225, 344, 1344, etc.). Even though every downstream uses `_vote_correct` to filter, the convention is unhygienic — it lets a None silently morph into 0 if any future code path skips the helper. Migrate to `outcome.get("change_pct")` everywhere and let None propagate.

- **The "never invert sub-50% — force HOLD instead" rule is honoured** in `_weighted_consensus` (gate at line 2485 force-skips, doesn't invert). The directional rescue path at line 2486-2507 also force-skips when neither direction has rescue accuracy. Confirmed clean.

- **MIN_VOTERS_METALS = 2 violates the project rule "MIN_VOTERS = 3 for all asset classes"** (CLAUDE.md, `.claude/rules/signals.md`). Line 956-960 documents the rationale (intraday metals horizon), but the rule is explicit. This is *de facto* drift from the spec; should be reconciled in the rule doc, not silently in code.

- **`_safe_accuracy` exists, `_safe_sample_count` exists, but neither is used consistently.** The sanitisation pass at line 2196-2233 is the primary defence, then downstream `.get(..., default)` patterns rely on the cleanup having already happened. If sanitisation is bypassed (regression, new caller, partial cache write skipping the pass), the unsafe `.get` paths leak NaN through to weight arithmetic. Defence-in-depth would route every `accuracy_data[sig]` access through the safe helpers.

- **Two-tier IC fallback logic (`per_ticker` → `global` with the `_IC_MIN_SAMPLES` floor)** has subtle defaults: per-ticker IC is fine-grained, global IC is a different population. Falling back can flip the sign of the multiplier. Consider rejecting (default 1.0) when per-ticker IC is below floor *instead* of falling back to global.

- **`signal_engine.py` at 4,206 lines is the largest single Python file in the repo and bears the marks of accumulated risk.** Every gate is conditioned on multiple state variables (regime, horizon, ticker, sample-tier, macro window, crisis, persistence, IC stability...). The matrix of permutations is unmanageable by inspection; the `_count_active_voters_at_gate` helper exists exactly to centralise one slice of that matrix but is one of many. A refactor pass extracting `_weighted_consensus` and `apply_confidence_penalties` into named pipeline stages with explicit input/output contracts would let each gate be tested in isolation. This is not a defect today, but it is the *probability that next defect goes undetected*.

- **Several dead-code references survive in `_default` ticker blacklists and regime gates** (e.g., `fibonacci` in `SLOW_SIGNALS_3H` at `short_horizon.py:13`, `funding` regime gate at line 1149 with a comment about 1d gating despite `funding` being crypto-only). The blacklists are append-only over months of audits; a quarterly prune of disabled signals from these lists would help the code match the live behaviour.
