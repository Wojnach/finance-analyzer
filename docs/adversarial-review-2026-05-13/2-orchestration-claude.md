# Claude adversarial review: orchestration

## Summary
Three direct-Popen sites still bypass `claude_gate` (P0). `agent_invocation` and `multi_agent_layer2` Popen Claude without a process group, so the kill path leaves grandchildren orphaned on Windows fallback and on POSIX. The first-cycle `force_report=True` flow always invokes Layer 2 on startup grace because trigger.py emits "no triggers" but main.py still classifies T3, defeating the grace period entirely. Multiple read-modify-write paths on `health_state.json` race the `_health_lock`, and the auth-scan `_agent_log_start_offset` is broken by log rotation.

## P0 — Blockers

- `portfolio/analyze.py:282` and `portfolio/analyze.py:746` — Direct `subprocess.run(["claude", "-p", prompt, ...])` calls bypass `claude_gate.invoke_claude_text`. CLAUDE.md rule and the module-level kill switch (`CLAUDE_ENABLED`) cannot disable these calls, and they don't appear in `claude_invocations.jsonl` cost tracking. They also don't perform auth-failure detection on stdout — `detect_auth_failure` IS called at line 296-300, but only AFTER the bypassed call completes, so a hung claude-with-not-logged-in still wastes the 60-120s timeout while burning subprocess slots. The whole point of the gate per CLAUDE.md ("Direct subprocess.Popen calls are FORBIDDEN"). Why it bites: every `/fin-analyze` cycle or `--watch` invocation circumvents the kill switch, rate-limit warning, in-process serialization (`_invoke_lock`), and token+cost accounting that bigbet.py was already migrated to (line 176-182 of bigbet.py shows the correct pattern). Fix: route both call sites through `claude_gate.invoke_claude_text` with caller="analyze_cli" / "watch_positions".

- `portfolio/agent_invocation.py:872` — `subprocess.Popen` for the primary Layer 2 agent does NOT set `creationflags=CREATE_NEW_PROCESS_GROUP` (Windows) or `start_new_session=True` (POSIX). On POSIX, `_kill_overrun_agent` at line 503 does `_agent_proc.kill()` — which signals only the direct child, NOT the Node.js helpers Claude CLI spawns (MCP servers, API client child). Grandchildren leak file handles / VRAM (if any local LLM was loaded) until they exit on their own. On Windows the kill uses `taskkill /F /T` (line 489) which does walk the tree — but only IFF the PID hasn't been reused, which is more likely when there's no process group barrier. Why it bites: the same incident class the `claude_gate._popen_kwargs_for_tree_kill` + `_kill_process_tree` helpers were written to prevent (Codex A-IN-2 2026-04-11), still wide open on the largest Claude consumer in the codebase. Fix: import `_popen_kwargs_for_tree_kill` + `_kill_process_tree` from `claude_gate` and use them.

- `portfolio/multi_agent_layer2.py:171` — Same defect as above for specialist subprocesses. Each of 3 specialists runs Claude CLI with `proc.kill()` cleanup at line 210 (no `creationflags`, no `start_new_session`). Worst case after a 30s `specialist_timeout_s` expiration: 3 orphaned Node.js parent processes + their MCP children, all of which the next call to `_invoke_lock` will queue behind. Also bypasses `claude_gate.invoke_claude` entirely — no kill switch, no log entry. Fix: route through `claude_gate.invoke_claude` (and accept the loss of true parallelism — at most we should serialise via the existing in-process lock anyway; the comment at line 770 "TODO: run specialists in background thread, collect results async" implies the parallel path was always provisional).

- `portfolio/main.py:1188` + `portfolio/trigger.py:170-199` — Startup grace is defeated by `force_report=True`. The initial run passes `force_report=True`, then at `main.py:809-822` the trigger path runs unconditionally: `reasons_list = reasons if reasons else ["startup"]`, then `classify_tier(["startup"])`. Inside `classify_tier` at trigger.py:458 the "first trigger of day" check fires (`state.get("last_trigger_date") != _today_str()` because no triggers were saved during grace), returning **T3**. So every loop restart spawns a T3 (900s/40 turns) Layer 2 invocation even though `check_triggers` correctly suppressed all actual triggers. Why it bites: a Task Scheduler restart-storm (e.g. PF-DataLoop logon + auto-restart 30s delay) burns Claude quota on a T3 every reboot. This is exactly the "spurious T3 full reviews every time the loop is restarted for a code update" the grace period was added to prevent (comment at trigger.py:66-68). Fix: in `main.py:run()`, when `force_report=True` AND `triggered=False`, write the summary but DO NOT invoke Layer 2 — explicitly bail before the `_is_agent_window()` block. Alternatively, classify_tier should detect the `["startup"]` reason as non-actionable.

- `portfolio/agent_invocation.py:904-916` — `health_state.json` read-modify-write WITHOUT `_health_lock`. `health.py` defines `_health_lock` (line 17) and every health module entry point (`update_health`, `heartbeat`, `reset_session_start`, `update_module_failures`, `update_signal_health_batch`) takes it. agent_invocation.py's "publish last_invocation_tier" path does NOT — it does its own `load_json` + `atomic_write_json` directly. While the loop is mid-flight, `heartbeat_keepalive` is ticking the heartbeat every 60s on a daemon thread (main.py:847) — that thread holds `_health_lock` for a write while agent_invocation has a stale read. Last-writer-wins: the heartbeat writer (which doesn't know about `last_invocation_tier`) reads health_state from BEFORE agent_invocation's write, modifies its own field, and writes back — silently dropping `last_invocation_tier`. Result: `_get_layer2_grace_s` in loop_contract.py:215 sees no tier, defaults to T3 grace (20m), even though the actual call was T1 — and the journal-activity contract waits 20m instead of 12m before firing. Fix: import `_health_lock` from `portfolio.health` (or expose a `update_health_field` helper there) and wrap the read-modify-write block.

## P1 — High

- `portfolio/agent_invocation.py:839` — `_agent_log_start_offset` is captured before subprocess spawn, then `_scan_agent_log_for_auth_failure` at line 428 does `f.seek(_agent_log_start_offset)`. If `portfolio/log_rotation.py` rotates `agent.log` between spawn and completion (which `_run_post_cycle` calls every hour via `rotate_all()` at main.py:408), the offset now points past EOF (rotated file is renamed; new agent.log is fresh). `f.seek(N)` past EOF on a smaller file returns successfully and `f.read()` returns empty bytes — auth-error detection silently no-ops. The bug is silent: claude prints "Not logged in" → auth scan reads `b""` → no auth_failure recorded → invocation marked as completion with status "incomplete" → operators see nothing in critical_errors. This is the same class of failure the agent_log scan was added to prevent. Fix: detect rotation by storing the inode (`stat().st_ino` on POSIX) or the file size at offset time + verifying current size >= offset on read; if not, scan from byte 0.

- `portfolio/main.py:842` — `if os.environ.get("NO_TELEGRAM")` is checked BEFORE `_is_agent_window()`. Result: NO_TELEGRAM short-circuits Layer 2 entirely (logs "[NO_TELEGRAM] Skipping agent invocation"), even outside the agent window. The original intent of NO_TELEGRAM was clearly "don't send messages", but it now ALSO disables the entire trigger system. Tests that import this module with NO_TELEGRAM=1 in env will silently skip the L2 path and produce false-pass results. Fix: check agent_window first, NO_TELEGRAM only on the send sites (`send_or_store` already honors it).

- `portfolio/main.py:1192` — Heartbeat write after initial run uses `atomic_write_text(DATA_DIR / "heartbeat.txt", datetime.now(UTC).isoformat())`, but `health.heartbeat()` writes to `data/health_state.json` (line 65-86 of health.py) and is what `loop_contract.check_layer2_journal_activity` reads via `load_json(HEALTH_STATE_FILE)`. So there are TWO heartbeat sinks — `heartbeat.txt` (used by `loop()` line 1145-1158 for crash detection) and `health_state.json` (used by everything else). Drift between them means a stale-but-existing `heartbeat.txt` says "loop alive" while `health_state.json.last_heartbeat` says "300s+ stale". Dashboard `/api/health` reads only `health_state.json`. Why it bites: a partially-hung loop where `atomic_write_text` succeeds but the `health_state.json` write fails (disk-full mid-cycle) reports "fresh" to crash-detection and "stale" to monitoring. Fix: drop `heartbeat.txt` entirely and have `loop()` consult `check_staleness()` from health.py for the previous-loop-crashed check.

- `portfolio/trigger.py:69` `_startup_grace_active` is module-level mutable state. If main.py is imported twice in the same process (test runner, dashboard side-load), or if `check_triggers` is called from a worker thread that didn't go through the `loop()` entry point, the grace flag is shared across all instances. Combined with the `saved_pid != current_pid` check at line 170, this works for forks but breaks for in-process re-entry (tests calling `run()` in a loop). Result: the grace is consumed on the first call and never re-applied. Fix: tie grace to a per-PID + per-thread state, or accept a `force_no_grace` parameter on `check_triggers`.

- `portfolio/agent_invocation.py:1331-1348` — In `_check_agent_completion_locked` cleanup, `_agent_log = None` etc. are cleared, but `_patient_txn_count_before` and `_bold_txn_count_before` are global mutable state and only reset at line 1347-1348 of the **happy completion path**. The **timeout path** at line 1154 (`_kill_overrun_agent`) returns early without resetting these. Next invocation snapshots the new counts at line 886-891, but if the new invocation hits `invoke_agent` reentrancy block (line 574-598) and bails because the old `_agent_proc` was killed but a new one is still running, the txn counts are NEVER updated. `_record_new_trades` at line 1086 then compares "current" to a count_before from a prior successful invocation — registering stale "new" trades that already happened hours ago. Fix: also reset these counters at the bottom of `_kill_overrun_agent`.

- `portfolio/agent_invocation.py:574-598` — The reentrancy check holds `_completion_lock` while calling `_kill_overrun_agent`, which in turn calls `subprocess.run(["taskkill", ...])` and `_agent_proc.wait(timeout=15)`. The watchdog thread takes the same lock every 30s. If kill takes >30s (`wait(timeout=15)` after taskkill could block on a hung Node), the watchdog thread queues behind us. Worse: any other caller (test, manual `check_agent_completion`) blocks. Holding the lock across a 15-30s subprocess.wait is the same pattern that motivated the watchdog in the first place. Fix: release the lock around `subprocess.run`/`wait` once the kill signal is sent; just hold it for the state-clear at the end.

- `portfolio/agent_invocation.py:481` — `_kill_overrun_agent` uses `platform.system() == "Windows"` to decide between `taskkill /F /T` and `_agent_proc.kill()`. POSIX path (line 503) is just `_agent_proc.kill()` — no process group, no descendants. Combined with P0 issue (no `start_new_session=True` at Popen), Linux/WSL agents leak grandchildren on every timeout. The codebase has `claude_gate._kill_process_tree` which handles this correctly with `os.killpg` — but agent_invocation reinvented the wheel and got it wrong. Fix: use the shared helper.

- `portfolio/multi_agent_layer2.py:218-230` — Auth-error scan happens inside the `for proc in procs` loop, which means: (1) it runs synchronously per-process AFTER the specialist `wait` (good), but (2) the `name` variable comes from `getattr(proc, "_name", "unknown")` — and that attribute was set inside `for name, prompt in prompts.items()` at line 156-179, but `name` is also the loop variable in the OUTER `for proc in procs:` loop. Reading `proc._name` is fine, but the auth-scan at line 223 says `log_path = DATA_DIR / f"_specialist_{name}.log"` using the **outer loop variable**, not `proc._name`. The outer loop's `name` is `proc` shadowed — wait, actually `name` here IS `getattr(proc, "_name", "unknown")` at line 201. So the log path is correctly per-specialist. But: `cleanup_reports` (line 235) unlinks all logs unconditionally, including ones from a still-running re-invocation. Race window if `wait_for_specialists` returns early and the next L2 invocation launches before cleanup. Fix: include a per-invocation ID in the log filename.

- `portfolio/telegram_notifications.py:35-81` — `send_telegram` swallows Telegram 400 responses for non-markdown errors. The retry without `parse_mode` at line 73 is only triggered when `err_desc` contains "parse" / "markdown" / "entity". A 400 from chat-not-found, bot-not-in-chat, invalid-chat-id, or rate-limit-429 (which Telegram sometimes returns as 400 with `Too Many Requests` in description) returns False at line 81 — silently. `send_or_store` callers (which is everyone) never see the error. Fix: log error description at WARNING for any 400 not handled by retry; expose status_code in return.

- `portfolio/weekly_digest.py:286` — Writes a custom row directly to `data/telegram_messages.jsonl` then calls `send_telegram` separately. `send_telegram` already routes through `message_store.send_or_store` for everything else in the codebase; weekly_digest bypasses that pipeline and writes a row with a `type: "weekly_digest"` field instead of the standard `category` field that `message_store` uses. Result: the digest row is invisible to message-store-based filters (mute_all unmute lists, message_throttle, dashboard `/api/telegrams`). Fix: call `send_or_store(msg, config, category="weekly_digest")` and drop the duplicate write.

- `portfolio/main.py:1248` — Heartbeat write happens AFTER `verify_and_act` (line 1257). If `verify_and_act` invokes `_trigger_self_heal` which spawns a Claude session via `invoke_claude`, that call holds `_invoke_lock` and blocks for up to 180s. The heartbeat is delayed by that 180s — re-tripping the dashboard 300s stale gate the keepalive was meant to prevent. Fix: write heartbeat BEFORE `verify_and_act`, or wrap `verify_and_act` in another `heartbeat_keepalive`.

- `portfolio/loop_contract.py:1243-1244` — `_check_journal_uniqueness` does `f.readlines()` on the entire `layer2_journal.jsonl`. Comment at line 1241-1242 admits this is wasteful but justifies it with "JSONL is small in practice (one entry per Layer-2 decision = a few hundred per year)". With dedup_writes (see check_journal_uniqueness_safe failure log) and a runaway invocation pattern, the file could grow to thousands. The contract runs every cycle and the cycle limit is 480s (line 115). Reading a 100MB journal every 60s is an easy way to push past 480s. Fix: use `load_jsonl_tail(JOURNAL_FILE, max_entries=_JOURNAL_UNIQUENESS_TAIL)`.

- `portfolio/weekly_digest.py:29` — `_load_jsonl(SIGNAL_LOG, since=week_ago)` calls `load_jsonl` (full read), then filters in Python. signal_log.jsonl is 68MB+ per CLAUDE.md note. Running this in the weekly job means a 68MB read + ~5s parse + ~1s filter, which can crash the dashboard if it shares the venv (memory spike). Fix: stream-parse the file like loop_contract does, OR query `signal_log.db` (SQLite mirror).

## P2 — Medium

- `portfolio/agent_invocation.py:39-46` — `_agent_log_start_offset`, `_agent_start_wall`, `_agent_start`, etc. are module-level globals modified under `_completion_lock` for some paths (lines 574-598) but NOT for the assignment at lines 867-871 (which happens BEFORE the Popen, no lock). The pre-Popen assignment is documented as intentional (comment 861-865), but it means between line 867 (`_agent_start = time.monotonic()`) and line 872 (`_agent_proc = subprocess.Popen(...)`), the watchdog could observe `_agent_proc is None` AND `_agent_start > 0` — a transient state. The watchdog's `_check_agent_completion_locked` short-circuits on `_agent_proc is None` (line 1139) so this is fine, BUT if a future refactor inverts that check, the bug reappears. Fix: document the invariant ("if _agent_proc is None, all other _agent_* state is ignored") at module top.

- `portfolio/agent_invocation.py:881-894` — `_patient_txn_count_before` / `_bold_txn_count_before` snapshot is wrapped in a `try/except` that swallows ALL exceptions and resets to 0. If the file is mid-rename during atomic-write (FileNotFoundError) we silently lose the baseline. `_record_new_trades` at line 1083 then sees `len(txns) <= count_before` with `count_before=0` only when `len(txns)==0`, so any prior txns "appear" as new and get recorded twice via `record_trade`. Fix: when load_json fails, set count_before to a sentinel like `None` and skip `_record_new_trades` rather than treating as 0.

- `portfolio/trigger.py:282` — `flip_cooldowns` is keyed by ticker but never pruned. If a ticker is removed from `SYMBOLS` (e.g. the 2026-04-09 removal of PLTR/NVDA/MU/SMCI/TSM/TTWO/VRT), its `flip_cooldowns` entry persists forever. Bounded growth (~5 tickers) so not catastrophic, but the `triggered_consensus` pruning at lines 119-125 does NOT apply to `flip_cooldowns` / `sustained_counts` / `stable_sentiment` / `sustained_sentiment`. Fix: extend the pruning loop to cover all per-ticker dicts.

- `portfolio/trigger.py:432-481` — `classify_tier` uses `state.get("last_full_review_time", 0)` AND `state.get("last_trigger_date") != _today_str()`. `last_trigger_date` is updated at line 349 only when `triggered=True`. So the first non-grace trigger of the day always elevates to T3 — including a low-conviction consensus crossing in ranging regime that should be T1 per the downshift logic at line 476. The first-of-day check at line 458 runs BEFORE the downshift logic. Fix: move the downshift check to run regardless of the first-of-day check, OR exclude downshift-eligible reasons from setting `last_trigger_date`.

- `portfolio/main.py:846` — `_is_agent_window()` is called on every cycle for every trigger, computing UTC weekday + DST + holiday check. Cheap, but the result is also computed by `get_market_state()` at line 451 of the same `run()` function. Two independent DST calculations per cycle, one for `market_state` and one for `_is_agent_window`. They can disagree at the exact moment of DST transition (last Sunday of March/October between 00:00 and 02:00 UTC). Fix: factor through a single `MarketContext` object passed down.

- `portfolio/main.py:759-774` — When `signals_failed > 0 OR _cycle_signal_failures` fires, the error alert is built with `int(_run_elapsed)` but the `_signal_failures` are only collected from `extra._signal_failures`, which is per-ticker. If `signal_engine` itself raises, the ticker is in `signals_failed` (not `_cycle_signal_failures`), so the message reports "5 ticker(s) failed entirely" with no names. Fix: track the failed ticker names in `signals_failed` (already known from `timed_out` block at line 631 — needs propagation).

- `portfolio/main.py:923` — Safeguard checks run every 100 cycles. At 600s cadence that's every 16.7 hours, not "every 100 minutes" as the comment claims. The comment is wrong post-2026-04-09 cadence bump (market_timing.py:20 confirms 600s). The actual interval matters: dead-signal detection runs only 1.4 times per day. Fix: tie to wall-clock elapsed like the log rotation at main.py:404-405 does.

- `portfolio/llama_server.py:553-568` — `query_llama_server` holds `_thread_lock` AND file lock for the entire HTTP query (up to 240s timeout per line 599). The metals loop and main loop both call this. If one cycle's query takes 200s, the other process queues for 200s — its own cycle budget evaporates. The comment at line 549-552 acknowledges this is intentional. But: the loop_contract `MAX_CYCLE_DURATION_S=480` allows a 240s LLM query + ~200s other work = exactly at the boundary. Any HTTP retry inside `_query_http` (not shown) would push past 480s. Fix: drop the query timeout to ~120s (long enough for completion, short enough to leave loop budget).

- `portfolio/llama_server.py:180` — `_kill_server_by_pid` catches all exceptions but the `pid` variable is only assigned inside the `if content` block, so if the pid file is empty, line 180 `logger.debug("Failed to kill server pid=%s", pid)` raises `NameError`. Caught by the bare `except`, but loses the diagnostic. Fix: initialize `pid = None` at top of function.

- `portfolio/llama_server.py:215` — `_is_server_alive()` uses `logger.warning` on every failure including normal "server not running" — that's not a warning, it's the expected state. Fills the log with noise. Fix: use `logger.debug`.

- `portfolio/health.py:185-196` — `check_agent_silence` falls back to parsing `invocations.jsonl` and writes the result to `health_state` to cache it. But the lock is taken AFTER `last_jsonl_entry` returns. Between the read and the lock-protected write, another thread could update `last_invocation_ts` — we overwrite their fresher value with our slightly-stale one. Fix: take the lock first, re-check the field, then read JSONL only if still missing.

- `portfolio/journal.py:28` and `journal_index.py:373` — Both do raw `with open(JOURNAL_FILE, encoding="utf-8") as f:` to scan the whole journal file. Comment in CLAUDE.md says "Atomic I/O only" — the journal IS append-only so this is technically safe, but: the file grows unbounded (no rotation entry for `layer2_journal.jsonl` is set up — main.py:360 prunes it to 5000 entries, but `journal_index.retrieve_relevant_entries` always reads the full file before BM25 indexing on every L2 invocation). At 5000 entries × ~1KB = 5MB, parse takes ~50-200ms — acceptable, but if a future bug bypasses the prune, this becomes unbounded. Fix: switch to `load_jsonl_tail(JOURNAL_FILE, max_entries=5000)`.

- `portfolio/journal_index.py:280` — `_compute_importance` is called once per entry per L2 invocation. With 5000 entries and the BM25 fit/score, this is ~250ms of redundant compute per invocation — the index is rebuilt from scratch every time. Fix: cache the index keyed by `JOURNAL_FILE.stat().st_mtime` so subsequent invocations within the same cycle reuse it.

- `portfolio/journal.py:35` — `datetime.fromisoformat(entry["ts"])` does not enforce timezone-aware comparison. If a legacy entry has a naive timestamp (no tzinfo), comparison `ts >= cutoff` at line 36 raises TypeError ("can't compare offset-naive and offset-aware datetimes"), is caught by the bare except, and the entry is dropped silently. Naive timestamps in old journals would never appear in the BM25 index. Fix: assume UTC for naive timestamps (`if ts.tzinfo is None: ts = ts.replace(tzinfo=UTC)`).

- `portfolio/gpu_gate.py:265` — `_THREAD_LOCK.release()` is in the `finally` of the outer `try`, but the inner `try` that wraps the yield can re-enter via the same module — `_THREAD_LOCK` is a non-reentrant `threading.Lock`. If `gpu_gate` is called from a callback inside a `gpu_gate` context (e.g. a signal handler running inside the model load), the second acquire blocks forever. Realistic scenario: any code that calls `query_llama_server` (which itself takes `_thread_lock`) from inside a `with gpu_gate("chronos"):` block — Chronos signal does this pattern. Fix: use `threading.RLock`.

- `portfolio/gpu_gate.py:198-243` — When `_THREAD_LOCK.acquire(timeout=)` succeeds but the file-lock acquire then fails (timeout), the function `yield False` and returns — the `finally _THREAD_LOCK.release()` fires. Good. But: the `yield True` path at line 255 is inside a `try:` whose `finally:` at line 256-264 attempts another `_release_lock()` (file lock cleanup). If `_release_lock` raises (it suppresses OSError per line 99 but other exceptions propagate), `_THREAD_LOCK.release()` at line 266 fires regardless. So thread lock is always released. OK, but: if `_release_lock` raises a non-OSError, the file lock is leaked. The next acquirer hits "GPU file-locked" and waits up to 60s for the stale-break sweeper. Fix: broaden the `suppress` in `_release_lock`.

- `portfolio/multi_agent_layer2.py:75-92` — `build_specialist_prompts` reads `f"Read {f}." for f in spec["data_files"]` and joins into the prompt. These paths are RELATIVE strings hardcoded in `SPECIALISTS` (e.g. `"data/agent_context_t2.json"`). Claude CLI is launched with `cwd=str(BASE_DIR)` at line 173, so the relative read works — IF and only if BASE_DIR matches at spawn time. If main.py is ever started from a different working directory (e.g. via systemd that chdir'd elsewhere), `BASE_DIR` (computed via `__file__` at line 32) still resolves to the repo root, BUT Claude's tool `Read` is given a relative path and resolves it against `cwd` of the spawn — which IS BASE_DIR. OK. But the prompt itself is text — Claude can choose to interpret the path differently. Fix: use absolute paths in the prompt.

- `portfolio/perception_gate.py:69-85` — Iterates `summary.get("signals", {})` and reads `conf > max_confidence` — but the gate is meant to SKIP low-strength invocations. The "skip when no non-HOLD signals" logic at line 79 means a tier-1 invocation triggered by `F&G crossed` (a bypass keyword per line 26) is correctly bypassed at line 52-55. But if the trigger reasons list is empty (e.g. `force_report=True` startup path with the bug above), no keyword matches and the gate falls through to "no non-HOLD signals" — returns `False`. Good. But for the `reasons_list = ["startup"]` case, "startup" doesn't match any bypass keyword, so gate runs full evaluation. Subtle: this is the only path that COULD block the startup spam from P0 — but it requires `perception_gate.enabled=true` in config which is NOT default. Fix: make `enabled=true` default, OR have the gate trip on reason="startup".

- `portfolio/regime_alerts.py:39-43` — `_get_last_regime` reads ALL of `regime_history.jsonl` then scans in reverse — O(n) per ticker per cycle, n grows by ~5 entries per regime change. Bounded by infrequency of changes but no rotation is registered. Over years, this becomes slow. Fix: use `last_jsonl_entry(REGIME_HISTORY_FILE, ...)` with a filter callback, or maintain a per-ticker latest-regime cache file.

- `portfolio/regime_alerts.py:46-79` — `check_regime_transition` uses string equality on regime names but the normalization at line 62-65 only handles "ranging" → "range-bound". The set of regimes in `VALID_REGIMES` at line 18 includes both `range-bound` and `ranging` as aliases — but `_get_last_regime` returns whatever was stored, and `current_regime` comes from `signal_engine` which may produce either. So transitions like `ranging → range-bound` are correctly normalized as "no change", but `ranging → trending-up` works. The whole thing is fragile to a future name addition. Fix: enforce a single canonical name at write time.

- `portfolio/reflection.py:43` — `_compute_strategy_metrics` matches BUY/SELL pairs per ticker but it appends ALL prior buys for a ticker into `buys[ticker]` then averages on first SELL — never popping. Result: every subsequent SELL after the first is computed against an ever-growing average that includes already-sold shares. Stat is wrong for any ticker traded >2 times. - `portfolio/llm_outcome_backfill.py:117-130` — `_load_existing_keys` reads the entire outcomes JSONL on every backfill call. The function is called once per backfill run (acceptable) but the comment at line 256 mentions hourly/daily scheduling — at that cadence the cost is fine. ## Cross-cutting observations

- The claude-gate enforcement is incomplete. Three call sites (`analyze.py`, `multi_agent_layer2.py`, `agent_invocation.py`) bypass it. The CLAUDE.md rule "Direct subprocess.Popen calls are FORBIDDEN" is documented but not enforced — `claude_gate.py:8` says "Doing so bypasses the kill switch, rate limiter, and invocation tracking" — but the violators have been in main for weeks. A pre-commit grep hook `subprocess.\(run\|Popen\).*claude` would have caught them.

- Layer 2 invocation has THREE distinct subprocess paths: (a) `agent_invocation.invoke_agent`, (b) `multi_agent_layer2.launch_specialists` (3 parallel), (c) `claude_gate.invoke_claude*` (all other callers). Only (c) gets tree-kill and centralized log/auth scan. (a) and (b) reimplement portions inconsistently. Consolidation would eliminate ~250 lines of duplicate kill/env/log logic.

- The `_agent_log_start_offset` mechanism in agent_invocation.py is fragile (P1). The same pattern should use a per-invocation log file (like `multi_agent_layer2` does at line 169 with `_specialist_{name}.log`) — that's simpler than offset tracking. Each Popen gets its own log; auth scan reads the whole file unconditionally. Tradeoff: more open file handles, but the open count is bounded by max active invocations (1 in normal flow).

- `health_state.json` is the single point of truth for >5 producers (main, agent_invocation, dashboard, contract, signal_engine via update_signal_health_batch). `_health_lock` protects most paths but `agent_invocation`'s tier publish (line 904-916) and the migration writeback in `check_agent_silence` (line 192-196) are outside the lock. Suggest a `health.update_field(key, value)` helper that always takes the lock.

- The `force_report=True` + `["startup"]` + first-of-day → T3 chain is the kind of emergent behavior the loop-contract layer is meant to catch — but loop_contract's checks fire AFTER `run()` completes, so by then the T3 has already been spawned. A pre-spawn check (e.g. inside `invoke_agent`) that rejects `reasons==["startup"]` for high tiers would close the loop.

- Three "is this Claude alive?" mechanisms exist: (a) `_agent_proc.poll()` (agent_invocation), (b) `last_invocation_ts` cache in health.py, (c) `invocations.jsonl` tail in `check_agent_silence`. They can drift — e.g. `_agent_proc.poll()` says "running" but health says "silent 7200s" because the cache was last updated by the previous invocation's start, not by the in-flight one's heartbeat. Suggest a single source-of-truth state file written by the watchdog.

- The `subprocess.Popen(..., stdin=subprocess.DEVNULL)` pattern is used in analyze.py (line 288) but NOT in agent_invocation.py:872 (no stdin spec — defaults to inheriting parent's stdin). For a long-running daemon, this means the Claude subprocess inherits the loop's stdin (often /dev/null already, but if loop is run with `python -i` or under a debugger, stdin is a real terminal). The CLAUDE.md note "no interactive stdin" is enforced via PF_HEADLESS_AGENT in env (line 858) — but a misbehaving Claude could in principle still read stdin. Adding `stdin=subprocess.DEVNULL` to all Popen sites is cheap defense-in-depth.

- The auth-failure echo bug (BUG-ECHO 2026-04-16) fix is solid (`_is_real_auth_marker_line` + fenced-code-block tracking in `detect_auth_failure`). But the helper scans only the first 16 lines (`_AUTH_SCAN_LINE_LIMIT=179`). If claude CLI ever changes to print "Not logged in" after some preamble (e.g. version banner extends past line 16), the detector silently misses it. Recommend periodic synthetic-test that pipes a known auth-error transcript through `detect_auth_failure`.

- The 2-stage atomic file write pattern (`atomic_write_json` → tmp + rename) is everywhere, which is good. The `check_atomic_write_residue` invariant at loop_contract.py:1144 catches dead .tmp files older than 5 min. But `_TMP_RESIDUE_MAX_FILES = 100` — if a previously-stuck write left 200 .tmp files, only the first 100 get reported. Minor.

- The Layer 2 grace window is per-tier in `LAYER2_JOURNAL_GRACE_S_BY_TIER` (loop_contract.py:179-193), but the underlying tier publish (line 914 of agent_invocation.py) is a race (P0 issue). When the race loses, grace defaults to T3 (20m) per `LAYER2_JOURNAL_GRACE_S_DEFAULT`. A T1 invocation racing through this gap waits 20m for a journal that should have arrived in 2m — a 12m alerting delay on every cycle where the race happens. The race surface is small but real, and the symptom (delayed alerts, not missed) is hard to spot.
