# Codex adversarial review: signals-modules

## Summary
`portfolio/signals/` is mixed. I did not find raw canonical-state I/O violations in this subtree, and several modules log failures correctly, but `forecast.py` contains two real production blockers plus multiple direct policy drifts from `CLAUDE.md`, and several time-dependent signals are non-deterministic because they fall back to wall-clock time instead of bar time.

The highest-risk themes are: fail-open safety gates, ineffective timeout handling, inconsistent timezone/time extraction, and silent exception-to-HOLD behavior that can neutralize signals without any operator-visible alert.

## P0 — Blockers (production breakage / data loss / silent wrong trades)
- `portfolio/signals/forecast.py:415-424` — Chronos “timeout” is ineffective. Why it bites: `future.result(timeout=...)` raises, but the `with ThreadPoolExecutor(...)` context then waits for the worker thread on exit, so a hung `forecast_chronos()` call can still block the signal path indefinitely and tie up the GPU gate. Fix: move Chronos inference to a killable subprocess, or manage executor shutdown outside the context and fail hard without waiting on the stuck worker.
- `portfolio/signals/forecast.py:541-546,557-570,588-613,697-721` — Accuracy gating fails open when accuracy data loading breaks. Why it bites: any exception in the accuracy loaders degrades to `{}` / `insufficient_data`, and the module then returns raw forecast votes instead of force-HOLDing; that silently disables the safety gate exactly when the accuracy backend is unhealthy. Fix: fail closed on accuracy-loader errors, or at minimum trip a circuit breaker / alert and suppress forecast trades until accuracy data is available again.

## P1 — High (will cause incidents)
- `portfolio/signals/forecast.py:484-496,542-543,551-581,635-638` — Forecast policy constants drift from `CLAUDE.md`. Why it bites: the code uses `55%` / `10` samples, `14d` or `30d` snapshots, and `0.5` / `0.6` regime discounts, while the accepted policy is `<47%` with `30+` samples, `70% recent 7d + 30% all-time`, and `0.75x` / `0.80x` penalties; live behavior no longer matches the project’s risk contract. Fix: centralize these values in one shared policy module and compute the required 7d/all-time weighted accuracy explicitly.
- `portfolio/signals/forecast.py:456-477,542-543,694-720` — The vote is dominated by 1h forecasts, but the per-ticker accuracy gate only loads 24h accuracy. Why it bites: `_health_weighted_vote()` double-counts 1h votes, yet `_load_forecast_accuracy()` asks for `horizon="24h"` only; good 1h models can be suppressed by bad 24h stats, and bad 1h models can pass because 24h happens to look okay. Fix: gate each horizon on its own accuracy before weighting, or compute composite accuracy with the same 2x/1x horizon weights used in voting.
- `portfolio/signals/forecast.py:70-104,315-317,786-787` — `kronos_enabled` / shadow mode is only read at import time despite the comment promising live config flips. Why it bites: operational enable/disable changes do nothing until process restart, so on-call cannot reliably kill Kronos during an incident. Fix: reload this flag per call or on a short TTL, and include the effective mode in the forecast cache key.
- `portfolio/signals/econ_calendar.py:41-45` — Time extraction is both type-fragile and timezone-wrong. Why it bites: anything other than a `pd.Timestamp` in `df["time"]` falls back to `datetime.now(UTC)`, and tz-aware timestamps are relabeled with `.replace(tzinfo=UTC)` instead of converted, which can shift event proximity by hours around the `<4h` and `<24h` SELL thresholds. Fix: parse with `pd.to_datetime(..., utc=True)` and never substitute wall-clock time when bar time is present.
- `portfolio/signals/intraday_seasonality.py:79-92` — The signal ignores `df["time"]` entirely and falls back to current UTC time when the index is not a tz-aware `DatetimeIndex`. Why it bites: a normal OHLCV frame with a `time` column and `RangeIndex` will produce different outputs depending on when the code runs, making backtests/replays non-deterministic. Fix: read the timestamp from `df["time"]` first, normalize to UTC, and only use the index as a secondary path.
- `portfolio/signals/gold_overnight_bias.py:44-57` — Same wall-clock fallback bug in a time-of-day signal. Why it bites: if the input frame carries timestamps in a `time` column or a naive/non-datetime index, the overnight/London-PM classification is based on the machine clock, not the bar being scored. Fix: use deterministic bar-time extraction identical to `calendar_seasonal.py`.
- `portfolio/signals/calendar_seasonal.py:463-486` — Local quorum is hardcoded to `2` active votes, not `MIN_VOTERS = 3`. Why it bites: the module can emit BUY/SELL on only two active voters, violating the consensus rule and allowing weaker calendar noise to become a tradable signal. Fix: enforce the shared `MIN_VOTERS = 3` rule on active BUY+SELL voters.
- `portfolio/signals/vwap_zscore_mr.py:94-99` — This module also emits on only `2` active voters. Why it bites: it directly contradicts the project-wide consensus floor and can turn a 2-vote mean-reversion blip into a live trade. Fix: require at least three active voters before returning non-HOLD.
- `portfolio/signals/news_event.py:547-580` — Every sub-signal exception is swallowed to HOLD with no logging. Why it bites: parser bugs, keyword regressions, or data-shape changes silently neutralize the news risk signal while the overall module still returns a plausible result. Fix: `logger.exception(...)` each failed sub-signal and surface a degraded-state flag in `indicators`.

## P1 — High (will cause incidents)
- `portfolio/signals/forecast.py:944-966` — The dedup “fix” is still a TOCTTOU race. Why it bites: one thread reads `should_log`, unlocks, appends, then another thread can do the same before `_last_prediction_ts` is updated, producing duplicate accuracy rows that distort the same gating stats the module relies on. Fix: keep the decision and reservation in one critical section, or use a per-ticker compare-and-set token.
- `portfolio/signals/claude_fundamental.py:888-891,925-934` — Refresh timestamps are advanced before refresh success is known. Why it bites: a transient Claude/yfinance failure still sets `_cache[tier]["ts"]`, suppressing retries for the full 5m/30m/2h cooldown while stale or empty results are served. Fix: separate “refresh in progress” from “last successful refresh”, or roll back the timestamp on failure.
- `portfolio/signals/news_event.py:45-49,52-96` — All tickers overwrite the same `data/headlines_latest.json`. Why it bites: concurrent ticker evaluations are last-writer-wins, so the fish monitor can show the wrong ticker’s headlines or flicker across symbols. Fix: persist per-ticker files or append JSONL keyed by ticker.
- `portfolio/signals/momentum.py:356-425` — Eight sub-indicator exceptions degrade to HOLD/NaN without any log. Why it bites: dtype/math regressions silently alter vote mix and confidence with no operator visibility. Fix: log each caught exception and expose degraded-subsignal metadata.
- `portfolio/signals/oscillators.py:505-578` — Same silent-degradation pattern across eight oscillators. Why it bites: this is another broad technical-signal surface where failures become invisible HOLDs. Fix: same as above.
- `portfolio/signals/vwap_zscore_mr.py:43-46,124-125` — Any unexpected exception blanks the whole signal to an empty HOLD without any log trail. Why it bites: a real defect becomes indistinguishable from “no setup”. Fix: log the exception and return a degraded result that preserves error context.
- `portfolio/signals/credit_spread.py:53-67,112-115` — Mutable module cache is unsynchronized; the same pattern appears in `hash_ribbons.py:51-108` and `crypto_evrp.py:51-53,71-73,107,121-123,171`. Why it bites: under threaded signal execution, duplicate network fetches and stale/partial cache overwrites are possible. Fix: use the lock pattern already present in `metals_vrp.py` and `metals_cross_asset.py`.

## P3 — Low (style / dead code / minor)
- `portfolio/signals/intraday_seasonality.py:95-107` — Unknown tickers default to `"crypto"` seasonality instead of failing closed. Why it bites: an unclassified equity/ETF inherits crypto hour/day multipliers. Fix: default to `"unknown"` / HOLD unless `asset_class` is explicitly provided.
- `portfolio/signals/calendar_seasonal.py:137-164` — The docstring says May-October is a SELL-bias period, but the implementation returns HOLD. Why it bites: intent is ambiguous, and future maintainers/tests can “fix” this in opposite directions. Fix: align code, comments, and tests on one policy.

## Tests missing
- `portfolio/signals/forecast.py:415-424` — No test that a hung Chronos call actually returns within the timeout instead of blocking on executor shutdown.
- `portfolio/signals/forecast.py:484-721` — No policy-lock test asserting the CLAUDE accuracy gate (`47%`, `30+` samples, `70/30` recency weighting, `0.75x/0.80x` regime penalties).
- `portfolio/signals/intraday_seasonality.py:79-92`, `portfolio/signals/gold_overnight_bias.py:44-57`, `portfolio/signals/econ_calendar.py:41-45` — No tests covering `time` column inputs, `RangeIndex`, naive timestamps, and non-UTC aware timestamps.
- `portfolio/signals/calendar_seasonal.py:463-486`, `portfolio/signals/vwap_zscore_mr.py:94-99` — No tests enforcing `MIN_VOTERS = 3` on active BUY+SELL voters.
- `portfolio/signals/news_event.py:45-49,52-96` — No concurrency test proving headline persistence remains ticker-correct under parallel evaluations.
- `portfolio/signals/claude_fundamental.py:888-891,925-934` — No test that a failed background refresh retries promptly instead of waiting out the full cooldown.

## Cross-cutting observations
- Exception hygiene is inconsistent. `econ_calendar.py:211-241`, `macro_regime.py:364-397`, `structure.py:223-244`, and `volume_flow.py:301-360` log sub-signal failures; `momentum.py:356-425`, `oscillators.py:505-578`, `news_event.py:547-580`, and `vwap_zscore_mr.py:124-125` mostly hide them.
- Time extraction is inconsistent across time-sensitive signals. `calendar_seasonal.py:379-389` robustly parses `df["time"]`, while `econ_calendar.py:39-45`, `intraday_seasonality.py:79-92`, and `gold_overnight_bias.py:44-57` fall back to the wall clock or mishandle timezone conversion.
- Atomic I/O is otherwise clean in this subtree. I found `load_json` / `atomic_append_jsonl` / `atomic_write_json` usage in `claude_fundamental.py:24,649-673`, `forecast.py:29,232,964`, `news_event.py:22,96`, and no raw `json.loads(open(...).read())`-style canonical-state reads here.
- I did not find stop-loss API usage, barrier-proximity logic, Telegram delivery code, or Claude subprocess bypasses inside `portfolio/signals/`; the serious issues are concentrated in forecast gating, time handling, races, and silent fallbacks.
