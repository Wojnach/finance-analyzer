# Codex adversarial review: infrastructure

## Summary
Two issues in the dashboard auth path are immediate blockers: auth fails open if the token file cannot be read, and Cloudflare Access is trusted on header presence alone. Outside auth, the dominant pattern is ad-hoc file/state handling outside the shared atomic helpers, which creates silent journal loss, stale cache behavior, and overlapping background workers.

## P0 — Blockers (production breakage / data loss / silent wrong trades)
- `dashboard/auth.py:60-67,81-83,121-123` — Dashboard auth fails open when `config.json` is missing, unreadable, or malformed. Why it bites: `_read_config_uncached()` swallows read/parse errors to `{}`, `_get_dashboard_token()` returns `None`, and `require_auth()` explicitly allows the request when `expected is None`, so a transient config read failure disables all auth. Fix: fail closed, log/alert the config read error, and refuse requests until a valid token is loaded.
- `dashboard/auth.py:132-135`, `dashboard/app.py:2080-2083` — Cloudflare Access trust is header-presence only. Why it bites: any direct client that can reach the Flask server can forge `Cf-Access-Authenticated-User-Email` and `Cf-Access-Jwt-Assertion`; the code never verifies the JWT, and the app binds on all interfaces. Fix: validate the Access JWT cryptographically or reject direct traffic unless it comes from a trusted proxy.

## P1 — High (will cause incidents)
- `portfolio/file_utils.py:222-227,349-392` — `prune_jsonl()` rewrites JSONL outside `jsonl_sidecar_lock()`. Why it bites: concurrent `atomic_append_jsonl()` writers can append after prune reads but before replace, and those lines are silently lost. Fix: hold the sidecar lock for the entire read-filter-replace sequence.
- `portfolio/process_lock.py:39-44,63-67`, `portfolio/file_utils.py:214-218` — Windows lock acquisition on a brand-new empty lock file is unreliable, and any resulting `OSError` is treated as benign contention. Why it bites: first startup can fail to obtain the singleton lock or misclassify a real lock failure, allowing duplicate loops/processes. Fix: seed a byte before locking, distinguish lock contention from other OS errors, and surface unexpected failures.
- `portfolio/subprocess_utils.py:132,175,247-252` — Job-object containment is assumed but not verified, and the fallback kill path omits `/T`. Why it bites: `AssignProcessToJobObject()` can fail without raising, leaving children outside the job; later cleanup then kills only the direct PID and leaves grandchildren orphaned. Fix: check the Win32 return value, fail hard on assignment failure, and use `taskkill /T /F` as the safety net.
- `portfolio/shared_state.py:30-32,68-75,154-215` — `_cached_or_enqueue()` never expires stale `_loading_keys` even though `_cached()` does. Why it bites: if a batch update crashes after enqueueing, the key can stay “loading” forever, so later callers stop refreshing it and keep returning stale/missing data silently. Fix: apply the same stale-key eviction before the enqueue check, or clear loading keys on timeout/error.
- `scripts/fix_agent_dispatcher.py:3-4,54-55,144-151` — Scheduled runs can overlap, and they race on a single hard-coded `fix_agent_state.json.tmp` without locking or the shared atomic helper. Why it bites: a 10-minute scheduler can overlap with a 15-minute agent timeout, so concurrent runs can clobber cooldown state and spawn duplicate fix agents. Fix: serialize runs with a process lock and write state via `atomic_write_json`.
- `dashboard/house_blueprint.py:211-216,280,330,359-365` — External markdown and `heatmap.html` are rendered into the authenticated dashboard origin without sanitization. Why it bites: a compromised repo or generated report can execute same-origin JS and drive authenticated dashboard actions. Fix: sanitize markdown/HTML, serve untrusted reports from a separate origin, or force download instead of inline render.
- `dashboard/app.py:1719-1738,1758-1772` — The single-thread Avanza snapshot queue is unbounded and timed-out requests are not drained. Why it bites: one hung `_avanza_snapshot_impl()` blocks the worker; every later request enqueues a future, waits 25s, times out, and leaves more work piling up behind the stall. Fix: bound the queue, coalesce duplicate requests, and reset the worker after timeout/hang detection.

## P2 — Medium (correctness / robustness)
- `portfolio/shared_state.py:48,93,100` — Cache timestamps are recorded before the expensive fetch runs. Why it bites: slow fetches age in cache while still in flight, causing premature expiry and unnecessary reload storms. Fix: stamp cache entries after `func(*args)` returns.
- `portfolio/vector_memory.py:267-281` — Journal ingestion uses raw `open(...)/json.loads(...)` and silently skips malformed lines. Why it bites: this violates the project’s atomic-I/O rule and can hide journal corruption while building incomplete memory state. Fix: route reads through the shared file helpers and surface corruption.
- `portfolio/vector_memory.py:29-31,34-54` — The module-level `_collection` singleton ignores `collection_name`. Why it bites: the first initialized Chroma collection is reused for all later names, so callers can silently read/write the wrong vector store. Fix: key collections by name or reinitialize when the requested name changes.
- `scripts/check_critical_errors.py:42-49` — Critical-error journal reads bypass the shared helpers and silently skip malformed lines. Why it bites: the startup gate can miss unresolved incidents or missed resolutions and make decisions on incomplete data. Fix: use the shared JSONL reader path and fail loudly on corruption.
- `scripts/fix_agent_dispatcher.py:84-91` — The dispatcher also raw-parses `critical_errors.jsonl` and skips malformed lines. Why it bites: auto-remediation can fire or stand down based on a truncated incident history. Fix: use the shared JSONL helper and treat corruption as actionable.
- `dashboard/app.py:898-905,910-917` — `/api/mstr_loop` full-scans journals with raw `json.loads` and `pass` on `JSONDecodeError`/`OSError`. Why it bites: corruption is hidden, the route can serve stale last-seen data, and the scan cost grows with log size. Fix: use the file helpers or a bounded tail reader and surface errors.
- `dashboard/app.py:84-94,1908-1917` — Two cache paths start their TTL before the expensive read/snapshot work begins. Why it bites: slow reads expire “early,” which increases refresh frequency and retry pressure exactly when the dependency is slow. Fix: timestamp entries after the data is collected.
- `dashboard/system_status.py:762-779` — Time-window filtering compares timestamps lexically as strings instead of parsing datetimes. Why it bites: mixed ISO forms (`Z`, `+00:00`, naive timestamps, differing precision) can undercount or overcount recent failures and Layer 2 activity. Fix: normalize to timezone-aware datetimes before comparison.
- `portfolio/notification_text.py:30-33` — `format_confidence()` crashes on `NaN`. Why it bites: `float('nan')` survives the cast and `int(round(...))` raises, so a bad upstream metric can turn a notification into an exception instead of degraded text. Fix: reject non-finite values and fall back to `"n/a"`.
- `dashboard/export_static.py:58,64-67,106-113` — The deprecated exporter still defaults to writing API snapshots under `dashboard/static/api-data` and uses raw config/file I/O. Why it bites: one invocation republishes authenticated API data under Flask’s public static tree. Fix: move the default output outside the served tree and remove the raw read/write path.

## P3 — Low (style / dead code / minor)
- `portfolio/subprocess_utils.py:173-179` — `popen_in_job()` can leak the job handle on the exception path after process creation. Why it bites: repeated failures slowly leak OS handles and muddy process-lifecycle debugging. Fix: close the job handle before returning on that path.
- `dashboard/app.py:793-800` — Logout hard-codes `"pf_dashboard_token"` instead of using the shared cookie constant. Why it bites: a future cookie-name change breaks logout silently. Fix: use the shared constant everywhere.
- `dashboard/app.py:1669-1672` — `/api/market_health` logs `"mstr endpoint error"`. Why it bites: incident triage points operators at the wrong subsystem. Fix: log the correct route/context name.

- `dashboard/auth.py:132-135` — No test that forged `Cf-Access-*` headers without a valid JWT are rejected.
- `portfolio/file_utils.py:349-392` — No concurrency test for `atomic_append_jsonl()` racing with `prune_jsonl()`.
- `portfolio/process_lock.py:39-67` — No Windows test for first lock acquisition on a brand-new zero-length file.
- `portfolio/shared_state.py:154-215` — No recovery test for a stuck `_loading_keys` entry after a batch-update crash.
- `scripts/fix_agent_dispatcher.py:3-4,54-55,144-151` — No overlap test for two dispatcher runs preserving cooldown state and avoiding duplicate agent launches.
- `dashboard/house_blueprint.py:211-216,359-365` — No sanitization test proving markdown/HTML input cannot execute script in dashboard origin.
- `dashboard/app.py:1719-1772` — No resilience test for a hung Avanza worker proving the queue stays bounded and recovers.

## Cross-cutting observations
- The project’s atomic-I/O rule is repeatedly bypassed on canonical config/journal/state files in `dashboard/auth.py`, `dashboard/app.py`, `dashboard/export_static.py`, `scripts/check_critical_errors.py`, `scripts/fix_agent_dispatcher.py`, and `portfolio/vector_memory.py`.
- Several paths prefer silent degradation over fail-closed behavior: auth config loading, journal readers, and queue/cache state all hide corruption or stuck state instead of alerting. That is the same incident class your rules are trying to prevent.
- `portfolio/config_validator.py`, `portfolio/message_store.py`, `portfolio/shadow_registry.py`, and most of the modern dashboard static JS/CSS looked comparatively clean in this scope; I did not find a high-confidence blocker there.
