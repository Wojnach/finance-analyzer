# Codex adversarial review: avanza-api

## Summary
The subsystem has split into three Avanza stacks that no longer enforce the same safety rules. The highest-risk defects are silent false-success around stop-loss deletion, account-scope drift across reads/cancels, and publicly exported trading paths that bypass the whitelist/locking hardening added elsewhere.

## P0 — Blockers (production breakage / data loss / silent wrong trades)
- `portfolio/avanza_control.py:395-404` — `delete_stop_loss_no_page()` ignores `_api_delete()`'s `ok/http_status` and returns success for any dict without `errorCode`, so `403`/`500`/other failed deletes become `True`. Why it bites: callers can proceed into a sell while the stop is still live. Fix: call `portfolio.avanza_session.cancel_stop_loss()` here, or at minimum gate on `result["ok"]`/`http_status` and keep the order lock.
- `portfolio/avanza_session.py:934-995,1034-1043` — `cancel_all_stop_losses_for()` defaults `account_id=None`, filters by `orderbook_id` only, then cancels each stop using that stop's own account id. Why it bites: one call can remove protective stops from unrelated accounts that happen to hold the same instrument, violating the whitelist rule. Fix: default to `DEFAULT_ACCOUNT_ID`, reject non-whitelisted accounts, and require an explicit opt-in for cross-account cleanup.
- `portfolio/avanza_session.py:676-717`, `portfolio/avanza_client.py:172-178` — the preferred BankID/session path returns all positions across all accounts, bypassing the whitelist filtering that exists in the TOTP fallback. Why it bites: strategy state can silently include long-term holdings from the wrong account and drive wrong trade decisions. Fix: filter session positions to `ALLOWED_ACCOUNT_IDS` before returning them.
- `portfolio/avanza_session.py:749-801,764-775`, `portfolio/avanza/trading.py:239-278,243-254` — the stop-loss placement functions never check “within 3% of current bid” or “near MINI barrier” and both deliberately downgrade sub-1000 SEK legs to warnings. Why it bites: the exact code that sends `/_api/trading/stoploss/new` does not enforce the project’s stop-loss safety policy. Fix: fetch live quote/barrier first, reject unsafe placements, and enforce the 1000 SEK floor if that rule is mandatory.

## P1 — High (will cause incidents)
- `portfolio/avanza/__init__.py:29-43`, `portfolio/avanza/client.py:59-67`, `portfolio/avanza/trading.py:80-92,128-138,159-161,239-278,351-355` — the exported `portfolio.avanza` mutators take `account_id` straight from config/override and never use `avanza_order_lock`. Why it bites: the clean public API bypasses both the account whitelist and the cross-process mutation lock that the legacy/session paths rely on. Fix: route these functions through the hardened helpers or duplicate the same whitelist/lock checks here.
- `portfolio/avanza_orders.py:68-79,132-215` — pending-order state fails open and is unsynchronized: unreadable JSON becomes `[]`, then both `request_order()` and `check_pending_orders()` do unlocked read-modify-write cycles. Why it bites: a transient read failure or concurrent writer can erase the confirmation queue or lose status updates. Fix: fail closed on unreadable state and serialize the whole RMW sequence with a file lock.
- `portfolio/avanza_client.py:345-368`, `portfolio/avanza_orders.py:349-364` — Telegram-confirmed orders execute through the legacy TOTP path, which validates only `volume >= 1` and `price > 0` and never enforces the 1000 SEK minimum. Why it bites: the human approval path can place orders that violate the Avanza floor the newer paths reject. Fix: add the same 1000 SEK guard in `portfolio.avanza_client._place_order()` or reuse `avanza_session._place_order()`.
- `portfolio/avanza_account_check.py:224-249` — `verify_default_account()` now converts API fetch failures into `{"ok": False}` and returns without raising even when `raise_on_mismatch=True`. Why it bites: a transient outage disables the only startup account verification unless every caller remembers to inspect the return dict. Fix: fail closed by default, or require an explicit caller opt-out backed by a last-known-good cache.
- `portfolio/avanza/account.py:74-94` — `get_buying_power()` returns an all-zero `AccountCash` when the account is missing or the overview shape drifts. Why it bites: callers cannot distinguish “real zero cash” from “account lookup failed,” which is the same silent-sizing bug `avanza_session.get_buying_power()` was already fixed to avoid. Fix: return `None`/raise on lookup failure and normalize the same alternate fields the session helper handles.
- `portfolio/avanza/trading.py:105-147` — `modify_order()` does no validation at all: no `price > 0`, no `volume >= 1`, no 1000 SEK floor, no whitelist/lock. Why it bites: an edit can turn a valid order into an invalid or policy-breaking one even though new orders are guarded. Fix: run the same validation/locking path as `place_order()` before calling `edit_order()`.

## P2 — Medium (correctness / robustness)
- `portfolio/avanza_tracker.py:64-69`, `portfolio/avanza_client.py:149-159`, `portfolio/avanza_session.py:1237-1242` — the tracker expects flat `lastPrice/changePercent` keys, but the preferred session path returns raw market-guide objects with nested `quote` fields. Why it bites: live session auth can silently publish `0.0` prices and changes for tracked instruments. Fix: normalize the session response before returning from `get_price()`, or parse nested quote fields in the tracker.
- `portfolio/avanza/types.py:333-345,364-371,494-506` — typed parsers lag the shapes handled elsewhere: `StopLoss` ignores nested `account.id`, `SearchHit` misses `orderBookId/title`, and `InstrumentInfo` ignores nested `keyIndicators` / `underlying.quote`. Why it bites: callers get empty IDs or zero barrier/leverage/underlying prices without an error. Fix: normalize against the shapes already used in `scanner.py` and `avanza_session.py`.
- `portfolio/avanza/scanner.py:52-69,308-315` — the scanner marks the TOTP client as `thread_safe` and fans out a thread pool over one shared singleton `avanza` client. Why it bites: this reintroduces shared-client races unless the underlying library explicitly guarantees concurrent `requests.Session` use. Fix: per-thread clients or a call-level lock. `(uncertain)`
- `portfolio/avanza_orders.py:272-282` — non-OK Telegram `getUpdates` responses are treated as “no confirmations” with no log unless an exception is thrown. Why it bites: `401`/`429`/API-level `ok=false` silently drop human approvals and look like an empty inbox. Fix: log and surface non-OK HTTP/API responses before returning `set()`.
- `portfolio/avanza_session.py:701-710` — P&L math zeroes positions whose current value is `0`, because `profit` and `profit_percent` are computed only when both `val` and `acq` are truthy. Why it bites: a total-loss position is reported as zero profit instead of `-acquired_value`. Fix: compute `profit = val - acq` whenever `acq` is known, and only guard the percentage divide.

## P3 — Low (style / dead code / minor)
- `portfolio/avanza_orders.py:264-266` — the Telegram offset fallback still does raw `read_text()` on a canonical state file, which violates the project’s atomic-I/O rule. Why it bites: it reintroduces an untracked non-atomic read path for broker-control state. Fix: migrate legacy text offsets once and keep all reads behind `load_json()`.
- `portfolio/avanza_account_check.py:141-151` — critical Telegram alert delivery failures are swallowed at debug level. Why it bites: production usually runs above debug, so missing account-mismatch alerts can disappear from operator view. Fix: log alert-send failures at warning/error with enough context to page on them.

## Tests missing
- `portfolio/avanza_control.py:395-404` — no regression test proves `delete_stop_loss_no_page()` returns failure on `403/500` and does not clear the sell path.
- `portfolio/avanza_session.py:934-1043` — no multi-account test covers two accounts holding the same `orderbook_id`; only the whitelisted account’s stop should be cancelled.
- `portfolio/avanza_client.py:172-178`, `portfolio/avanza_session.py:676-717` — no test asserts the preferred session-backed `get_positions()` filters to `ALLOWED_ACCOUNT_IDS`.
- `portfolio/avanza_orders.py:68-79,132-215` — no concurrent-state test covers `request_order()` racing `check_pending_orders()` or an unreadable pending-orders file.
- `portfolio/avanza_session.py:749-801`, `portfolio/avanza/trading.py:239-278` — no test enforces the stop-loss policy gates for bid distance, MINI barrier proximity, and sub-1000 SEK legs.
- `portfolio/avanza/types.py:333-345,364-371,494-506` — no parser fixtures cover nested `account.id`, `orderBookId/title`, `keyIndicators.barrierLevel`, and `underlying.quote.last`.

## Cross-cutting observations
- The hardening is uneven: `avanza_order_lock.py` and `avanza_resilient_page.py` are reasonable, but many public call paths bypass them.
- Account scoping is inconsistent across modules: some helpers whitelist, some default to “all accounts,” and the unified package trusts `config["avanza"]["account_id"]` directly.
- The same business capability exists in three stacks (`avanza_session`, legacy `avanza_client`, and `portfolio.avanza`), and their validation/error semantics have already drifted apart enough to create incident-class behavior differences.
