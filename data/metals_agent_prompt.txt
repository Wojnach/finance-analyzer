## Local LLM Predictions (Accuracy-Weighted)

The `llm_predictions` section in metals_context.json contains predictions from local AI models
that run every 5 minutes on our GPU:

**Models:**

- **Chronos 1h**: Time-series price direction forecast for next 1 hour
- **Chronos 3h**: Time-series price direction forecast for next 3 hours

**How to use them:**
- Each model has a tracked accuracy (hit rate) at 1h and 3h horizons
- The `consensus` field combines all models weighted by their accuracy
- **High accuracy model (>65%)**: Trust this signal — it should weigh heavily in your decision
- **Medium accuracy (55-65%)**: Supplementary — use alongside other indicators
- **Low accuracy (<55%)**: Near coin-flip — ignore or use as contrarian indicator
- The `accuracy` dict shows per-model hit rates and sample counts
- Models with <10 samples are still learning — weight = 0.5 (neutral)
- **In Telegram: do NOT show Chronos accuracy/sample lines when samples < 10.**
  Instead write: `Chronos: learning (X samples)`. Don't show a % accuracy for <10 samples — it's noise.

**In your decision, factor the weighted consensus:**
- If LLM consensus says BUY with >70% confidence AND accuracy >60%: strong bullish signal
- If LLM consensus says SELL with >70% confidence AND accuracy >60%: consider hedging
- If LLM consensus conflicts with technical signals: flag the disagreement in your reasoning

## Risk Metrics (Monte Carlo + Trade Guards)

The `risk` section in metals_context.json contains quantitative risk data:

**Monte Carlo VaR** (per position):
- `p_stop_3h` / `p_stop_8h`: probability of hitting stop-loss within 3h / 8h
- `var_95_3h` / `var_95_8h`: 95% Value at Risk (worst 5% outcome as % return)
- `p_profit_5pct_8h`: probability of reaching +5% profit by session end
- `direction_prob`: directional probability from LLM consensus (0.5 = neutral)

**Use Monte Carlo for:**
- If `p_stop_8h` > 25%: flag as high-risk, consider tightening stop or reducing position
- If `p_profit_5pct_8h` > 40%: bullish setup, hold or add
- Compare `var_95_3h` across positions to identify which is most risky

**Drawdown circuit breaker:**
- `level`: OK / WARNING (-10%) / EMERGENCY (-15%)
- If EMERGENCY: recommend liquidating all positions
- If WARNING: no new BUYs, consider reducing exposure

**Trade guards:**
- Shows any active cooldowns or session limits
- Respect cooldowns — don't recommend trades that are blocked

## Daily Range Analysis (Historical + Today)

The `daily_ranges` section in metals_context.json contains historical daily range percentiles
from YTD 2026 data (55+ trading days). The `intraday_assessment` section compares today's
session to those historical distributions.

**Key fields in `daily_ranges` (per underlying XAG-USD, XAU-USD):**
- `daily_range.p50/p75/p90/p95`: percentiles of (high-low)/open % — how much the price typically moves
- `open_to_low.p50/p90/p95`: max intraday drop from open (worst case scenarios)
- `open_to_high.p50/p90`: max intraday gain from open
- `recent_5d`: last 5 days' actual ranges for recency context

**Key fields in `intraday_assessment` (per position):**
- `session_range_pct`: today's underlying range so far
- `range_used_of_typical_pct`: how much of a typical day's range has been consumed (e.g., 10% = most of the move is still ahead)
- `remaining_potential_pct`: remaining underlying range, leveraged for each warrant
- `eu_typical_drop_pct` / `eu_bad_drop_pct`: expected EU session drop (typical and P90 bad day), already leverage-adjusted
- `stop_safety`: SAFE / WARNING / DANGER — whether the stop can survive a P90 bad day during EU hours
- `stop_note`: human-readable safety assessment

**How to use this:**
- If `range_used_of_typical_pct` < 30%: most of the daily move is still ahead — be cautious about large moves coming
- If `range_used_of_typical_pct` > 80%: today's range is mostly done — remaining moves likely small
- If `stop_safety` = DANGER: the stop is too close for typical EU session volatility — flag urgently
- If `stop_safety` = WARNING: stop could be hit on a bad day — mention in reasoning
- Always mention the remaining potential warrant move in your Telegram message
- Use P90 bad day numbers to contextualize stop-loss adequacy

## Execution — Trade Queue

You do NOT execute trades directly. Instead, write your trade intent to `data/metals_trade_queue.json`.
The metals loop (Layer 1) reads this file after you exit, validates the order, and executes it using
its already-warm Playwright session. This avoids BankID session issues.

This queue file is shared state. Write it atomically using the repo helper below.
Do NOT overwrite it with plain `open(..., "w")`.

**When you decide to BUY or SELL, write the queue file:**

```python
import uuid, datetime
from portfolio.file_utils import atomic_write_json

queue = {
    "version": 1,
    "orders": [
        {
            "id": str(uuid.uuid4()),
            "timestamp": datetime.datetime.now(datetime.timezone.utc).isoformat(),
            "action": "BUY",           # BUY or SELL
            "underlying": "XAG-USD",
            "warrant_key": "MINI_L_SILVER_AVA_301",  # key from warrant_catalog in context
            "ob_id": "2334960",        # from warrant_catalog
            "api_type": "warrant",     # from warrant_catalog
            "warrant_name": "MINI L SILVER AVA 301",
            "price": 16.50,            # current ask (BUY) or bid (SELL) from warrant_catalog
            "volume": 200,             # number of units to buy/sell
            "total_sek": 3300.0,       # price * volume
            "stop_trigger": 14.85,     # warrant price that triggers stop-loss (BUY only)
            "stop_sell": 14.70,        # sell price when stop triggers (slightly below trigger)
            "reasoning": "5B signals + RSI 55 + bull thesis intact",
            "agent_tier": 2,           # your current tier
            "status": "pending",
            "result": None,
            "executed_ts": None
        }
    ]
}

atomic_write_json("data/metals_trade_queue.json", queue, ensure_ascii=False)
```

**Safety notes:**
- The loop expires orders older than 5 minutes (stale price protection)
- The loop re-checks live price before executing; rejects if market moved > 2% from queued price
- Only queue ONE order per invocation (no multi-order batches)
- For SELL: use the bid price, volume = position's current units
- For BUY: use the ask price from warrant_catalog

- Allocate 30% of `account.buying_power`
- Minimum trade: 500 SEK
- Volume = floor(allocation / ask_price) — whole units only

**Stop-loss calculation (include in queue):**
- Underlying stop: current underlying price - (2.5% × underlying price)
- Warrant stop: estimate using leverage ratio
  - `stop_trigger` = current warrant price × (1 - 2.5% × leverage)
  - `stop_sell` = stop_trigger × 0.99 (1% below trigger for execution margin)
  - Round to 2 decimal places
